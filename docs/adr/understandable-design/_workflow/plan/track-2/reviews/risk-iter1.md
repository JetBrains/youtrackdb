**Certificate**: Exposure E1 (the per-round fidelity spawn the Phase 4 loop calls)
**Location**: track-2.md `## Plan of Work` concern 2 and `## Interfaces and Dependencies` (which lists the fidelity-check agent + `create-final-design.md` as in-scope but `edit-design/SKILL.md` as out-of-scope, Track 1); the gap is in the staged `edit-design/SKILL.md:505-510` (spawn-contract table) and `:515-521` (params-file contract).
**Issue**: Track 1's staged `edit-design/SKILL.md` already names the `phase4-creation` second check as the **fidelity check** (lines 457-460: "the fidelity-check role and its `phase4-creation` wiring are built in the Phase 4 track; this skill ... swaps only the second check"). But the two places in that same file that actually wire a per-round second check do **not** carry a fidelity row: the spawn-contract table at 505-510 lists only `design-author` / `readability-auditor` / `absorption-check` / `comprehension-review`, and the params-file contract at 515-521 names only the author's, auditor's, and absorption check's per-round inputs — not the fidelity check's (its `episodes`/`design_final_path`/`output_path` inputs). So after Track 2 adds the fidelity-check agent definition and reworks `create-final-design.md`, the loop that is supposed to *spawn* it every `phase4-creation` round still has no `subagent_type` entry, no allow-list row, and no params for it. The fidelity check would exist as an agent and be referenced in prose, but the dual-clean loop has no contract to invoke it with. Likelihood high (the omission is already on disk in Track 1's frozen output); impact: the Phase 4 fidelity check is non-functional or the implementer silently edits a file the track scoped out, breaking the stacked-diff boundary the track's own sizing justification rests on.
**Proposed fix**: Resolve the cross-track interface explicitly in the track file before Phase B. Either (a) pull the staged `edit-design/SKILL.md` fidelity spawn-contract row + params line into Track 2's in-scope set with a one-line note that it completes the hook Track 1 left open (preferred — it keeps the per-round second-check wiring in one file), or (b) if it must stay a Track-1 edit, record it as a Track-1 follow-up the Phase 4 promotion depends on and add an acceptance line that the fidelity spawn contract resolves. Whichever path, add an acceptance criterion that the `phase4-creation` round actually spawns the fidelity check (the table row + params exist and name the agent's basename).

### R2 [should-fix]
**Certificate**: Exposure E2 (the `create-plan` Step 1c auto-resume contract the collapse rewrites)
**Location**: track-2.md `## Plan of Work` concern 3 ("re-specify the crash-recovery path to fire only on a dirty or absent plan after a committed-and-clean design (Step 1c becomes crash-recovery-only)") against `create-plan/SKILL.md:131-275` (Step 1c, a 13-branch routing block) and `:527-551` (the design->plan session-boundary block).
**Issue**: D15 calls itself "a real machinery change, not a cosmetic convenience ... it rewrites the `create-plan` auto-resume contract" and the track scopes that to Step 1c plus the boundary block. The risk is under-specification of the rewrite. Step 1c today is **branch-exhaustive**: it routes seven distinct artifact combinations (`design.md` committed-clean -> 4b; `design.md` uncommitted/dirty -> 4a; plan-no-design `lite`; plan-no-design `full`/absent; `minimal` resume with ledger; `minimal` interrupted before ledger seed; fresh start; both-exist), and the block's closing paragraph (265-275) asserts "a defined resume path for every artifact combination, so a committed, clean `design.md` with no plan is never a dead end." The collapse removes the *happy-path* boundary (4a flows into 4b in one invocation), but the track's re-spec only names that one arm. After the collapse, several branches change meaning and the track says nothing about them: (1) the "`design.md` committed-clean, no plan -> auto-resume into 4b" branch is now reachable **only** as crash recovery, not as the normal flow — its prose still describes the normal flow; (2) the "uncommitted/dirty `design.md` -> resume Step 4a" branch must still exist (a crash mid-design-authoring), so "Step 1c becomes crash-recovery-only" must keep *both* the dirty-design and the dirty/absent-plan recovery arms, not just the plan arm the concern names; (3) the boundary block at 527-551 ("Step 4a ends the session ... it does not flow straight into Step 4b. The user re-invokes `/create-plan`") directly contradicts the collapsed flow and must be rewritten, but it is only obliquely in scope ("touch only where the auto-resume contract ... describes the 4a/4b boundary"). Likelihood medium; impact: a partial rewrite leaves Step 1c self-contradicting (the "every combination has a path" invariant broken, or a crash mid-authoring routed wrong), which is exactly the auto-resume contract a running phase reads — the prose analogue of a crash-recovery bug.
**Proposed fix**: Before Phase B, enumerate in the track file (or the Phase A decomposition) **every Step 1c branch** and its post-collapse disposition: which arms survive as crash-recovery-only, which arm's prose flips from "normal flow" to "crash recovery", and that the dirty/uncommitted-`design.md` recovery arm is retained alongside the dirty/absent-plan arm. Add an acceptance line that after the rewrite Step 1c still routes every artifact combination (the "never a dead end" property holds), and that the 527-551 boundary block no longer instructs the user to re-invoke across a boundary on the happy path. This is testable by static read + a worked dry-run of each combination, matching the track's prose acceptance model.

### R3 [suggestion]
**Certificate**: Assumption A1 (by-reference orchestration holds — gate A6)
**Location**: track-2.md `## Invariants & Constraints` ("Constraint (by-reference, hard for D15): if any author spawn returns more than a thin summary, the 4a/4b boundary is retained rather than collapsed") and `## Validation and Acceptance` ("By-reference orchestration is confirmed (author spawns return thin summaries only) before the collapse is applied"); design.md:559-563; `design-author.md:54-56`.
**Issue**: The by-reference contract is real and built — `design-author.md §54-56` mandates "return **only a thin summary** ... Never return the drafted content in your reply." So the *agent definition* enforces gate A6. The residual risk is that "confirmed before the collapse is applied" is a runtime property of the live `Agent`-tool harness, not a static property of the prompt: the same staged `edit-design/SKILL.md:543-545` flags that the byte-identical-prompt assumption "needs a live confirmation when wired", and the prior-episode handoff records gate A7's warm-up wait mechanism as deferred for the same reason. Track 2's acceptance asserts the confirmation but does not say *how* it is performed in a prose-only, static-read acceptance model (there is no live `/create-plan` run inside this branch — the routine is staged and non-live until Phase 4 promotion per S7). So "confirmed" risks degrading to "asserted by reading the agent definition", which is weaker than the gate's intent (an author that *obeys* its definition vs. a harness that *enforces* it). Likelihood low (the definition is unambiguous); impact: if the live harness ever surfaces the full draft despite the definition, the collapse silently regresses context isolation with no static check that would have caught it.
**Proposed fix**: Make the acceptance for concern 3 name the confirmation method available on a staged branch: a static read that the author definition's by-reference clause is intact and that the `create-plan` Step 4b wiring passes `output_path` + partial-fetches (so the orchestrator never receives the draft), plus an explicit note that the live-harness confirmation is a Phase-4-promotion / first-live-run gate carried forward as a deferred item (the same shape Track 1 used for gate A7). This keeps the "retain the boundary if by-reference cannot hold" fallback honest rather than presuming the live property from a static read.

### R4 [suggestion]
**Certificate**: Assumption A2 (the fan-out cache warm-up's wait mechanism — gate A7 — applies to the Step 4b track loop too)
**Location**: track-2.md `## Plan of Work` concern 1 ("run the same dual-clean inner loop") against the staged `edit-design/SKILL.md:527-545` warm-up block and the live `create-plan/SKILL.md:711-746` Step 4b cold-read spawn.
**Issue**: Concern 1 reuses "the same dual-clean inner loop", which includes the fan-out cache warm-up. The staged loop (`edit-design/SKILL.md:534-537`) explicitly defers the wait mechanism to wiring (gate A7) and makes the warm-up "a cost lever, not a correctness dependency ... the loop must produce correct dual-clean output with the warm-up disabled." When concern 1 wires this loop into `create-plan` Step 4b, the Step 4b path inherits the same deferral: there is no harness-confirmed non-blocking fixed-delay mechanism named, and the prior-episode handoff records this as deferred. The risk is not correctness (the disabled path is correct by design) but that Step 4b's acceptance could over-claim a working warm-up. Likelihood low; impact: a Phase B implementer might block on wiring a warm-up that the harness cannot express, or mark the warm-up as acceptance-blocking when the design makes it optional.
**Proposed fix**: Add a one-line note to concern 1 (or the Phase A decomposition) that the Step 4b loop inherits gate A7's warm-up deferral: the warm-up is optional, the warm-up-disabled (N-cold-prefix) path is the correctness baseline, and acceptance for Step 4b does not require a working warm-up. This mirrors the staged `edit-design` treatment so the two loops stay consistent.
