## Findings

### R1 [should-fix]
**Certificate**: A1 (the D1+D3 wiring-pair constraint)
**Location**: `track-2.md` §Plan of Work boundaries 1 (A→C `steps-partial`) and 4 (track-advance `decomposition-pending`); `design.md` D1/D3; the constraint is stated in `track-2.md` §Invariants & Constraints and D3's "wiring-pair" bullet.
**Issue**: The track correctly names the D1+D3 pair as the hard constraint: both the A→C `steps-partial` append (boundary 1, two sites in `track-review.md` at `:596` and `:1048`) and the track-advance `decomposition-pending` append (boundary 4, `track-code-review.md:1409`) MUST land together, or a `phase=C` track ships with no `substate` and silently triggers the roster fallback when it should not — the exact silent-default failure mode this branch exists to fix. The track asserts the constraint holds "because both append sites are in this track." That is true at the *track* granularity but the risk lives one level down: decomposition (the next Phase A sub-step) has not happened yet, and nothing in the track file forbids splitting boundary 1 into one step and boundary 4 into another. Two single-file steps — "wire `track-review.md`" and "wire `track-code-review.md`" — are the natural decomposition (the §Interfaces in-scope list is organized per file), and the track is doc-only with no test gate that would fail a partial landing (Validation is review-of-edits, the Track 1 tests already pass dormant). If the steps land in separate commits and a resume happens between them, an A→C-appended track that has not yet had its track-advance site wired would still route correctly for the *current* track (its own A→C append set `steps-partial`); the break only manifests for track N+1, whose `decomposition-pending` append site was not yet wired — leaving N+1 with no `substate` on a `phase=C` ledger. The blast radius is the resume routing for any track that follows a wired track; the likelihood is moderate because the file-per-step decomposition is the obvious one and the constraint is invisible to the per-step review (a doc-only diff has no failing test to catch it).
**Proposed fix**: Add a decomposition constraint to `## Concrete Steps` guidance (or carry it forward as a step-level note the decomposer must honor): the D1 A→C append and the D3 track-advance append must land in the **same step**, or in steps that share one mergeable commit, so no intermediate commit leaves track N+1 without a `decomposition-pending` append while a wired track is already advancing past A→C. The simplest satisfaction is one step covering both `track-review.md` and `track-code-review.md` append wiring; if the decomposer splits per file, it must order the `track-code-review.md` track-advance site no later than the `track-review.md` A→C site and note in both step descriptions that the pair is co-dependent. State the constraint where the decomposer will read it, not only in §Invariants where it reads as already-satisfied.

### R2 [should-fix]
**Certificate**: A2 (boundary 3 ride-site fires on every iteration)
**Location**: `track-2.md` §Plan of Work boundary 3 (`review-done-track-open`, "around `:743`"); `track-code-review.md:732-746` (the per-iteration Progress commit) vs `:826-:834` (the `Track complete` Progress entry) vs `:1401-:1423` (the post-approval `Mark <track> complete` commit).
**Issue**: The track plans to add `--substate review-done-track-open` to "the pre-approval code-review-complete Workflow-update commit (around `:743`, the Progress entry recording the passed iteration)." Tracing `track-code-review.md`, the commit at `:743` is the **per-iteration** Progress commit — `Track-level code review iteration N complete (N/3 iterations)` — and it fires once per review iteration, including iterations that did **not** pass (the gate-check found more findings and the loop continues). There is no separate commit for the `[x] Track complete` Progress entry at `:826-:834`: that entry is written to disk uncommitted and is first committed only by the post-approval `Mark <track> complete` commit at `:1423` (boundary 4). So the only *committed* boundary that is strictly before user approval is the per-iteration commit, and "the passed iteration" is not a distinct commit — it is whichever iteration happens to be the last one. An implementer that reads "add `--substate review-done-track-open` to the `:743` commit" literally will append it on **every** iteration's commit, so a two-iteration review records `review-done-track-open` at the end of iteration 1 (before iteration 2 even runs). A resume after that iteration-1 commit would route the track as review-passed/awaiting-completion while review is still in progress, re-rendering a completion panel for an un-reviewed-clean track. The track's prose ("the Progress entry recording the passed iteration") shows the *intent* is correct, but the cited ride site does not distinguish passing from non-passing iterations, so the wiring is under-specified in exactly the way that produces the wrong state.
**Proposed fix**: Make boundary 3's append conditional on the gate clearing, not on every iteration commit. Two viable shapes: (a) append `--substate review-done-track-open` only in the step-6 path (`:826`, "when all reviews pass") and give that path its own committed boundary — but step 6 currently writes only to disk and is committed downstream at `:1423`, so this would need a new pre-approval commit, enlarging scope; or (b) keep the ride site at the per-iteration commit but gate the append to fire only on the iteration whose gate-check returns all-clear (the loop-exit-with-pass branch), not on iterations that continue the loop. Either way, the step description must state the *condition* (gate cleared / all reviews pass), not just the line number `:743`, because the line `:743` commit is iteration-generic. Confirm during decomposition which commit is the unambiguous "review passed, approval pending" boundary; if none exists pre-approval without a new commit, surface that as a scope question (it mirrors the boundary-2 "needs a new commit" situation).

Add a guard that the commit + append fire only when every roster step is `[x]`/`[~]` (the normal completion path), not on a context-warning or two-failure early exit where steps remain `[ ]`. Cross-reference the design's "incidentally fixes a latent wart" note so the reviewer understands the previously-uncommitted flip is now intentionally committed, and cite Track 1's `steps-done-review-pending` ledger-path test as the verification that the slug this commit writes resolves correctly.

### R4 [suggestion]
**Certificate**: A3 (no enum-membership guard on `--substate`)
**Location**: staged `workflow-startup-precheck.sh:1697` (`reject_bad_ledger_value "substate" ... bare`); `track-2.md` all five append sites; Track 1 deferred finding WI2 (recorded in `track-1.md` §Track completion).
**Issue**: Track 1 landed `--substate` validating only bare-token-ness — no enum-membership check (WI2, deferred and explicitly declined for Track 2 at Pre-Flight). Track 2 is the first and only writer of `substate` values. A typo in any of the five hard-coded slugs (`steps-partial`, `steps-done-review-pending`, `review-done-track-open`, `decomposition-pending`) passes validation and is written to the ledger verbatim; the resolution at `workflow-startup-precheck.sh:1983` then emits that bad slug directly as `state.substate` (`if [ -n "$substate" ]; then ... return 0`), so a misspelled slug routes as an unknown sub-state with no diagnostic. The likelihood is low — the slugs are written into prose once and reviewed — and the impact is bounded (a single mistyped append in one document, caught the first time that boundary is exercised on a resume), so this is a suggestion, not a blocker. But because the enum guard was declined and Track 2 is the sole writer, the typo risk has no backstop anywhere in the system.
**Proposed fix**: No code change is in Track 2's scope (the guard is Track-1 territory and was declined). Mitigate in the track instead: in the §Validation acceptance, add an explicit check that each of the five appended slugs is byte-identical to the four canonical slugs the design enumerates (`design.md:198-199` / the script's `:1043-:1062` comment), and that the append-cadence-table review confirms each boundary writes the slug its row names. This makes the typo a review-gate catch rather than a runtime-only catch. Optionally note WI2 in the track's Surprises log so a future Track-1 follow-up has the context.

### R5 [suggestion]
**Certificate**: A4 (replan-revert substate is forward-hygiene, never read on the replan resume)
**Location**: `track-2.md` §Plan of Work boundary 5; `inline-replanning.md:244-268` (the `--phase 0` reset + commit); staged `workflow-startup-precheck.sh:1952-1959` (the `phase=0` arm returns a null substate before the `phase=C` substate read).
**Issue**: Boundary 5 appends `--substate steps-partial` on the inline-replan commit, "distinct from the existing `--phase 0` reset append." Tracing the resolution: the replan path appends `--phase 0` (`:249`), which is last-value-wins, so the resolved phase becomes `0`. `determine_state_from_ledger` handles `phase=0` in the `0 | A | D | Done` arm (`:1952`), emitting `{phase:"0", substate:null}` and returning **before** the `phase=C` arm that reads `substate` (`:1981`). So the `substate=steps-partial` written on the replan commit is **never read** on the replan resume itself — the next session routes to State 0 (Phase 2 plan re-review) on the `phase=0` signal, regardless of the substate value. The append is forward-consistency hygiene: it keeps the within-track signal correct for when the reopened track later re-advances to `phase=C` and a `substate` read fires again. This is not a defect in the plan — the track's framing ("keeps the within-track signal consistent when the replan reopens a closed track") is accurate — but the wording could mislead an implementer into thinking the substate drives the replan resume, or into dropping the still-required `--phase 0` append (which is what actually routes the resume). The risk is a subtle one: omitting `--phase 0` would silently break the replan's State-0 reset, and the substate append alone would not compensate.
**Proposed fix**: In boundary 5's step description, state plainly that `--substate steps-partial` is appended **in addition to** the existing `--phase 0` reset (same commit, same line or adjacent line), and that the `--phase 0` append remains the routing signal for the replan resume; the substate is forward-consistency only and is not read until the reopened track re-advances to `phase=C`. This prevents an implementer from treating the two as alternatives. No behavior change is needed beyond keeping both appends.
