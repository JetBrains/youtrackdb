## Findings

### A1 [blocker]
**Certificate**: Challenge: D(staging) — Consequence ("The track must NOT be a single-step `risk:high` track — Phase A must decompose it into ≥2 steps")
**Target**: Decision Log entry "Use §1.7 staging, not the prose-rule opt-out" → its recorded planning Consequence, cross-referenced by the "Self-trap" Surprise.
**Challenge**: The decision's stated escape hatch — "Phase A must decompose it into ≥2 steps so Phase C's track pass runs the full selection on the cumulative staged diff" — is presented as a clean mitigation, but it collides head-on with the decomposition rulebook. `track-review.md` §Step Decomposition, **High-risk isolation** (lines 841-844): "Put each HIGH-category change in its own `high`-tagged step, sized by the change itself with no file cap — a HIGH change that legitimately spans many files stays one step so its step-level dimensional review sees the whole change at once." And the Coherence rule (lines 832-840): "for `high` steps coherence is mandatory: a `high` step is one coherent, logically continuous change." This branch IS one coherent HIGH change: a single control-flow correction (full-selection-at-the-step + premise fix) threaded across four files that all state one rule. The decomposer's rules therefore *push toward* one HIGH step, not away from it; "file count alone never forces a split" (line 840). To get ≥2 steps the decomposer would have to either (a) split a coherent HIGH change against the mandatory-coherence rule, or (b) manufacture a second `low`/`medium` step out of nothing — but a `low`/`medium` slice of a gate-change is itself a control-flow edit, which the prose-only cap (`risk-tagging.md:299-309`) explicitly refuses to cap to `low`. So the mitigation the staging decision leans on is not freely available; the natural decomposition of this change is exactly the single-step-`high` track the bug fires on. Under the *live* (still-buggy) rules that staging deliberately keeps in force, this branch's own Phase C would then skip the track pass and — because the diff is `.claude/workflow/*.md` — step-level review fires zero workflow reviewers (`review-agent-selection.md:145-155`). The decision's consequence is not just a planning note; it is the thing that determines whether this branch reviews itself at all, and as written it is not reliably reachable.
**Evidence**: `track-review.md:841-844` (High-risk isolation, no file cap, "stays one step"), `track-review.md:832-840` (mandatory coherence for `high`, "file count alone never forces a split"), `risk-tagging.md:299-309` (prose-only cap does not apply to a gate/dispatch edit), `review-agent-selection.md:145-155` (a `.claude/workflow/*.md` high step draws zero step-level reviewers).
**Proposed fix**: Resolve before any artifact derives. Either (1) record an explicit decomposer-side justification that this coherent HIGH change MAY be carved into two `high` steps for the express purpose of forcing the Phase C track pass — and surface the tension with High-risk isolation so Phase A does not later reject the split as incoherent; or (2) reconsider the opt-out (see A3), which self-applies the fix and makes the mitigation moot; or (3) record that the single-step-`high` track is acceptable because the staged-path normalization + always-run reviewers (consistency + context-budget) still fire at the track pass — but that contradicts the §1.7 staging premise that keeps the *live* skip-gate buggy for this branch. The current text asserts the ≥2-step route as if it were free; it is not, and the gate should not clear until the decision records which of these holds.

### A2 [should-fix]
**Certificate**: Assumption test: "If Phase A decomposes it to one `risk:high` step …" (the Self-trap Surprise assumes the step is `high`).
**Target**: Assumption embedded in the "Self-trap" Surprise — that a single step on this change is tagged `high`.
**Challenge**: The self-trap analysis is correct in its conclusion but skips the one rule that makes the conclusion sound, leaving a reader unable to verify it. The Surprise asserts the step would be `risk:high` and that step-level review would then fire zero workflow reviewers. Both are true *only because* the change is a gate/dispatch/control-flow edit, which the prose-only cap (`risk-tagging.md:285-309`) refuses to cap to `low`: "A workflow-machinery step that edits ONLY prose (no hook/script/settings change AND no gate/dispatch/schema change) is at most `low`." A naive reader could reasonably ask "these are `.md` edits — why isn't this prose-only/`low`, in which case step-level review is skipped entirely and the track pass runs the full selection, and there is no trap?" The answer is that this edit fails the second half of the qualifier (it changes a gate/dispatch rule), so it stays HIGH. That is the linchpin of the whole self-trap, and the log never states it. **Verdict: HOLDS** — the assumption is correct — but the rationale is incomplete and the gate's job is to force the rationale onto the log.
**Evidence**: `risk-tagging.md:285-309` (prose-only cap: "no gate/dispatch/schema change" qualifier; "a control-flow-changing edit to the same file is HIGH"), `risk-tagging.md:311-315` (cap is orthogonal to the workflow-machinery file-set predicate).
**Proposed fix**: In the Self-trap Surprise (or the staging Decision), add one sentence: this change is NOT prose-only-capped because it edits gate/dispatch control flow (the `risk-tagging.md` prose-only cap explicitly excludes "gate/dispatch/schema change"), so the step is unavoidably `high` — which is what makes the single-step-high trap fire rather than the prose-only `low` path that would sidestep it.

### A3 [should-fix]
**Certificate**: Challenge: D(staging) — "Use §1.7 staging, not the prose-rule opt-out".
**Target**: Decision Log entry "Use §1.7 staging, not the prose-rule opt-out".
**Challenge**: The decision rejects the §1.7(k) prose-rule opt-out on the ground that the opt-out "forbids touching `step-implementation.md`." Argue the rejected alternative: the opt-out's defining property is that the fix is edited **live** and self-applies before this branch's own review runs — which dissolves the entire self-trap (A1, A2) at the root. Under staging, the decision itself records the cost: "Live rules stay buggy during this branch, so the self-trap … is real," and it then has to invent the fragile ≥2-step mitigation (A1) to work around a trap it chose to keep live. The opt-out pays none of that: edit live, the corrected rule is in force, this branch's single-step-`high` track is reviewed correctly by the very rule it adds. The log's counter — that the opt-out can't touch `step-implementation.md` and would leave sub-step 4a's inline summary un-qualified — is real but narrower than the decision implies: sub-step 4a (lines 433-450) only *restates* the `hook-safety, prompt-design` step-level subset and points at §Step-level routing as the single source of truth; the operative widening can land in `review-agent-selection.md` / `code-review-protocol.md` (which the opt-out CAN edit) with sub-step 4a left as a pointer. The decision treats "must edit `step-implementation.md`" as decisive when the edit-surface analysis itself says the single source of truth is §Step-level routing, not sub-step 4a. **Survival: WEAK** — staging is defensible (predictable live rules, one consistent rule body for reviewers), but the rationale understates that the rejected opt-out eliminates the self-trap the chosen path must then mitigate.
**Evidence**: `step-implementation.md:433-450` (sub-step 4a restates the subset and defers to §Step-level routing as single source of truth), `review-agent-selection.md:104-106` ("the dispatch points in step-implementation.md (step) and track-code-review.md (track) consume it" — §Step-level routing is the source, the dispatch sites consume), research-log §Decision Log Consequence (live rules stay buggy → self-trap is real).
**Proposed fix**: Strengthen the staging rationale to weigh the dissolved-self-trap benefit of the opt-out explicitly, and state why predictable-live-rules + one-consistent-body still wins despite it — or, if the self-trap mitigation (A1) proves not freely reachable, flip to the opt-out. Either way the decision's "forbids touching `step-implementation.md`" ground should be qualified: the single source of truth is §Step-level routing, and `step-implementation.md` carries only a consuming restatement.

### A4 [should-fix]
**Certificate**: Open-question challenge: "Tier — `minimal` vs `lite`".
**Target**: `## Open Questions` entry "Tier — `minimal` vs `lite`" (and its relation to the `## Decision Log` tier entry).
**Challenge**: The tier choice is the single most load-bearing decision in a `minimal`-tier branch: it sheds `design.md` and the full plan, and it sets the entire downstream artifact set. The Decision Log entry "Tier = minimal" records the *Gate verdict* (Gate 1 = no, Gate 2 = single track) and the adversarial-lens addition, but the *footprint evidence* that justifies single-track — "a localized ~4-file rule correction … one cohesive logical unit well under the single-track footprint bound" — lives only in the Open Questions note, which is still phrased as pending ("Pending user confirmation at Step 4") even though the Decision Log says "User confirmed." An unresolved-looking Open Question that bears on a load-bearing decision is, per the gate's own rule, at least a `should-fix`: it must be folded into the Decision Log or explicitly waived. Here it is half-folded — the verdict moved to the Decision Log but the justifying footprint count and the "single cohesive unit" claim did not, and the Open Questions entry was never struck through or marked RESOLVED the way the staging Open Question was (lines 52-53 show the pattern: "RESOLVED — staging; see Decision Log entry").
**Evidence**: research-log §Open Questions lines 49-50 (tier question still reads "Pending user confirmation") vs §Decision Log lines 22-25 ("User confirmed" but no footprint count recorded); research-log lines 52-53 (the staging Open Question shows the correct RESOLVED-and-point-to-Decision-Log pattern this one is missing).
**Proposed fix**: Mark the "Tier — `minimal` vs `lite`" Open Question RESOLVED and point it at the Decision Log entry (mirroring the staging entry's pattern), and move the load-bearing footprint claim ("~4 files, one cohesive logical unit, well under the single-track footprint bound") into the Decision Log "Tier = minimal" entry so the justification for shedding `design.md` and the plan is recorded where the decision lives, not in a still-open-looking question.

### A5 [suggestion]
**Certificate**: Simplification / scope challenge: edit-surface "single source of truth" claim.
**Target**: `## Surprises & Discoveries` entry "Edit surface — four workflow files" → its claim that "The single source of truth for the operative widening rule is §Step-level vs track-level routing."
**Challenge**: The edit-surface analysis correctly enumerates four files, but its "single source of truth" framing risks pointing the fix at the wrong primary edit. §Step-level vs track-level routing (`review-agent-selection.md:94-171`) governs *which reviewers fire at a high step* — it is where the "run the full track-pass-equivalent selection at the step" widening would land, so that part of the claim is right. But the *bug itself* — the false "fully reviewed in Phase B" premise and the skip it licenses — is stated in `code-review-protocol.md` §Single-step tracks (lines 55-64, "step-level dimensional review already ran against the identical diff") and `track-code-review.md` §Single-Step Track (lines 102-118, "the step-level review in Phase B already covered the identical diff … fully reviewed in Phase B"). The acceptance criteria name the skip-gate text as the thing whose premise must be corrected. So the fix has two distinct edit obligations: (1) widen the step-level selection (in §Step-level routing), and (2) correct the skip premise (in `code-review-protocol.md` / `track-code-review.md`) to read "valid only once the full selection ran at the step." The log's "single source of truth = §Step-level routing" phrasing could be read as collapsing (2) into (1), leaving the false premise text untouched. **Survival: HOLDS** — the four-file enumeration is complete and correct — but the "single source of truth" label is about the *routing* rule, not the *skip premise*, and conflating them is a real planning hazard.
**Evidence**: `code-review-protocol.md:55-64` (§Single-step tracks states the false premise + skip), `track-code-review.md:102-118` (§Single-Step Track restates "fully reviewed in Phase B"), `review-agent-selection.md:94-171` (§Step-level routing = which reviewers fire at a step, the routing rule). research-log §Surprises "Edit surface" lists all four files but labels only §Step-level routing the single source of truth.
**Proposed fix**: Clarify in the edit-surface entry that §Step-level routing is the single source of truth for *the routing/widening rule*, while the *skip premise correction* is a separate, mandatory edit to `code-review-protocol.md` §Single-step tracks and `track-code-review.md` §Single-Step Track — so decomposition does not drop the premise-correction edit by treating the routing note as the only authoritative site.
