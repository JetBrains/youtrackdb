# Track 3: Edge traversal — `out`, `in`, `both`, `outE.inV`, `inE.outV`, `bothE.otherV`

## Description

Extends the recognized step set with edge-traversal patterns. Adds
handlers in a new `GremlinStepWalker` for **directional, single-label
vertex traversal only** in Phase 1 of this track:

- `VertexStep` with direction OUT/IN and exactly one edge label —
  produces a `SQLMatchPathItem` via `SQLMatchPathItem.outPath`/`inPath`
  helpers populated through `MatchPatternBuilder.addEdge`. The walker
  follows each `addEdge` with `addNode(targetAlias, "V", null, false)`
  to register the target alias in `aliasClasses` so downstream tracks'
  `has`-style filters and class refinements bind correctly.
- Anonymous intermediate vertex aliases — generated by the walker
  under a private prefix (e.g. `$g2m_anon_N`) chosen to be unique
  within the produced pattern. The existing
  `MatchExecutionPlanner.DEFAULT_ALIAS_PREFIX` convention is
  intentionally not reused: it is package-private to
  `internal/core/sql/executor/match` and the
  `MatchExecutionPlanner.assignDefaultAliases` re-assignment never
  runs on the translator's path (consistency review CR4 —
  `buildPatterns` short-circuits when `pattern != null`, which is
  always the case for the `MatchPlanInputs` ctor). The walker also
  runs a **collision pre-flight**: it scans every step's
  `Step.getLabels()` set and declines the entire traversal if any
  user-provided label starts with the anon prefix.
- Boundary alias rebinds from the start-step alias (`$g2m_v0`) to
  the **chain's terminal node alias** so `YTDBMatchPlanStep` projects
  the right column from each result row.

**Scope reduction (iteration 1 corrections — five reviewer-converged
blockers carved out of Phase 1 scope):**

The track originally proposed full edge-traversal coverage including
`both`/`bothE`, `outE.inV` pairing, multi-label edges, and
`bothE.otherV`. Three independent Phase A reviews surfaced
blocker-level mismatches between Gremlin semantics and current MATCH
IR / planner support:

1. **Multi-label edges** (`out("knows","follows")`) — `MatchPatternBuilder.addEdge`
   accepts a single `String edgeLabel`; `SQLMatchPathItem.outPath`
   accepts a single `SQLIdentifier`. There is no IR-level edge-label
   `IN [...]` filter slot. Supporting multi-label requires a
   shared-builder retrofit (variadic `SQLMethodCall.params`) which is
   out of this track's scope.
2. **`both` and `bothE`** — Gremlin emits one row per matched edge
   (multiset); MATCH `MatchEdgeTraverser` deduplicates via `RidSet`.
   Self-loop scenarios (`g_V_hasLabelXloopsX_bothXselfX`) produce
   different cardinalities. Fundamental semantic divergence — needs
   dedicated handling in a follow-on track.
3. **`outE.inV` / `inE.outV` pairing with edge step label** — when
   `g.V().outE("knows").as("e").inV()` is translated, `PatternEdge`
   has no alias slot and `SQLMatchPathItem` cannot carry an edge
   alias, so the user-bound `"e"` would silently disappear and
   downstream `select("e")` would return wrong results.
4. **Edge-returning terminal steps** (`outE`/`inE`/`bothE` without
   paired vertex step) — `BoundaryOutputType` only declares
   `ELEMENT` and `YTDBMatchPlanStep.projectElement` is hardcoded to
   `result.getVertex(boundaryAlias)`. Edge translations would
   produce null traversers.
5. **`bothE.otherV()`** — `otherV` is "endpoint not equal to source",
   not the same as `inV`. `SQLMethodCall.graphMethods` has no
   `otherV` primitive; faithful translation would need an IR
   construct that does not exist.

This track therefore lands the load-bearing walker abstraction +
directional single-label `out(label)`/`in(label)` recognition only.
Every other shape declines under D3 all-or-nothing and runs natively
unmodified. Specifically, the walker explicitly declines:

- `both()`, `bothE()` (any direction BOTH on `VertexStep`)
- multi-label `out("a", "b")` (more than one edge label)
- `outE`, `inE`, `bothE` returning Edge class (`VertexStep.getReturnClass() == Edge.class`)
- any `VertexStep` whose `getLabels()` (the step's `as(...)` labels)
  is non-empty when the step is edge-returning
- `EdgeVertexStep` and `EdgeOtherVertexStep` (no peek-ahead pairing
  in this track)

Tracks 4-10 extend this walker via the `StepRecogniser` registry; a
dedicated future track is needed for `both`/`bothE`, multi-label,
edge aliases, edge-returning boundaries, and `otherV` once IR support
or semantic accommodations land.

**Verification methodology change (iteration 1):**

The original "cross-checked against equivalent SQL `MATCH` queries"
verification only proves the translator faithfully encodes MATCH
semantics — it cannot catch Gremlin/MATCH semantic divergences
(self-loops, multi-edges, edge alias). Replaced with **strategy-on
vs strategy-off equivalence tests**: a parameterised JUnit test
class runs each Gremlin traversal twice — once with the strategy
disabled (native pipeline) and once enabled (translated) — and
asserts result-set equality (multiset where Gremlin is multiset,
set where Gremlin is set) plus presence of `YTDBMatchPlanStep` in
the translated case. The strategy's top-level try/catch (which
declines back to native on unexpected throw) cannot mask walker
bugs because absence of the boundary step is asserted to be a test
failure when the shape is recognized.

**Gate replacement** (carryover from prior track's all-or-nothing
revision): the strategy currently declines any traversal whose
`getSteps().size() > 1` (the gate lives in `GremlinToMatchTranslator`,
not the strategy itself). This track replaces that hard-decline with
a `GremlinStepWalker` that classifies each step via a registry of
`StepRecogniser` instances and declines the entire traversal only
when at least one step is unrecognized. The walker becomes the
load-bearing entry point for Tracks 4-10 as well — they extend the
registry rather than touching the walker proper.
