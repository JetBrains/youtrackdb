## Findings

### WI1 [Minor] — §1.7(k) staging-consumer list names "staged-delta prep", a mechanism with no separately documented default-live consumer

- **File:** `.claude/workflow/conventions.md` (lines 1204-1206)
- **Axis:** phase output → next-phase input (consumer handshake completeness)
- **Cost:** the opt-out's load-bearing "every staging consumer already defaults to live with no edits" claim rests on four named mechanisms; three resolve to a documented default-live consumer, one ("staged-delta prep") does not resolve to any consumer the reviewer can confirm the complement against — a reader cannot verify that disabling it is a no-op under the marker-absent path.
- **Issue:** §1.7(k) lists the staging mechanisms the opt-out disables as "write-routing, the pre-commit live-path gate, staged-delta prep, the Phase 4 promotion guard." Three have a confirmable default-live complement: write-routing and the promotion guard are the two signals in §1.7(c) (the implementer writes live when the marker is absent; the directory-presence guard skips when no `.claude/` subtree is present), and the pre-commit gate defaults to live at `implementer-rules.md:298` ("When the marker is absent (the default), the implementer writes to live paths normally"). "Staged-delta prep" appears nowhere in `.claude/workflow/`, `.claude/skills/`, or `.claude/agents/` except inside this clause itself — there is no separately named "staged-delta prep" consumer with a documented marker-absent path. It is plausibly subsumed by the reads-precedence / write-routing machinery (§1.7(d)/(e)), but the term as written points at no confirmable consumer, so the "defaults to live with no edits" assurance is unverifiable for that one item. This does not strand the LLM: the actual default-live mechanism is marker-absence, which every real consumer already honors, so the opt-out works regardless. It is a documentation-completeness gap in the consumer enumeration, not a behavioral hole.
- **Suggestion:** either drop "staged-delta prep" from the list (the three confirmable consumers fully cover the staging surface the opt-out disables), or add a half-clause naming the consumer it refers to (e.g. "staged-delta prep under §1.7(e) write-routing") so a future opt-out branch's reviewer can confirm the default-live complement without searching for a phantom consumer.

### WI2 [Minor] — §1.7(k) stamp-advance trigger is durable, but the cross-track "after Track 2, not now" timing lives only in the track episode

- **File:** `.claude/workflow/conventions.md` (lines 649-663)
- **Axis:** cleanup and idempotency (deferred-cleanup trigger completeness)
- **Cost:** a future opt-out branch reading only §1.7(k) learns to run `/migrate-workflow` "after the last commit touching the drift-gate pathspec" but is given no guidance that, on a multi-track branch, "last" means the branch's final such commit across all tracks, not the current track's — a single-track reading risks an early migrate that re-drifts on the next track's workflow commit.
- **Issue:** §1.7(k)'s "Mandatory stamp-advance" paragraph correctly names the trigger as "after the last commit touching the §1.6(b) SHA-computation pathspec (`.claude/workflow`, `.claude/skills`, or `.claude/agents`)" and correctly excludes out-of-pathspec edits (`house-style.md`, hooks). That rule is durable and complete for a single-track opt-out branch. But on this branch the operative timing — that the migrate fires after Track 2, not after Track 1 even though Track 1 lands the last drift-gate-pathspec commit *of Track 1* — lives only in the Step 5 episode's `**Critical context**` ("the last drift-gate-pathspec commit on the branch — which is after Track 2, not now ... The Phase C session should Suppress the drift gate, not migrate yet"). The rule says "last commit on the branch," which is technically correct, but a reader executing track-by-track has no in-rule cue that "last on the branch" can fall in a later track, and the episode that carries the cue is append-only bookkeeping a future opt-out branch will not have. For *this* branch the resume path is complete (the episode carries it). The gap is durability: the next opt-out branch that spans multiple tracks gets the rule without the multi-track timing nuance.
- **Suggestion:** add one clause to §1.7(k) making the multi-track scope explicit, e.g. "on a multi-track opt-out branch, 'last commit' is the final drift-gate-pathspec commit across all tracks; Suppress the gate on every intervening Phase C and migrate once, after the branch's last such commit." This turns the per-branch episode note into the durable rule the (l) bootstrap paragraph already promises for "every opt-out branch after."

## Evidence base

#### C1 — "staged-delta prep" consumer resolution

Confirmed-with-caveat. Searched `.claude/workflow/`, `.claude/skills/`, `.claude/agents/` for `staged-delta` / `staged delta` / `delta prep`: the only hit is `conventions.md:1205` inside §1.7(k) itself. The other three named mechanisms each resolve to a consumer with a documented marker-absent (default-live) path: write-routing and the Phase 4 promotion guard are the two §1.7(c) detection signals (`conventions.md:944-977`), and the pre-commit live-path gate defaults to live at `implementer-rules.md:298`. The complement (marker absent ⇒ live) is therefore confirmable for three of four; "staged-delta prep" has no separately documented consumer, so the finding is a documentation-completeness gap, not a behavioral one — the marker-absence default that every real consumer honors makes the opt-out work regardless.

#### C2 — stamp-advance multi-track timing

Confirmed. §1.7(k) (`conventions.md:649-663`) defines the migrate trigger as "after the last commit touching the §1.6(b) SHA-computation pathspec," and §1.6(b) (`conventions.md:638-660`) defines that pathspec as `git log ... -- .claude/workflow .claude/skills .claude/agents` — so the rule's trigger and pathspec are internally consistent and the `/migrate-workflow` §4.8 final-batch resume path exists (`migrate-workflow/SKILL.md:598`, with killed-mid-batch recovery documented at `:607`). The durable rule is complete for a single-track branch. The multi-track timing ("after Track 2, not now; Suppress every intervening Phase C") appears only in the Step 5 episode `**Critical context**` block (`plan/track-1.md`, Episodes §Step 5), which is append-only bookkeeping a future branch will not inherit. The (l) paragraph promises (l) is "the durable wiring for every opt-out branch after," so folding the multi-track timing into §1.7(k) matches the stated intent.
