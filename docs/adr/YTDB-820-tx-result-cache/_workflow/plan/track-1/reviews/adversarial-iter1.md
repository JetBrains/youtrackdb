## Findings

### A1 [blocker]
**Certificate**: V1 — Invariant I1 "cache cleared on every tx-end path" via a `clear()` in `beginInternal`.
**Target**: Plan of Work step 10 ("`clear()` in `beginInternal` and `clearUnfinishedChanges`"); Integration Points (`beginInternal` line 164).
**Challenge**: The plan says cache `clear()` is added to `beginInternal`, but does not say it must sit inside the `txStartCounter == 0` guard. `beginInternal` only runs its reset block (including `localCache.clear()` at `FrontendTransactionImpl.java:182`) when `txStartCounter == 0` (the FIRST begin). The `txStartCounter++` and the early-return-on-nested branch live OUTSIDE that guard. An unconditionally-placed cache `clear()` fires on every nested `begin()` (`DatabaseSessionEmbedded.java:1350`, taken when `currentTx.isActive()`), wiping a cache the outer tx already populated — and if a live view is iterating, clearing its pinned entry mid-iteration breaks I9/I7. Worse, `query()` itself calls `beginReadOnly()` (line 628); `beginReadOnly` is a no-op when a tx is active (line 1363-1364), so inside a user tx the clear is dodged — but the very first `query()` that opens an implicit read-only tx with no prior user `begin()` runs `beginInternal` with `txStartCounter==0`, populates, returns a view, and a SECOND bare `query()` re-enters `beginReadOnly`→ no-op (tx still active), so it does NOT clear. The hazard is solely the nested explicit `begin()` and any future caller that calls `beginInternal` with `txStartCounter>0`.
**Evidence**: `FrontendTransactionImpl.beginInternal` (lines 164-204): reset logic gated on `txStartCounter == 0` (line 174), counter bumped unconditionally at line 194. `DatabaseSessionEmbedded.begin()` line 1350 calls `currentTx.beginInternal()` on the already-active path. `beginReadOnly()` line 1363 returns early when active.
**Proposed fix**: Decomposition must specify the cache `clear()` is placed INSIDE the `txStartCounter == 0` block of `beginInternal`, co-located with `localCache.clear()`, so it mirrors the existing first-begin-only reset and never wipes mid-tx on a nested begin. Add a test: open an explicit tx, populate the cache, call `begin()` again (nested), assert the cache survives the nested begin. Alternatively, drop the `beginInternal` clear entirely (a fresh tx instance starts with a null/empty cache, so the only value is tx-instance reuse) and rely solely on the tx-end `clear()`.

### A2 [should-fix]
**Certificate**: V2 — Invariant I1 violation-scenario for "exception-during-iterate → size()==0".
**Target**: Invariant I1; Validation and Acceptance bullet 4 ("exception-during-iterate ... leave `cache.size()==0`").
**Challenge**: An exception during iterate is thrown inside `CachedResultSetView.next()`, which the CONSUMER calls AFTER `query()` has already returned. By then `query()`'s try/catch (`DatabaseSessionEmbedded.java:642-645`, which calls `rollback()` on exception) has exited — it only guards the parse-and-execute prefix, not consumer-paced iteration. An iterate-time exception propagates to the consumer's caller and triggers NO automatic `clear()`/`rollback()`. The cache stays non-empty until the next genuine tx-end path. So the literal claim "exception-during-iterate leaves `size()==0`" is only true if the test wraps iterate in a tx that then commits/rolls back (which fires `clear()` via `FrontendTransactionImpl.clear()` line 972). The invariant the cache must actually uphold is narrower and harder: an exception mid-`stream-pull` inside `next()` must leave the shared `IdempotentExecutionStream` and the entry's pause/resume counters in a state where a RE-query of the same key does not return a truncated/duplicated result.
**Evidence**: `query()` catch at `DatabaseSessionEmbedded.java:642-645` wraps only parse+execute, not the returned view's iteration. `clear()` (the cache sink) is reached only from `close()` (line 949) and `rollbackInternal()` (line 385), both tx-end. `closeActiveQueries` (line 3431) runs at `clear()` line 973, before `clearUnfinishedChanges` (line 998) — confirming the plan's ordering claim, but it fires only on tx-end, not on a stray iterate exception.
**Proposed fix**: Reword I1 and the acceptance bullet to "an exception during iterate, followed by tx-end (commit or rollback), leaves `size()==0`." Add a distinct test asserting that after an iterate-time exception (e.g. a storage read failure mid-stream-pull), re-querying the same `CacheKey` within the still-open tx either re-drives cleanly or surfaces a consistent failure — never a partial/duplicated result from a half-consumed shared stream.

### A6 [blocker]
**Certificate**: V3 — Invariant I10 violation-scenario for ORDER BY + LIMIT under in-tx DELETE.
**Target**: Invariant I10/I4; Plan of Work step 4 ("SKIP/LIMIT → K0_NONE check runs first"); Clarifications.
**Challenge**: `OrderByStep` is a blocking materializer; with `maxResults` set (derived from SKIP+LIMIT) it uses a bounded min-heap that keeps only the top-N rows and DISCARDS every lower-ranked row immediately (`OrderByStep.java:153-158`). So for `SELECT … ORDER BY x LIMIT n`, `entry.results` would contain only n rows; rows ranked n+1, n+2, … were never materialized. The correctness floor I10 demands the cached view equal a fresh uncached `query()`. Construct: cache `ORDER BY x LIMIT 10` (rows R1..R10 stored; R11 discarded). In-tx DELETE R5. A fresh uncached query now returns R1..R4,R6..R11 — R11 must surface. But the cache cannot produce R11: it was never stored and the delta only holds the in-tx mutation (the DELETE of R5), not the un-materialized R11. Result cardinality drops to 9 where fresh returns 10 → I10 violated. The plan's ONLY defense is that `classify` checks `stmt.skip != null || stmt.limit != null` FIRST and routes to K0_NONE (design.md:325), whose version-gate invalidates the whole entry on any mutation, forcing a fresh re-execute. This holds — IF the classify ordering is implemented exactly so and never regresses. The invariant is therefore entirely dependent on one ordering check; there is no defense-in-depth if a later refactor reorders `classify` or a new RECORD-eligible-but-LIMIT-bearing shape slips through.
**Evidence**: `OrderByStep.initBoundedHeap` (lines 130-169) keeps at most `maxResults` and discards the rest; `init` (line 114) takes the bounded path whenever `maxResults != null`. Design.md:325 "SKIP/LIMIT routes to K0_NONE … Classify checks `stmt.skip != null || stmt.limit != null` before any shape-specific check." Track step 4 "SKIP/LIMIT → K0_NONE check runs first."
**Proposed fix**: Make the SKIP/LIMIT-first ordering a hard, tested invariant of `ShapeClassifier`, not a comment. Add a unit test asserting `classify(SELECT … ORDER BY x LIMIT n)` returns `K0_NONE` (and `SKIP`-only, and combined), and an equivalence test: cache `ORDER BY x LIMIT n`, DELETE a top-N cached row, assert the second view's RID sequence equals a parallel uncached `query()` (proving the K0_NONE re-execute, not a stale top-N replay). Document in `classify` that the SKIP/LIMIT gate MUST precede shape detection precisely because the blocking-heap discards beyond-N rows and the RECORD delta-build cannot recover them.

### A7 [should-fix]
**Certificate**: V4 — assumption test on view/stream lifetime under `WeakValueHashMap` activeQueries.
**Target**: Invariant I3 ("paused stream lifetime ≤ its `CachedEntry`"); Decision D4/D9; Decision D9 wrapper threading.
**Challenge**: `activeQueries` is a `WeakValueHashMap` in embedded mode (`DatabaseSessionEmbedded.java:256`). `closeActiveQueries` (line 3431) iterates `new ArrayList<>(activeQueries.values())` — only the still-strongly-reachable entries. A `CachedResultSetView` that the caller stopped holding can be GC'd and silently vanish from `activeQueries` before tx-end, so its `close()` may never run via `closeActiveQueries`. The cache's correctness here hinges on the entry (not the view) owning the `IdempotentExecutionStream`: the stream stays alive as long as the `CachedEntry` is in the map, and is closed by `entry.close()` at cache `clear()`/evict — independent of whether the view was GC'd. That is exactly D9's design, and it survives. But the plan never states that this weak-map behavior is WHY the stream must be owned by the entry; an implementer who instead parks stream lifetime on the view (or decrements `liveViewCount` only in `view.close()`) gets a leaked or prematurely-closed stream when the view is GC'd without `close()`.
**Evidence**: `activeQueries = serverMode ? new HashMap<>() : new WeakValueHashMap<>()` (line 256). `closeActiveQueries` iterates a snapshot of `.values()` (line 3432). Design D9 (plan lines 228-242) threads the wrapper into both the entry and the `LocalResultSet` slot; design line 782 decrements `liveViewCount` in `view.close()` AND on natural exhaustion. A GC'd-without-close view never fires either decrement.
**Proposed fix**: Document in the track (Interfaces or Idempotence and Recovery, Phase A) that (a) the paused stream MUST be owned by `CachedEntry`, not `CachedResultSetView`, because `activeQueries` is a weak-value map and a view may be GC'd without `close()`; and (b) `liveViewCount` is a best-effort pin, so `removeEldestEntry` correctness must not assume every view decrements (a GC'd view leaks one pin until tx-end clears the entry — acceptable, but state it). Reference-accuracy caveat: `WeakValueHashMap` semantics read directly from the field initializer (line 256), not via PSI usage search.

### A8 [suggestion]
**Certificate**: V5 — violation-scenario probing the `addRecordOperation` collapse vs the track's "type reflects the FIRST status" summary.
**Target**: Context and Orientation (`addRecordOperation` bullet, line 510 collapse); Decision D5/D21.
**Challenge**: The track's Context says collapse folds saves "into a single op whose `type` reflects the FIRST status." That is imprecise. The actual collapse switch (`FrontendTransactionImpl.java:591-612`): UPDATED+DELETED → type becomes DELETED (NOT the first status); CREATED+DELETED → DELETED; CREATED+UPDATED → stays CREATED (the `case CREATED` branch only acts on DELETED/CREATED inputs, so UPDATED is a no-op leaving type=CREATED). So "FIRST status" is correct only for CREATE+UPDATE; for UPDATE→DELETE and CREATE→DELETE the type becomes the LATER (DELETED) status. The design's prose (design.md:358) gets this right ("CREATE+UPDATE stays CREATED … `FrontendTransactionImpl.java:604-610`") and the `cached_at_build` dispatch column handles it, so this is not a correctness gap in the design — but the track's one-line gloss could mislead an implementer authoring the dispatch table into thinking type always pins to the first status.
**Evidence**: collapse switch at `FrontendTransactionImpl.java:591-612`: UPDATED→`txEntry.type = DELETED` (line 594); CREATED+DELETED→`txEntry.type = DELETED` (line 606); CREATED+UPDATED falls through with no assignment (lines 604-610). Design.md:358 states the CREATE+UPDATE-stays-CREATED rule correctly.
**Proposed fix**: Tighten the track's Context wording to "collapse folds successive saves on one RID into a single op: CREATE+UPDATE keeps type CREATED, UPDATE→DELETE and CREATE→DELETE become DELETED; the op's `version` is re-stamped to the latest mutation. The `cached_at_build` dispatch column (D21) distinguishes a CREATED op already in the populate snapshot from a true post-populate CREATE." Aligns the track with the design and the code.

### A9 [suggestion]
**Certificate**: A4cert — assumption test on delta-build WHERE re-evaluation context.
**Target**: Decision D1 (per-query WHERE re-eval); Key signatures (`SQLWhereClause.matchesFilters(Identifiable, CommandContext)`).
**Challenge**: `DeltaBuilder.buildForRecord` re-evaluates the cached query's WHERE against each in-tx mutated record via `matchesFilters(Identifiable, ctx)`, which calls `baseExpression.evaluate(currentRecord, ctx)` (`SQLWhereClause.java:50-54`). For a parameterized RECORD query (`SELECT FROM C WHERE prop = :p`), the re-eval must bind `:p` to the SAME value the original cache-miss execution used, or a post-populate record could be wrongly included/excluded relative to a fresh `query(sql, {p: …})`. The `CommandContext` passed to `buildForRecord` (its signature is `(CachedEntry, FrontendTransactionImpl, CommandContext)`) must therefore carry the original query's param map (and any non-LET, non-context-variable bindings). LET/`$current`/`$matched` WHEREs are already routed to K0_NONE, removing the hardest cases — but plain `:param` WHEREs stay RECORD and DO flow through this re-eval. The plan does not state where the delta-build's `CommandContext` comes from or that it must reproduce the original param bindings.
**Evidence**: `matchesFilters(Identifiable, CommandContext)` → `baseExpression.evaluate(currentRecord, ctx)` (`SQLWhereClause.java:50-54`); params resolve from the ctx at evaluate time. Key-signature line in track lists `buildForRecord(CachedEntry, FrontendTransactionImpl, CommandContext)`. NonDeterministicQueryDetector bypasses context-variable WHEREs (design.md:559) but not plain `:param` WHEREs (params are deterministic and belong in the CacheKey).
**Proposed fix**: Decomposition should pin down that `buildForRecord`'s `CommandContext` carries the original query's resolved parameters (e.g. capture them on the `CachedEntry` at populate, or reuse the populate-time ctx), and add a RECORD equivalence test with a parameterized WHERE (`WHERE prop = :p`) where a post-populate UPDATE moves a record across the `:p` boundary, asserting the cached view matches a parallel uncached parameterized `query()`.

#### V1 — Violation scenario: cache `clear()` placement in `beginInternal` wipes a live cache on nested begin
- **Invariant claim**: I1 — the cache is cleared on every tx-end path (and only there); a populated cache survives until the tx ends.
- **Violation construction**:
  1. Start state: explicit user tx active (`txStartCounter == 1`), cache holds entry E for `SELECT FROM C WHERE p`.
  2. Action sequence: caller invokes `session.begin()` again → `DatabaseSessionEmbedded.begin()` line 1349 sees `currentTx.isActive()` true → calls `currentTx.beginInternal()` (line 1350).
  3. Intermediate state: `beginInternal` runs with `txStartCounter == 1` → the `txStartCounter == 0` reset block (lines 174-186) is SKIPPED; only `txStartCounter++` (line 194) runs.
  4. Violation point: IF the new cache `clear()` is placed unconditionally in `beginInternal` (the plan does not say it must be inside the `txStartCounter==0` guard), it fires here, wiping entry E and closing E's pinned stream mid-life.
  5. Observable consequence: a live `CachedResultSetView` over E loses its backing entry → I9 cardinality break or NPE; cache benefit destroyed for the nested-begin pattern.
- **Feasibility**: CONSTRUCTIBLE (nested `begin()` is a supported reentrant-tx idiom; the plan's clear-placement ambiguity is real). Mitigated trivially by placing the clear inside the `txStartCounter==0` block — hence WEAK survival (the decision survives with a one-line placement constraint the decomposition must add).

#### V2 — Violation scenario: exception-during-iterate does not clear the cache
- **Invariant claim**: I1 — "exception-during-iterate → `cache.size()==0`."
- **Violation construction**:

  2. Action sequence: consumer iterates; a storage read fails mid `stream-pull` inside `view.next()`.
  3. Intermediate state: the exception propagates to the consumer's caller. `query()`'s try/catch (`DatabaseSessionEmbedded.java:642-645`) already returned and is not on the stack.
  4. Violation point: nothing calls `clear()` (`FrontendTransactionImpl.java:972`, reached only from `close()`/`rollbackInternal()`). `cache.size()` is still ≥ 1.
  5. Observable consequence: the literal invariant text is false unless a tx-end follows; the real risk is a half-consumed shared stream that a re-query of the same key replays inconsistently.
- **Feasibility**: CONSTRUCTIBLE for the literal wording; the underlying safety (re-query consistency after a mid-stream failure) is the substantive concern. WEAK survival — the intent is sound, the wording and the missing re-query test need fixing.

#### V3 — Violation scenario: ORDER BY + LIMIT bounded-heap discards beyond-N rows, breaking I10 if reached as RECORD

  4. Violation point: a fresh uncached query returns R1..R4,R6..R11 (R11 promoted); the cache's sorted-merge has no R11 to promote (never materialized).
  5. Observable consequence: cached view returns 9 rows, fresh returns 10 → cardinality drop, I10 violated.
- **Feasibility**: INFEASIBLE under the current plan PROVIDED `classify` routes any SKIP/LIMIT to K0_NONE before shape detection (design.md:325; track step 4) — K0_NONE invalidates on the DELETE and re-executes fresh. CONSTRUCTIBLE only if that ordering regresses or a LIMIT-bearing RECORD path is ever introduced. WEAK survival — correctness rests on a single ordering check with no defense-in-depth; promote it to a tested invariant.
