## Findings

### CQ1 [should-fix] Dead production field `CachedEntry.k0InvalidationCount` and its two accessors

- **File**: `core/src/main/java/com/jetbrains/youtrackdb/internal/core/sql/executor/cache/CachedEntry.java` (line 119; accessors 243-249)
- **Issue**: `CachedEntry` carries `private int k0InvalidationCount` plus `getK0InvalidationCount()` and `incrementK0InvalidationCount()`. The K0_NONE strike count that the cache actually uses lives in `QueryResultCache.k0Strikes` (a per-key map on the cache), because invalidation removes the entry and an entry-local counter would be lost — `QueryResultCache.lookup` increments `k0Strikes`, never the entry field. A grep across `core/src/main` shows the field and both accessors have zero production callers; the only references are in `CachedEntryTest.k0InvalidationCountIncrementsMonotonically`. This is the Step-3-deferred BC1, carried to this track review by the track's Decision Log. The field's own Javadoc ("incremented by the cache in a later step") never came true and is now misleading: a future reader wiring K0 strikes would reasonably reach for this field and silently fork the strike count from the authoritative `k0Strikes` map.
- **Suggestion**: Remove the field, both accessors, and the `CachedEntryTest` case that exercises them. If a per-entry strike counter is wanted later for class-scoped invalidation (a documented v2 lever), reintroduce it then with a real call site.
- **Reference-accuracy caveat**: PSI (`steroid_execute_code`) timed out this session, matching the track file's note that PSI was unavailable across all steps; the no-production-caller conclusion rests on `grep -rn k0InvalidationCount core/src` rather than `ReferencesSearch`. The field is package-private-via-public-accessor and the cache package is small, so a grep miss would require a reflective or string-named access, which none of the cache classes use.

### CQ2 [should-fix] `@Nullable getIdentity()` fed into the non-null skip-set probe on an unstated invariant

- **File**: `core/src/main/java/com/jetbrains/youtrackdb/internal/core/sql/executor/cache/CachedResultSetView.java` (`isSkipped` line 331-336; `pullOneFromStream` line 358-360), and `DeltaBuilder.java` (line 1603-1604 in the diff: `record.getIdentity()` into `entry.getCachedRids().contains(rid)`)
- **Issue**: `Result.getIdentity()` is `@Nullable`, but `isSkipped` calls `r.isIdentifiable() && delta.shouldSkip(r.getIdentity())` and `pullOneFromStream` calls `entry.getCachedRids().add(r.getIdentity())` under the same `isIdentifiable()` guard, feeding the value straight into `Set<RID>` membership where a null would be a silent miss (skip-set never matches) rather than a loud failure. Correctness rests on the unstated cross-type invariant `isIdentifiable() == true → getIdentity() != null`. This is the Step-5-deferred BC1, which the track Decision Log explicitly flags Phase C to weigh promoting. The invariant is plausible but is nowhere asserted or documented, so a future change to `Result.isIdentifiable()` semantics would degrade into a wrong-result merge with no test signal — exactly the I10 floor this track is built to protect.
- **Suggestion**: Add a one-line `assert` or an explicit null-guard at the two probe sites (e.g. in `isSkipped`: `var rid = r.getIdentity(); return rid != null && delta.shouldSkip(rid);`), and a short comment stating the `isIdentifiable → non-null` contract the code relies on. An `assert` keeps the hot path branch-free in production while surfacing a contract breach in the `-ea` test runs the suite already mandates.

### CQ3 [suggestion] Global cache rate metrics are defined and registered but never recorded; the "later steps" comment is now stale

- **File**: `core/src/main/java/com/jetbrains/youtrackdb/internal/common/profiler/metrics/CoreMetrics.java` (lines 39-114 in the diff: the five `QUERY_CACHE_*_RATE` definitions and the `GLOBAL_METRICS` registration)
- **Issue**: Five `MetricDefinition` rate metrics (`QUERY_CACHE_HIT_RATE`, `QUERY_CACHE_MISS_RATE`, `QUERY_CACHE_SPLICE_FAILURE_RATE`, `QUERY_CACHE_K0_INVALIDATION_RATE`, `QUERY_CACHE_OVERFLOW_RATE`) are defined and added to `GLOBAL_METRICS`, but nothing in the track ever records into them — a grep for the five names outside `CoreMetrics.java` returns nothing. The per-tx `QueryCacheMetrics` counters are incremented inside `QueryResultCache` (hits/misses/k0/overflows at lines 139/148/155/204), but there is no bridge from those per-tx counts to the global rate metrics. The plan scopes Track 1's `CoreMetrics` work as "counters (no increments yet)", so registering definition-only is on-plan and not a defect. The code-quality problem is the **comment**: the block comment "Definitions only; the increment/record wiring lands with the cache lookup and eviction code in later steps" reads as a promise that the recording arrives within this work, yet all seven steps have landed without it (recording is in fact a later-track / v2 concern). A reader auditing observability will hunt for a recorder that does not exist.
- **Suggestion**: Reword the `CoreMetrics` comment to state plainly that the global rate metrics are registered in this feature but recording is deferred to a later track (or to the metrics-export wiring), rather than "later steps". This is the Step-3-deferred BC2 generalised: BC2 specifically asked to reconcile the `overflows` metric Javadoc with its LRU-eviction increment site — the `QUERY_CACHE_OVERFLOW_RATE` description ("entry overflows") and the `QueryCacheMetrics.overflows` Javadoc ("entries removed because populating crossed the per-entry record cap") both describe the per-entry record cap, but the only increment site is `evictEldestIfUnpinned` (LRU eviction), not the per-entry `maxRecordsPerEntry` cap (which has no enforcement site in this track at all). Align both descriptions with the actual increment (LRU eviction) or note the per-entry cap is not yet enforced.

### CQ4 [suggestion] `TxDeltaCursor.hasNextInject()` has no caller

- **File**: `core/src/main/java/com/jetbrains/youtrackdb/internal/core/sql/executor/cache/TxDeltaCursor.java` (line 84)
- **Issue**: `hasNextInject()` is a public method with no production caller and no test caller — a grep across `core/src` finds it referenced only by its own Javadoc and the Javadoc of `advanceInject()`. The view's merge consults `peekInject()` (null-signals exhaustion) and `advanceInject()` instead, so `hasNextInject()` is unused. The sibling accessors `injectSize()` / `getSkipSet()` / `getInjectList()` are documented "for assertions and view replay" and are exercised by `DeltaBuilderTest`, so those are legitimate test-support API; `hasNextInject()` is not.
- **Suggestion**: Remove `hasNextInject()`, or if it is intended as part of the cursor's public contract for Tracks 2/3, add a brief "reserved for the MATCH/aggregate cursor consumers" note so it is not mistaken for dead code at the next review.

### CQ5 [suggestion] `getQueryResultCache()` re-reads the global flag on every disabled-state call

- **File**: `core/src/main/java/com/jetbrains/youtrackdb/internal/core/tx/FrontendTransactionImpl.java` (lines 1367-1376 in the diff)
- **Issue**: The lazy initializer memoises only the enabled case: when the flag is off, `queryResultCache` stays null, so every `getQueryResultCache()` call re-enters the `if (queryResultCache == null)` branch and re-reads `GlobalConfiguration.QUERY_TX_RESULT_CACHE_ENABLED.getValueAsBoolean()`. With the feature off by default, this is the common path, and `getQueryResultCache()` is called at least once per `query()` (via `serveThroughCache`) plus the bulk-DML hook. The cost is a single global-config read per query, which is small, but it is paid forever on the zero-behaviour-change default path the design treats as the compatibility floor. The method Javadoc says "the flag is read once at creation", which is accurate only when the flag is on; when off it is read every call.
- **Suggestion**: Either accept it and correct the Javadoc to "read on every access while disabled, once at creation while enabled", or add a `private boolean cacheDisabledResolved` latch so the disabled state is also memoised for the transaction's life (consistent with the design's "read once" intent — toggling mid-tx is already documented as not retroactive). The latch is the cleaner match to the stated contract.

### CQ6 [suggestion] Duplicated parse scaffolding across cache test classes

- **File**: `core/src/test/java/com/jetbrains/youtrackdb/internal/core/sql/executor/cache/DeltaBuilderTest.java` (lines 3765-3772 in the diff) and `CachedResultSetViewTest.java` (lines 3333-3340)
- **Issue**: Both test classes carry a near-identical `parseOrderBy`/`parseSelect` helper that constructs a `YouTrackDBSql` over a `ByteArrayInputStream`, casts to `SQLSelectStatement`, and wraps the checked exception in an `AssertionError`. `NonDeterministicQueryDetectorTest`, `ShapeClassifierTest`, and `QueryResultCacheTest` instead parse through `YqlStatementCache.get(...)`, a simpler one-liner. The duplication is small and test-only, but it is the kind of copy-paste that drifts: two copies of the same parse-and-cast idiom will diverge the next time the parser API changes.
- **Suggestion**: Lift the parse helpers into a small package-private test utility in the cache test package (or standardise on `YqlStatementCache.get` where a fresh non-cached instance is not required — note `DeltaBuilderTest` deliberately bypasses the statement cache via `new YouTrackDBSql(...)`, so the utility must preserve that). Low priority; fold in only if the file is touched again.
