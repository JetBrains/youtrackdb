## Findings

### R1 [should-fix]
**Certificate**: Assumption — RecordOperation target class for the `version: long` field
**Location**: track-1 `## Context and Orientation` ("`RecordOperation` (`internal/core/tx/`) gains a `version: long` field") and `## Plan of Work` step 2; real target `core/.../internal/core/db/record/RecordOperation.java:29`.
**Issue**: The track names `internal/core/tx/RecordOperation` as the class that gains a mutable `version: long` re-stamped on the collapse path. That class is a Java `record` (immutable, `record RecordOperation(DBRecord record, RecordOperationType type)` at `internal/core/tx/RecordOperation.java:6`) — you cannot re-stamp a field on it. The class `FrontendTransactionImpl` actually uses (import line 29) and mutates in `addRecordOperation` is `internal/core/db/record/RecordOperation` (a mutable `final class` with `public byte type` at line 35, `DELETED=1/UPDATED=2/CREATED=3`, and several mutable `public long` counters). The design is feasible on the correct class — adding a mutable `public long version` and re-stamping it in the collapse switch is consistent with the existing mutable counters — but an implementer taking the cited path literally would target the immutable record, hit the immutability wall, and either thrash or re-architect. The two same-named classes are exactly the kind of collision that wastes an implementation cycle. Likelihood the wrong file is opened first: high (the path is explicit); impact: a stalled step, not a correctness bug, because the import disambiguates on inspection.
**Proposed fix**: Correct the Context and Plan-of-Work references to `internal/core/db/record/RecordOperation` and add a one-line note that the tx-package `RecordOperation` is an unrelated immutable record sharing the simple name. Confirm the re-stamp lands on the same mutable instance the collapse switch mutates (the map is RID-keyed and `RecordOperation.equals/hashCode` key on record identity only, so the instance is stable across collapse — this holds).

### R2 [should-fix]
**Certificate**: Exposure — `query()` / `executeInternal()` gate site
**Location**: track-1 `## Context and Orientation` ("`query()` (line 617) parses via `SQLEngine.parse()` then routes through `executeInternal` (line 702)"); `DatabaseSessionEmbedded.java:617`, `:652`, `:702`.
**Issue**: The Context claim that `query()` routes through `executeInternal` is wrong. `query(String, Object...)` (line 617) and `query(String, Map)` (line 652) each call `SQLEngine.parse(...)`, then `statement.execute(this, args, true)` and `queryStartedLifecycle(original)` **directly** — they never reach `executeInternal`. `executeInternal` (line 702) is reached only via the public `execute(...)` overloads (lines 690-699). So the cache gate has at least three distinct entry sites that call `statement.execute()`: the two `query` overloads and `executeInternal`. Blast radius if only one is wired: queries entering through the un-wired path bypass the cache silently (a hit-rate miss, benign), but worse — if the put/lookup is wired in one path and the bulk-DML `invalidateAll` or the `cacheCodeDepth` bracket is wired in another, a view built on one path can be served stale or re-entrancy can leak, breaking I10/CR1. The Component Map correctly lists both "query() / executeInternal()", so the intent is both; the Context prose is the trap.
**Proposed fix**: Fix the Context prose to state that `query()` and `executeInternal()` are independent `statement.execute()` call sites, and have decomposition place the gate at a single shared sink (wrap `statement.execute(...)`/`queryStartedLifecycle` or a private helper that all three call) rather than duplicating gate logic across three methods. A shared sink also makes the `cacheCodeDepth` increment/decrement bracket and the re-entrancy bypass exist in exactly one place.

### R3 [should-fix]
**Certificate**: Exposure — tx-end `clear()` sink and `closeActiveQueries()` ordering
**Location**: track-1 `## Context and Orientation` ("`clearUnfinishedChanges` (the single tx-end sink)") and `## Plan of Work` step 10 ("`clear()` in `beginInternal` and `clearUnfinishedChanges`"); `FrontendTransactionImpl.java:972-999`, `:948`, `:385`, `:698`.
**Issue**: `clearUnfinishedChanges()` (line 998) is **not** the single tx-end sink. The single common sink is the private `clear()` (line 972): commit funnels `doCommit`→`close()` (line 698→949), rollback calls `clear()` directly (line 385) and then `close()`→`clear()` again (line 400→949, so `clear()` runs twice on rollback), and `close()` calls `clear()` (line 949). `clear()` calls `session.closeActiveQueries()` FIRST (line 973) and `clearUnfinishedChanges()` LAST (line 993). Consequences for the cache: (a) hooking the cache `clear()` in `clearUnfinishedChanges` means the registered `LocalResultSet`s (and any `CachedResultSetView`s) have already been closed by `closeActiveQueries()` at the top of `clear()` — so the cache's own `entry.close()` is the *second* close of the same underlying stream, making the D9 `IdempotentExecutionStream` wrapper strictly load-bearing for I6 (not merely defensive); (b) on rollback `clear()` runs twice, so the cache `clear()` must itself be idempotent (the plan asserts I6 idempotent `clear()` — consistent, but the double-invocation path should be an explicit test); (c) since `clear()` is private, the cache clear can hook either at the head of `clear()` or inside `clearUnfinishedChanges` — but the *ordering relative to* `closeActiveQueries()` is the thing that makes the wrapper necessary and must be stated.
**Proposed fix**: Restate the Context/Plan-of-Work hook target as `clear()` (the common sink) with `clearUnfinishedChanges()` as its terminal step, and document that `closeActiveQueries()` precedes the cache clear within `clear()` — making the D9 idempotent-close wrapper a hard requirement, not an optimization. Add an explicit rollback-path test asserting the double `clear()` invocation leaves `cache.size()==0` and fires the underlying stream close exactly once (I1 + I6).

### R6 [suggestion]
**Certificate**: Assumption — the `mutationVersion` increment + stamp rides the existing `addRecordOperation` rollback try
**Location**: track-1 `## Plan of Work` step 2 / Decision Record D5 ("the stamp/increment pair rides the existing `addRecordOperation` rollback try"); `FrontendTransactionImpl.java:510-628`.
**Issue**: `addRecordOperation` has an outer try (lines 517-628) whose `catch` calls `rollbackInternal()` and rethrows. Several validation throws sit before the actual op is added/collapsed (unloaded record at 519, embedded at 527, temporary-record at 556, duplicate-instance at 582). If the `mutationVersion` increment is placed early (before the put/collapse) and one of those throws fires, the counter advances without a corresponding committed op. Within a single tx this is benign for two reasons: (a) the throw triggers `rollbackInternal()`→`clear()`, which ends the tx and clears the cache, so no K0_NONE entry survives to be falsely invalidated by the orphaned bump; (b) `mutationVersion` is monotonic and only relative ordering matters (`op.version > populateMutationVersion`), so a skipped value is harmless. The risk is low and self-healing, but it is an undocumented assumption that the increment placement does not matter — and it only holds because the throw is fatal to the tx. If a future refactor makes any of those validation throws non-fatal (caught and continued), an orphaned increment would falsely invalidate live K0_NONE entries.
**Proposed fix**: Place the `mutationVersion++` and the `version` stamp **after** the op is successfully added/collapsed (after line 613, before line 621) so the counter only advances on a real mutation, and add a one-line comment that the increment must follow successful op insertion. This removes the dependency on the throw being fatal and makes the invariant local.

#### Exposure: tx-end `clear()` sink and `closeActiveQueries()` ordering
- **Track claim**: `clearUnfinishedChanges` is "the single tx-end sink"; cache `clear()` is hooked in `beginInternal` and `clearUnfinishedChanges` (Plan step 10).
- **Critical path trace**:
  1. Commit: `commitInternalImpl` @`FrontendTransactionImpl.java:236` → `doCommit` @632 → `close()` @698 → `clear()` @949.
  2. Rollback: `rollbackInternal` @356 → `clear()` @385 (BEGUN/COMMITTING case), then `close()` @400 → `clear()` @949 (second invocation).
  3. `close()` @948 → `clear()` @949.
  4. `clear()` @972: `session.closeActiveQueries()` @973 (FIRST) → record unload loop @976-988 → `clearUnfinishedChanges()` @993 (LAST) → `recordSerializationContext.clear()` @995.
  5. `closeActiveQueries` @`DatabaseSessionEmbedded.java:3431` iterates `new ArrayList<>(activeQueries.values())` and calls `rs.close()` on each registered `ResultSet` (the `LocalResultSet`/`LocalResultSetLifecycleDecorator`).
- **Blast radius**: if the cache clear is mis-placed so it runs before `closeActiveQueries`, or if the underlying stream is closed twice without an idempotent wrapper, the second close throws or double-frees a storage cursor (I6 violation) during pool shutdown. Rollback's double `clear()` (steps 1-2) requires cache `clear()` idempotency (I1/I6).
- **Existing safeguards**: `closeActiveQueries` snapshots the values into a fresh `ArrayList` (3432) so concurrent map mutation during close is safe; `activeQueries` is a `WeakValueHashMap` in embedded mode (`:256`) so a GC'd view is silently dropped from close (the cache must strongly pin the shared stream — D4/D9). `assertOnOwningThread` is NOT called by `clear()`/`close()` (documented cross-thread exception, `:130-131`).
- **Residual risk**: MEDIUM — correctness rests on the D9 idempotent-close wrapper being threaded into both the cache entry slot and the `LocalResultSet` stream slot, and on cache `clear()` being idempotent under the rollback double-invocation. Both are planned; the ordering fact is mis-stated in the track.
