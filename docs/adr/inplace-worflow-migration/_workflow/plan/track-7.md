# Track 7: Staging architecture for workflow-modifying branches

## Purpose / Big Picture
After this track lands, workflow-modifying branches accumulate their workflow document changes under `<plan-dir>/_workflow/staged-workflow/.claude/{workflow,skills}/...` during execution, so the branch's own sessions read a stable live workflow at develop's state until Phase 4 promotes the staged subtree in one commit.

<!-- Reserved for Move 2 — ADDED/MODIFIED/REMOVED triad. Empty until Move 2 lands. -->

The in-place migration (Tracks 1–6) handles drift on feature plans whose execution does not edit `.claude/workflow/**` or `.claude/skills/**`. Track 7 codifies the inverse case: a plan whose execution is itself the source of workflow drift. Writing directly to the live paths during execution forces the branch to migrate against its own work — plan citations go stale immediately, reviewers flag anchors that disappear at the next commit, and the drift gate trips on the branch's own changes. Staging isolates workflow document edits in a per-branch subtree under `_workflow/`; the live paths in the branch's checkout stay at develop's state; a "Promote workflow changes" commit at Phase 4 copies the staged subtree into the live paths immediately before the final-artifacts commit. The existing cleanup commit then removes `_workflow/` (the staged subtree included), so the squash-merge into `develop` carries only the live workflow files plus `design-final.md` and `adr.md`. Forward-applicable only: the current branch finishes under the existing in-place model; Track 7 codifies the new convention for the next workflow-modifying branch.

## Decision Log

#### DL1: Constraints declaration drives implementer enforcement gate; directory presence drives Phase 4 promotion

- **Alternatives considered**: Constraints declaration paired with an opt-in glob mirror (the Phase 1 wording — dual mechanism with no tie-breaker); directory presence as the single signal extended to the implementer (works after the first staged write but offers no enforcement before); glob check alone (a planner who forgets to declare workflow-modifying still routes by glob, which is fragile); the chosen split.
- **Rationale**: Constraints is the only signal that can drive enforcement from the very first step of a workflow-modifying plan; directory presence requires a staged write to exist first. The Phase 4 promotion guard already uses directory presence as the operational artifact (no staged content → no promotion). The two surfaces serve distinct purposes: Constraints answers "is the gate active"; directory presence answers "is there content to promote." `conventions.md` defines the canonical marker spelling so two planners cannot spell the marker two different ways.
- **Risks/Caveats**: Two signals must agree by construction. The conventions.md §1.7 staging subsection states "Constraints declaring workflow-modifying without any staged content is allowable but produces a no-op at Phase 4 promotion" (legitimate dry-run or abandoned-work shape). The inverse (staged content without Constraints declaration) is unreachable when the implementer enforcement gate works as designed; Constraints undeclared means no gate, so no staged write happens. Track 7 itself does not declare workflow-modifying in Constraints because the convention does not exist during its own execution; see DL4.
- **Implemented in**: Track 7 — Step 1 (conventions.md canonical marker spelling and detection rule), Step 2 (implementer enforcement gate keyed off Constraints), Step 4 (Phase 4 promotion guard keyed off `staged-workflow/` directory presence).

#### DL2: Pre-commit gate refuses live workflow path writes on workflow-modifying plans

- **Alternatives considered**: Policy-only rule in `implementer-rules.md` ("write to staged-workflow if Constraints declare workflow-modifying"; no commit-time gate); pre-commit gate mirroring the ephemeral-identifier check pattern (chosen).
- **Rationale**: A3 demonstrated that policy-only enforcement is bypassed by `Bash`, `Edit`, `Write`, or `steroid_apply_patch` against absolute live paths. The drift gate's `git log $BASE_SHA..HEAD -- .claude/workflow .claude/skills` walk catches the bypass after the commit lands, defeating the staging convention's purpose. A pre-commit gate refusing `git diff --cached` matches against live `.claude/workflow/**` or `.claude/skills/**` outside the Phase 4 promotion commit closes the bypass surface. The pattern mirrors the existing ephemeral-identifier check at `implementer-rules.md`:253-285. Cost: one bash check per implementer spawn on workflow-modifying plans.
- **Risks/Caveats**: Phase 4 promotion is the legitimate exception. The gate's allow-clause keys off the active commit's prepared message prefix `Promote workflow changes from <plan-dir>/_workflow/staged-workflow`; when that prefix is present, the gate passes. False negatives re-introduce the bootstrap problem; false positives are recoverable by re-routing the write through the staged subtree.
- **Implemented in**: Track 7 — Step 2 (implementer-rules.md gate addition).

#### DL3: Reads precedence — staged copy authoritative when present

- **Alternatives considered**: Reads always hit live (the Phase 1 wording, which under-specifies the multi-step authoring case); reads hit staged when a staged copy exists and live otherwise (chosen).
- **Rationale**: T2 and A5 surfaced the intra-track multi-step authoring case. Step N adds rule X to staged `conventions.md`; Step N+M cites rule X. With reads-always-live, Step N+M reads live (where X does not exist yet), then either reads stale content before writing to staged or cannot cite the rule it just authored. The chosen rule keeps Step N+M working against current state. The original "reads hit live" goal was cross-track plan-citation coherence; the chosen rule keeps that property because cross-track callers (drift gate, plan-slim rendering, sibling track citations) do not have staged copies of the files they read, so they hit live anyway. The chosen rule is a strict generalization that fixes the intra-track case without losing the cross-track property.
- **Risks/Caveats**: Implementer must check for the staged copy before every read of a `.claude/workflow/**` or `.claude/skills/**` file. One-line bash check (`if [ -f "$STAGED_DIR/<rel>" ]; then read staged; else read live`); cheap. Reviewers and consumers outside the implementer (drift gate, plan-slim renderer) keep reading live unchanged.
- **Implemented in**: Track 7 — Step 1 (conventions.md staging subsection lands the reads-precedence rule).

#### DL4: Sibling-worktree rejection rationale, forward-applicable carve-out, I6 precision

- **Alternatives considered**: Accept D14 as written, accepting A1's "self-defeating dogfood" critique unchallenged; reopen D14 to accept the sibling-worktree alternative via inline ESCALATE replan; strengthen D14's rationale through a track-scoped Decision Log entry without modifying D14 directly (chosen).
- **Rationale**: A1 correctly observed that D14 rejects sibling-worktree on dogfood grounds but accepts the same dogfood loss in its own Risks/Caveats. The strengthened rationale does not change D14's choice — staging via subtree stays — but adds substantive non-dogfood grounds: Tracks 4a/4b deliberately retired the develop-worktree dependency from `/migrate-workflow` (the in-branch migration is the project's anti-worktree direction); the convention-spread cost is bounded (six in-scope files); a sibling-branch approach adds a second source of truth that merges into develop alongside the feature branch, creating a coordination surface every workflow-modifying plan would have to manage. ## Context and Orientation

The workflow already separates ephemeral planning artifacts from live workflow rules: planning happens under `docs/adr/<dir-name>/_workflow/` (`implementation-plan.md`, `design.md`, `plan/track-N.md`, `design-mutations.md`), which the Phase 4 cleanup commit removes before merge; live workflow rules under `.claude/workflow/` and `.claude/skills/` are read by every session and survive merge into `develop`. The drift gate (Track 3) and the migration (Tracks 4a/4b) handle the case where live workflow rules move on `develop` while the branch carries pre-rebase artifacts. They do not handle the case where the branch is itself authoring the workflow change.

`.claude/workflow/implementer-rules.md` is the rulebook the Phase B per-step implementer sub-agent reads on every invocation. It owns the contract for what the implementer is and is not allowed to touch and how to spell file paths in writes. Today it has no awareness of staging; an implementer told to edit `.claude/workflow/X.md` writes there directly.

`.claude/workflow/step-implementation.md` is the Phase B orchestrator. It assembles the implementer prompt by stitching `implementer-rules.md`, the plan-slim render, and the step description together. It owns the call site that would forward a staging directive to the implementer.

`.claude/workflow/workflow.md` § Final Artifacts is the canonical specification for the two-commit Phase 4 shape (final-artifacts + cleanup). `.claude/workflow/prompts/create-final-design.md` is the prompt that drives Phase 4 execution and references the same two-commit shape. Both files need to learn the three-commit shape (promote-staged-workflow + final-artifacts + cleanup) that fires only on workflow-modifying plans.

`.claude/workflow/workflow-drift-check.md` carries the `git log` pathspec the drift gate uses for its Phase 2 walk (`.claude/workflow/ .claude/skills/`). The staged subtree lives under `docs/adr/*/_workflow/staged-workflow/.claude/...`, a different prefix, so the pathspec naturally excludes the staged paths. A defensive comment at the pathspec site protects this exclusion against a future change that broadens the pathspec.

`.claude/workflow/conventions.md` is the shared-rules home. The staging convention (the staged-subtree path, the workflow-modifying-branch detection rule, the "writes to live workflow route to staged" rule, the I6 invariant) lands here so every consumer (planner, implementer, Phase 4 producer) resolves to one source of truth.

The bootstrap problem in concrete terms: a branch that edits `.claude/workflow/X.md` to add a new section directly in its own tree changes the rules its own subsequent sessions read. Every plan citation that anchored to the old section structure of X.md goes stale at the commit point — Phase A reviewers see references that no longer resolve, Phase B implementers read a moving target, Phase C cumulative diffs include the workflow churn alongside the substantive change, and the drift gate at the next session-start sees the branch's own commits as drift. Staging isolates the edits from the live workflow until Phase 4; the live workflow stays at develop's state; reviewers and implementers see one consistent rule surface throughout execution.
