## Context and Orientation
The step-level logical filters split into two families. `and` / `or` (the `ConnectiveStrategy` form — only `AndStep` / `OrStep` extend `ConnectiveStep`), `not`, and `where(traversal)` each carry a **child sub-traversal** of arbitrary recognized steps and are translated by the sub-walker. `where(P)` (`WherePredicateStep`) is distinct: it carries a `startKey` + predicate + a `$matched.<label>` reference, not a child traversal, so the sub-walker does not apply to it (technical review T3). The recognisers implement the post-Track-3 contract `Outcome recognize(StepCursor, RecognitionContext)` (head via `cursor.take()`, trailing shape via `takeIf` / `takeWhile`, returning `ACCEPTED` / `DECLINE`). A logical child is translated by a **sub-walker**: a nested `StepCursor` over the child sub-traversal, run against the same registry, whose recognition context inherits the parent `boundaryAlias`.

The sub-context contract is load-bearing — a naive fresh nested `WalkerContext` per child produces silent wrong results, not declines (adversarial A4):
- **Alias minting delegates to the parent's `AliasSequence`.** `AliasSequence` restarts at 0 per context (WalkerContext.java:336-351), so per-child contexts mint `$g2m_anon_0` twice for `and(__.out("a"), __.out("b"))`; in MATCH one alias is one binding, silently over-constraining to "both edges reach the *same* vertex" and dropping every source whose two targets differ.
- **`pinBoundary` / `setSingleReturnColumn` are swallowed.** A hop child dispatches `VertexHopRecogniser`, which re-pins the boundary / single RETURN column to its target (VertexHopRecogniser.java:80-84; `setSingleReturnColumn` clears and replaces the projection, WalkerContext.java:320-330) — on a delegating context that moves the outer traversal's result column onto the child's target.
- **`putAliasFilter` is captured per-combinator.** OR children's WHERE contributions must be collected as composable `SQLBooleanExpression`s for `MatchWhereBuilder.or`, never committed through `putAliasFilter` — which in HEAD **AND-composes** a second same-alias contribution (Track 4: an existing entry is merged via `WHERE.and(...)`, not replaced; WalkerContext.java:292-305). AND-composition is the wrong connective for OR, so committing OR fragments individually would AND them (risk review R4; the older "override rather than compose" wording was stale against Track 4's merge behaviour).
- **Pattern contributions forward for AND edge-bearing children, and are captured for NOT.** A NOT edge-bearing child's output goes to the `notMatchExpressions` sink, never the positive `patternBuilder`.

`hasNot(key)` and logical `not(...)` share the `final NotStep` runtime class (A2), so one recogniser owns `NotStep.class` and branches the values-child presence form before `hasEdgeHops`. Edge-bearing NOT emits a detached `SQLMatchExpression` into a new `notMatchExpressions` list on `WalkerContext` (a new `RecognitionContext` sink, wired into `buildResult`'s `MatchPlanInputs.builder(...)`), produced by the A5 builder capability. The recogniser declines only when the NOT origin alias is absent from the positive pattern (A6); the eager-build safety net is the backstop for the disqualifying shapes (R4).

The `GremlinPlanCache` (D5) closes the seam Track 4 leaves open. Track 4's predicate adapter renders comparison values as inline literals (`MatchLiteralBuilder.toLiteral`, today's behavior). This track adds `bindParam(value) → SQLPositionalParameter` (slot allocation + value recording) to `RecognitionContext`, implemented by `WalkerContext`, and switches `GremlinPredicateAdapter` to emit `SQLPositionalParameter` for **predicate comparison values** — so recognisers and the adapter, which see only `RecognitionContext`, can reach it. Structural tokens stay inline and must **not** parameterize (class names for `classEquals`, `~label` values, RIDs for `@rid IN`), because a structural token bound as a param serves a wrong plan. The switch must be **total** over every comparison-value form (scalar `Compare`, `Contains` elements, `Text`/`TextP` operands, `between`/`inside`/`outside`) — a value left inline both key-collapses and bakes into the shared plan (R3). The boundary step installs the per-walk param map through `ctx.setInputParameters(map)`. The cache key is **not** `SQLMatchStatement.toGenericStatement()` — that serialises only `matchExpressions`, never `notMatchExpressions` (A1 blocker), and collapses class-name string literals to `?` (R1). The fingerprint is instead synthesised from the pre-plan `MatchPlanInputs` (positive pattern + `notMatchExpressions` + insertion-ordered alias filters), taken before the planner's `HashMap` copy so two walks of one shape key identically (R2), with `SQLPositionalParameter` values rendered as `?` and structural tokens rendered **verbatim** so distinct classes / RIDs stay discriminating (R1). The cache reuses the YQL plan-cache `MetadataUpdateListener` schema-change invalidation hook, and get/build/put all stay inside `apply()`'s `RuntimeException` net so a disqualifying NOT shape still degrades to native on the first uncached walk and is never stored (R5). RID-bearing traversals bypass the cache (R3 — they fingerprint per id set and gain no reuse; see Track 4 Decision Log for the RID-inline decision they follow).

- The plan cache serves one plan for the same traversal shape across distinct predicate values, and a plan reused with a second value returns the **second** value's multiset — not the first's baked in (R3); `eq(null)` (bare `IS NULL` AST) and `eq(v)` (`= ?`) occupy distinct cache entries (Track 4 A1 pin + A3); collection-size classes do not collide; a schema change invalidates the cache (D5 / A3 / R6).
- Distinct `hasLabel` labels (`hasLabel("Person")` vs `hasLabel("Company")`) occupy **distinct** cache entries in both polymorphic and non-polymorphic modes — structural tokens stay discriminating in the fingerprint (R1).
- Two NOT-differing shapes (`g.V().not(__.out("a"))` vs `g.V().not(__.out("b"))`, and a NOT shape vs the same positive pattern without NOT) occupy **distinct** cache entries — the fingerprint enumerates `notMatchExpressions`, which `SQLMatchStatement.toGenericStatement()` omits (adversarial A1 blocker regression test).

- The plan-cache fingerprint is stable across two independent walks of the same shape (R2) and value-independent across predicate values (deterministic, shape-pure walk-to-slot parameter ordering, R3).

## Idempotence and Recovery
The translator remains stateless per walk; the plan cache is the first cross-execution mutable surface this track adds. Recovery rides the existing D3 all-or-nothing decline — a declined sub-walk or combinator leaves the outer walk discarded whole (`RecognitionContext` no-mutation discipline, A7).
- **Step 1 (sub-walker):** a declined child sub-walk must not commit partial state to the outer context — the capture-boundary unit test is the gate. Sub-context alias delegation is idempotent per child invocation.
- **Steps 2–4 (combinators):** an untranslatable child declines the whole traversal; OR with an edge-bearing child declines without emitting a partial OR clause.
- **Step 5 (cache):** schema-change invalidation clears stale entries; RID-bearing shapes bypass the cache entirely (R3). A cache miss falls through to eager build — same as today's `useCache=false` path.
