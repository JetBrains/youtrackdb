# Track 4: Filtering — predicates (has / hasLabel / hasId, P / Text / TextP)

## Purpose / Big Picture
After this track the Gremlin predicate surface translates: property / label / id predicates (`has` / `hasLabel` / `hasId`), the `P` / `Text` / `TextP` predicate algebra, and the bare presence form `has(key)` (`IS DEFINED`). Polymorphic-mode `hasLabel` translation is contingent on the boundary-node re-typing determination in the Decision Log (R1) — if it declines, `hasLabel`-bearing traversals fall back to native in the default polymorphic config. The step-level logical filters and the plan cache split off to Track 5.

The bare presence form `has(key)` recognises via `TraversalFilterStepRecogniser` and emits `IS DEFINED` (D-IS-DEFINED). Predicate comparison values render as inline literals here (today's `MatchLiteralBuilder.toLiteral` behavior); Track 5 flips them to positional parameters when it lands the `GremlinPlanCache` (D5). The step-level logical filters (`and` / `or` / `not` / `where`), the `hasNot(key)` negation, and the sub-walker split off to Track 5 (adversarial A1, user-approved 2026-07-15).

## Context and Orientation
Track 3 left a `GremlinPredicateAdapter` skeleton (it translated only the `has(...)` inside edge-filter chains). Track 4 makes it the full chokepoint between TinkerPop's predicate algebra and `SQLBooleanExpression`. The mapping is mostly mechanical (`Compare.eq/neq/gt/gte/lt/lte` → YTDB operators 1:1, `Contains.within/without` → `SQLInCondition` / `SQLNotInCondition`), but several corner cases require care and are spelled out in design §"Predicate translation":
- **NULL semantics:** `P.neq(null)` rewrites to `field IS NOT NULL` (both sides exclude absent — correct). `P.eq(null)` must rewrite to `field IS DEFINED AND field IS NULL`, **not** a bare `field IS NULL` (A1): YTDB `IS NULL` conflates absent and literal-null (`MatchWhereBuilder.isNull` Javadoc), while native `HasContainer.test` returns false on an absent property's empty iterator, so a bare `IS NULL` over-matches — a vertex lacking the key would translate-match but native-exclude (vertex A `nickname=null` and vertex B without `nickname`: native returns {A}, bare `IS NULL` returns {A, B}). Both `isDefined` and `isNull` factories shipped in Track 1. (The rewrite is needed at all because YTDB's `=`/`!=` return FALSE on null operands, diverging from TP.)
- **Absent-property semantics:** Track 3's `GremlinPredicateAdapter` already emits `has(k, neq(v))` as `k IS DEFINED AND k <> v` — the MATCH executor treats `<>` on an absent property as true while native excludes absent rows, so raw `<>` over-matches. Track 4 generalizes this to a **rule**, not an operator list (A2): any predicate whose translated SQL evaluates *true* on an absent property gets the `IS DEFINED` guard, because native `HasContainer.test` excludes absent rows for every predicate (empty-iterator → false). That provably covers every negated form — `Contains.without` → `SQLNotInCondition`, the `TextP` `not*` variants and `P.not(...)` built via `MatchWhereBuilder.not` (NOT of a false positive → true on absent) — plus `neq` (already guarded) and `eq(null)` (A1 above). The four range comparisons (`gt` / `gte` / `lt` / `lte`) likely already agree (comparison on absent → false) but are audited to confirm.
- **Singleton-collection equality:** `P.eq([a])` / `P.neq([a])` (size-1 literal) **decline** under D3 — `QueryOperatorEquals` auto-unboxes singletons against scalars, and field cardinality is unknown at translation time in schema-less/mixed classes. Size 0 and ≥2 translate normally.
- **`between`:** Gremlin `between` is right-exclusive `[lo, hi)`; YTDB `SQLBetweenCondition` is closed `[lo, hi]`. Translate to `AND(>=lo, <hi)`, never `SQLBetweenCondition`. `inside`/`outside` are open both ends → `AND(>lo, <hi)` / `OR(<lo, >hi)`.

The g2m translator runs **before** `YTDBGraphStepStrategy` (`YTDBGraphStepStrategy.applyPrior()` returns `{GremlinToMatchStrategy.class}`), and `GraphStep` is not a `HasContainerHolder`, so at translation time no GraphStep fold has happened, no `YTDBHasLabelStep` *instance* appears in the step list (the class exists; the fold that would mint one runs later), and `hasLabel` is never folded onto the start step. `has(key,value)`, `hasLabel(label)`, and `hasId(id)` therefore all arrive as a single `HasStep` distinguished only by `HasContainer` key — a property key, `T.label.getAccessor()` (`~label`), or `T.id.getAccessor()` (`~id`). One `HasStepRecogniser` iterates `HasStep.getHasContainers()` and branches on the key: `~label` → narrow by **re-typing the boundary node's class** (`MatchPatternBuilder.addNode` className-overwrite, so the scan narrows to `L` rather than a full `V` scan that rejects rows in a WHERE) — exact `classEquals(L)` in non-polymorphic mode, subclass-inclusive `{class:L}` re-typing or an `SQLInstanceofCondition` WHERE in the default polymorphic mode (Decision Log R1/A5) — handled **only for a single `eq(L)`** container; a multi-label `hasLabel(L1,L2)` within-container (`P.within([...])`) declines under D3 (A3, detail in Plan of Work step 2); `~id` → an `@rid IN [...]` WHERE clause on the alias filter, built as a **record-attribute** `@rid` IN (a `SQLRecordAttribute` left side + RID literals) via a shared helper extracted from `StartStepRecogniser.buildRidInExpression` — **not** `MatchWhereBuilder.in`, whose plain-`SQLIdentifier` left side is read as an absent property and is never promoted (`promoteStaticRidsFromFilters` recognises a RID-IN only through `findRidInList` → `isBareRidExpression`, which requires a `@rid` `SQLRecordAttribute`); the single-id case then collapses through `promoteStaticRidsFromFilters`, and because `hasId` is set-membership (not `g.V(ids)` seek semantics) the branch must NOT inherit StartStepRecogniser's duplicate-id decline; a property key → the predicate adapter. This container-key branching runs **before** the adapter's reserved-`~`-key decline. `MatchWhereBuilder.classEquals` has no production caller yet — Track 4's folded `hasLabel` is its first (Track 3's rework deleted the interim `MatchClassFilters` helper). `has(key)` desugars (TinkerPop 3.8.1) to `TraversalFilterStep(__.values(key))` — it does not land on `HasStep`, so it bypasses `HasStepRecogniser` and routes through the new `TraversalFilterStepRecogniser` Case A, emitting `IS DEFINED` (not `IS NULL`, which over-matches null-valued properties). `hasNot(key)` desugars to `NotStep(__.values(key))`, the same `final` class as logical `not(...)`, so its handling moves wholly to Track 5's single `NotStep` recogniser (adversarial A2); Track 4 owns only the `has(key)` presence form.

Track 4's recognisers implement the post-Track-3 contract `Outcome recognize(StepCursor, RecognitionContext)` (head via `cursor.take()`, trailing shape via `takeIf` / `takeWhile`, returning `ACCEPTED` / `DECLINE`), the model Track 3's rework left in place of the retired `boolean recognize(Step, WalkerContext)` + manual `stepIndex` form.

Same-alias filter contributions must **AND-compose**, not overwrite. Track 4 routinely contributes two WHERE clauses to one alias — `g.V(ids).has(k,v)` (start `@rid IN`, then `has`) and `hasLabel(L).has(k,v)` — so `putAliasFilter` (and the `buildResult` merge) must AND an incoming clause with any existing one via `MatchWhereBuilder.and` rather than replace it. An overwrite silently drops the earlier filter and returns a wrong (over-large) multiset. Track 5's `where` / `and` contributions reuse this same AND-composition.

## Plan of Work
1. **Full predicate adapter** (`GremlinPredicateAdapter`): the complete `P` set (`Compare`, `Contains`, composite `P.and/or/not`, `inside`/`outside`/`between` decompositions), the NULL and singleton-collection rules above, and custom-predicate decline (`getBiPredicate()` not `Compare`/`Contains`/`Text`).
2. **Single `HasStepRecogniser`** (keyed on `HasStep.class`; D9 permits one recogniser per exact runtime class, and `has` / `hasLabel` / `hasId` all produce `HasStep`). It iterates `HasStep.getHasContainers()` and branches on container key: a property key → the predicate adapter (`has(key,value)` / `has(key,predicate)`, multiple property containers AND together); `~label` → narrow by **re-typing the boundary node's class** via `MatchPatternBuilder.addNode`'s documented className-overwrite (so the scan is narrowed to `L`, not a full `V` scan that rejects rows in a WHERE — A4): non-polymorphic re-types to `L` and attaches exact `classEquals(L)` (subclasses filtered out); the default polymorphic mode re-types to `{class:L}` (MATCH matches subclasses) or uses an `SQLInstanceofCondition` WHERE, per Decision Log R1/A5. A `~label` container is handled **only when it carries a single `eq(L)`**; a multi-label `hasLabel(L1,L2)` arrives as one within-container (`P.within([...])`) that neither `classEquals` (single name) nor `MatchWhereBuilder.in("@class",…)` (the plain-`SQLIdentifier` `@class` trap) can build, so it **declines** in Track 4 under D3 and the whole traversal falls back to native (A3; multi-label translation via record-attribute `@class IN` / OR-of-`SQLInstanceofCondition` is a possible follow-up). `~id` → an `@rid IN [...]` WHERE on the alias filter built via the **record-attribute** form (a shared helper extracted from `StartStepRecogniser.buildRidInExpression`, not `MatchWhereBuilder.in`; single and multi both route through `@rid IN`, the single case collapsed by the planner's `promoteStaticRidsFromFilters`, and `hasId` set-membership means the branch does NOT inherit StartStepRecogniser's duplicate-id decline). The `~label` / `~id` interception runs before the adapter's reserved-key decline. No separate `HasLabelStepRecogniser` / `HasIdStepRecogniser` — no `HasIdStep` class exists, and no `YTDBHasLabelStep` *instance* appears in the step list at translation time (the GraphStep fold that would mint one runs later, in `YTDBGraphStepStrategy`). ## Idempotence and Recovery
The translator is stateless and side-effect-free: every step builds a MATCH plan from an immutable traversal and mutates no persistent state, so re-running a translation is idempotent by construction. Recovery is uniform — the D3 all-or-nothing contract landed in Track 2: any unrecognised or untranslatable shape makes the strategy DECLINE, discarding the partial `WalkerContext` (Track 3's rework discards the whole context on decline) and leaving the traversal to execute natively. No partial MATCH plan ever executes.
- **Step 1 (AST layer):** the new `SQLEndsWithCondition` / find-mode `SQLMatchesCondition` / collate `SQLContainsTextCondition` are pure AST + evaluator additions with no persistent state; a plan carrying them deep-copies via `copy()` on `clone()` / cache-get, so recovery rides the SQL engine's existing plan-clone path. The existing-`CONTAINSTEXT` behavior change is guarded by the default-unchanged regression test — a defect there is caught at build, not in production.
- **Step 2 (adapter):** an unsupported predicate (custom `BiPredicate`, a size-1 collection under D3, an unhandled `Text`/`TextP` shape) returns a decline from the adapter that propagates to the whole-traversal decline-to-native; no half-built WHERE clause is emitted.
- **Step 3 (recognisers):** the translate-all-containers-then-single-`putAliasFilter` ordering (A6) guarantees an untranslatable `HasContainer` declines with zero `WalkerContext` mutation; a multi-label `~label`, a reserved-`$` key, or an unconvertible id all decline to native rather than throw or emit a wrong filter.
