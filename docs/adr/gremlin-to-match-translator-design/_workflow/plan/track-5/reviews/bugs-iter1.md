## Findings

### BG1 [should-fix]
**Certificate**: C-hasedges
**Location**: `SubTraversalPredicateAdapter.addNode` (SubTraversalPredicateAdapter.java:180-184); triggering caller `HasStepRecogniser.recognize` (HasStepRecogniser.java:156-164).
**Issue**: The adapter classifies a child as edge-bearing vs pure-filter through the `hasEdges` flag, which `hasEdges()` (SubTraversalPredicateAdapter.java:121-123) exposes for the Step-2/3 combinators to read back. `addNode` sets `hasEdges = true` unconditionally:

```181:184:core/src/main/java/com/jetbrains/youtrackdb/internal/core/gremlin/translator/strategy/SubTraversalPredicateAdapter.java
  public void addNode(String alias, String className) {
    capturedPattern.addNode(alias, className, null, false);
    hasEdges = true;
  }
```

But `addNode` is not exclusively an edge-target contribution. `HasStepRecogniser` — a Track-4 recogniser already in the production registry that `subWalk` dispatches against (the same `recognisers` map is threaded through `GremlinStepWalker.subWalk`) — calls `ctx.addNode(boundary, labelClass)` to re-type the *existing boundary node's* class for a folded `hasLabel(L)`:

```156:164:core/src/main/java/com/jetbrains/youtrackdb/internal/core/gremlin/translator/strategy/HasStepRecogniser.java
    if (labelClass != null) {
      // Re-type the boundary node's class so the scan narrows to L (addNode overwrites the prior
      // class). ...
      ctx.addNode(boundary, labelClass);
      if (!ctx.polymorphic()) {
        whereExprs.add(WHERE.classEquals(labelClass));
      }
    }
```

`hasLabel(L)` is a pure-filter contribution (a `@class` narrowing on the boundary alias, part of Track 4's predicate surface) — it adds no hop, no new vertex, no edge. Yet routing it through `addNode` flips `hasEdges` to `true`, so a pure-filter child that contains a `hasLabel` is reported as edge-bearing. The failure surfaces in the combinators that consume the classification (this step lands the classification; Steps 2-3 read it), and the C&O `hasEdges`-driven contract makes the two wrong:
- **OR** "requires all children pure-filter and declines if any child carries edges" (track-5.md `## Context and Orientation`). `or(__.hasLabel("Person"), __.hasLabel("Company"))` — a legal all-pure-filter OR — would see `hasEdges()==true` on both children and wrongly decline the whole traversal to native. (Decline is result-safe, but it silently forfeits MATCH for a shape the track claims to translate.)
- **NOT** branches on `hasEdgeHops`: "pure-filter → `MatchWhereBuilder.not(...)` into `where`; edge-bearing → a detached `SQLMatchExpression` into the `notMatchExpressions` sink". `not(__.hasLabel("Person"))` is a pure filter (`@class` predicate) but would be classified edge-bearing and routed to the detached-anti-join path — a NOT expression whose only captured pattern content is a re-typed origin node with no hop, which `manageNotPatterns` is not built to consume correctly (its first guard requires the origin alias in the positive pattern; a re-typed-boundary-only detached expression is not the `NOT {as:p}.out(...)` shape). That is a wrong-results / wrong-decline risk, not merely a missed optimization.

The `hasEdges` semantics the class Javadoc states ("an edge-bearing child contributes at least one pattern fragment") conflate "touched the pattern builder" with "added a hop/edge"; a boundary re-type is a pattern-builder touch that is not edge-bearing.
**Refutation considered**: (1) Could `hasLabel` inside a combinator child be out of scope, so `addNode` is only ever an edge target here? No — `HasStepRecogniser` owns `HasStep.class` in `PRODUCTION_RECOGNISERS` (GremlinStepWalker.java:98-103) and `subWalk` dispatches child steps against that same registry, so `and`/`or`/`not(__.hasLabel(L)...)` is a first-class recognised child shape (Track 5 combinators translate arbitrary recognised children). (2) Could a downstream combinator re-derive pure-vs-edge some other way, making `hasEdges` advisory? The C&O and Plan-of-Work steps 2-3 explicitly key OR-decline and NOT-routing on `hasEdges`, so it is load-bearing, not advisory. (3) Is the boundary re-type genuinely pattern-free? `ctx.addNode(boundary, labelClass)` only re-registers the class on the already-present boundary alias (MatchPatternBuilder.addNode merges class onto the existing node) — no new alias, no edge — so it is pure-filter by every operational measure. **Verdict: CONFIRMED.**
**Reference-accuracy caveat**: mcp-steroid PSI was unreachable this session; the "which recognisers call `ctx.addNode`" caller set was established by grep over `**/gremlin/translator/strategy/*.java` plus a direct read of `HasStepRecogniser`, `StartStepRecogniser`, and `GremlinPatternAssembler`. Grep can miss a polymorphic or recently-renamed `addNode` caller; the concrete `hasLabel` caller cited here was read directly, so the finding does not rest on the grep alone, but an additional pure-filter `addNode` caller (if any) would only widen the misclassification.
**Suggestion**: Do not derive edge-bearing from `addNode`. Either (a) set `hasEdges = true` only in `addEdge` / `addEdgeAsNode` (the true hop contributions) and leave `addNode` classification-neutral — a bare `addNode` on the boundary is a re-type, and a hop's target `addNode` is always paired with a preceding `addEdge` that already flips the flag (see `GremlinPatternAssembler.appendFoldedHop`, which calls `addEdge` then `addNode`); or (b) flip `hasEdges` only when `addNode` introduces an alias other than the current `boundaryAlias()`. Option (a) is the smaller, clearer change and keeps the flag meaning "contributed a hop/edge".

### BG2 [suggestion]
**Certificate**: C-reserved
**Location**: `GremlinStepWalker.rejectReservedPrefixLabels` (GremlinStepWalker.java:279-298), invoked only from `walk` (GremlinStepWalker.java:153); `subWalk` (GremlinStepWalker.java:254-268) drives child steps without it.
**Issue**: The top-level walk fails *loud* (throws `ReservedAliasException`, which the strategy's throw-safety net re-throws rather than degrading to native) when any user `as(...)` label starts with the reserved `$` prefix, because that namespace is reserved for the minted `$g2m_` aliases and YouTrackDB query-context variables (GremlinStepWalker.java:47-55, 271-298). `subWalk` deliberately omits this scan (its Javadoc lists the reserved-prefix scan among the "top-level-only" machinery, GremlinStepWalker.java:208-211), and `rejectReservedPrefixLabels` iterates only `traversal.getSteps()` — the *outer* step list — so labels on steps *inside* a combinator child sub-traversal (e.g. `g.V().and(__.out("knows").as("$g2m_anon_0"))`) are scanned by neither the top-level pre-flight nor the sub-walk. A `$`-prefixed label on a child step thus escapes the loud guard the design relies on to keep the `$` namespace private.
**Refutation considered**: (1) Does this produce wrong results today? No — in this step no combinator recogniser is registered (`PRODUCTION_RECOGNISERS` has only Graph/Vertex/Has/TraversalFilter steps, GremlinStepWalker.java:98-103), so `and(...)` currently declines at the top level and `subWalk` is not reached in production; and even once Steps 2-4 register the combinators, no recogniser consumes `as(...)` step labels as aliases until the Track-6 as-label work, so a child `$` label is presently inert rather than colliding. `has("$foo", ...)` keys are separately guarded inside the predicate adapter via `WalkerContext.isReservedHasKey` (WalkerContext.java:171-175), which the adapter delegates, so only bare `as(...)` labels are exposed. (2) Is the omission intentional? The Javadoc calls the scan top-level-only, but the rationale given (it is "purely lexical") argues for running it on child labels too, not against it — a lexical scan is exactly as cheap on a child list. So the omission reads as an oversight of the child-label case, not a considered exclusion. **Verdict: CONFIRMED (latent).**
**Reference-accuracy caveat**: mcp-steroid PSI was unreachable; "no recogniser consumes child `as(...)` labels yet" was established by reading the current recogniser set and grepping the strategy package rather than by a PSI usage query, so a label-reading path introduced elsewhere could make this live sooner than Track 6.
**Suggestion**: When the combinators land, extend the reserved-prefix guard to child sub-traversal steps — either scan the child step list inside `subWalk` before dispatch, or have `rejectReservedPrefixLabels` recurse into each step's local children — so the `$`-namespace guarantee holds uniformly and a child `$` label fails loud exactly as a top-level one does.

## Evidence base

#### C-hasedges: addNode unconditionally sets hasEdges, but hasLabel re-typing is a pure-filter addNode → misclassification (BG1)
- CONFIRMED-as-issue: `SubTraversalPredicateAdapter.addNode` (:181-184) sets `hasEdges=true`; `HasStepRecogniser` (:156-164) contributes the pure-filter `hasLabel(L)` boundary re-type through `ctx.addNode(boundary, labelClass)`, and `subWalk` dispatches child `HasStep`s against that same production recogniser, so a `hasLabel`-bearing pure-filter child is reported edge-bearing to the Step-2/3 combinators (OR wrong-decline; NOT wrong-route). See BG1 body; reference-accuracy caveat applies (grep + direct reads, no PSI).

#### C-reserved: the reserved-'$'-prefix label guard is top-level-only and skips child sub-traversal steps (BG2)
- CONFIRMED-as-issue: `rejectReservedPrefixLabels` (:279-298) runs only from `walk` over the outer step list; `subWalk` (:254-268) omits it; a `$`-prefixed `as(...)` label inside a combinator child is rejected by neither path. Latent (no combinator registered yet; child labels not yet consumed as aliases). See BG2 body; reference-accuracy caveat applies.

#### C-alias-delegation: a declined child advances the parent's AliasSequence — but this is by-design and benign
- **Hypothesis**: The capture boundary claims "a declined child leaves the parent's committed state untouched" (class Javadoc; `decline_doesNotCommitPartialStateToOuterContext`). But `nextAnonVertexAlias()` / `nextEdgeAlias()` delegate straight to the parent (SubTraversalPredicateAdapter.java, "reads and alias minting delegate to the parent"), and the parent's `AliasSequence.next()` increments its counter (WalkerContext.java:381-383). So a declined child that mints an alias *does* mutate outer-context state (the counter), seemingly violating the invariant.
- **Refutation**: (1) The delegation is the deliberate core of the design — a per-child counter would collide `$g2m_anon_0` across siblings (track-5.md `## Context and Orientation`, adversarial A4), so the shared counter is *required*, and advancing it is the mechanism that makes `siblingChildren_mintDistinctAliasesFromParentSequence` pass. (2) The "committed state" the capture boundary protects is the pattern / alias-filter / boundary metadata — all of which are buffered in the adapter (`capturedPattern`, `capturedAliasFilters`, `capturedEdgeFilters`, `hasEdges`) and verified untouched by the unit test — not the alias counter, which is intentionally shared. (3) A declined sub-walk is terminal for the whole walk under all-or-nothing (`dispatchAll` returns false → the combinator returns DECLINE → `GremlinStepWalker` discards the entire `WalkerContext`), so a counter advanced by a declined child is thrown away with the rest of the context and can never produce an alias collision or gap in a *surviving* plan. Even a combinator that drove a child speculatively and then abandoned it (without declining the whole walk) would only leak an alias-number *gap*, never a collision, and gaps do not affect correctness (aliases need only be distinct). **Verdict: REFUTED — not a bug; the counter delegation is required and its mutation is discarded on decline.**

#### C-capture-boundary: pattern / filter / boundary contributions from a declined child do not reach the parent
- **Hypothesis**: A declined child might commit a partial pattern node, alias filter, or boundary re-pin to the parent before the unrecognised step is hit.
- **Refutation**: Every mutating `RecognitionContext` method on the adapter writes adapter-local buffers, never the parent: `addNode`/`addEdge`/`addEdgeAsNode` → `capturedPattern` (a fresh `MatchPatternBuilder`, SubTraversalPredicateAdapter.java:181-208); `putAliasFilter` → `capturedAliasFilters` (:211-224); `putEdgeFilter` → `capturedEdgeFilters`; `pinBoundary` / `setSingleReturnColumn` are no-ops (:234-245, swallowed). None call through to `parent`. The parent is a `WalkerContext` whose `patternBuilder`, `aliasFilters`, `boundaryAlias`, and `returnAliases` are only reachable through those (overridden) methods. `decline_doesNotCommitPartialStateToOuterContext` drives a fixture that contributes an alias filter + a node, then declines on a trailing unrecognised `count()`, and asserts the partial contribution reached the *adapter* (non-vacuous) while the parent's four committed surfaces are untouched — matching the traced behavior. **Verdict: MATCHES — the capture boundary holds.**

#### C-refactor: the dispatchAll/subWalk extraction preserves the top-level walk's behavior
- **Hypothesis**: Factoring the dispatch loop out of `walk()` into `dispatchAll` could change the top-level path (reserved-prefix scan, flag resolution, boundary-invariant assertion, or `buildResult`).
- **Refutation**: `walk` (GremlinStepWalker.java:138-202) still runs, in order: empty-traversal gate (:143), `rejectReservedPrefixLabels` (:153), polymorphism resolution + null-decline (:160-164), edge-label-verification + schema resolution (:168-176), `WalkerContext` construction (now with the added `recognisers` arg, :178), then `if (!dispatchAll(...)) return null;` (:183-185), the unchanged terminator-invariant assert + defensive decline (:195-199), then `buildResult` (:201). `dispatchAll` (:213-239) is the former inline loop verbatim — same missing-recogniser decline, same DECLINE decline, same non-advancement assert + defensive decline — with `return null` rewritten to `return false` and the boolean lifted to the caller. No top-level-only step moved into or out of the shared loop; the sub-walk reuses `dispatchAll` and adds only the empty-child gate (:260-263) mirroring `walk`'s empty gate. **Verdict: MATCHES — behavior-preserving refactor.**

#### C-nested: recursive walkChild isolates each level's buffered state and bottoms alias minting at the top-level context
- **Hypothesis**: A nested combinator (`and(and(...), ...)`) driving a grandchild through the adapter's own `walkChild` could corrupt the enclosing adapter's buffers or restart alias minting.
- **Refutation**: `SubTraversalPredicateAdapter.walkChild` (:247-250) returns `GremlinStepWalker.subWalk(child, this, recognisers)`, which allocates a *fresh* adapter wrapping `this` (:258). The grandchild's `addNode`/`putAliasFilter`/etc. therefore land in the grandchild adapter's own buffers, leaving the enclosing adapter's `capturedPattern` / `capturedAliasFilters` / `hasEdges` untouched (verified by `nestedWalkChild_drivesGrandchildAgainstRegistry`, which asserts the grandchild — not the parent adapter — carries the captured hop). Alias minting on the fresh adapter delegates to `this`, which delegates to its own parent, so the chain bottoms out at the top-level `WalkerContext.AliasSequence` — the grandchild mints `$g2m_anon_0` from the shared sequence, confirming no per-level restart. Propagating a grandchild's edge-bearing classification up to the enclosing adapter is the (later) combinator recogniser's responsibility (it reads `grandchild.hasEdges()` and commits via the enclosing `ctx`), not the adapter's, so the fresh-adapter isolation is correct rather than a lost-update. **Verdict: MATCHES — recursion is state-isolated and alias-correct.**
