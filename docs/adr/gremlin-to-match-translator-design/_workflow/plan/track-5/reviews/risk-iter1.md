# Track 5 Risk Review — iteration 1 (logical filters + GremlinPlanCache, D5)

Verdict: **PASS with should-fix findings — no blockers.** The track's shape is sound: the sub-walker interception contract (A4), the single `NotStep` recogniser (A2), the eager-build native-decline net (R4), and the RID cache-bypass (R3) all trace to correct HEAD behavior. The concentrated risk is the D5 plan-cache fingerprint. The plan names it "the post-walk generic-statement fingerprint" and reasons about structural tokens (A3), but three fingerprint mechanics are either missing or backwards against HEAD, and each fails as a *silent wrong result*, not a decline: distinct class names collapse to one key (R1), the fingerprint has no source on the additive path (R2), and value-independence depends on a total switch to positional params that the plan states but does not bound (R3). One sub-walker justification is stale against HEAD's AND-composing `putAliasFilter` (R4). The NOT edge cases (R5) and the determinism suite (R6) validate, with a coverage-gap note on R6.

PSI was not used this session; every symbol audit is `Read`/`Grep` against source, so findings that turn on a caller/override search carry a reference-accuracy caveat in their evidence entry.

## Findings

### R1 [should-fix]
**Certificate**: C1 (Exposure — the generic-statement fingerprint collapses structural string tokens)
**Location**: track-5.md `## Context and Orientation` (the D5 paragraph, ≈46: "Structural tokens stay inline and must **not** parameterize … class names for `classEquals`"), `## Plan of Work` step 5; `MatchWhereBuilder.classEquals` (`MatchWhereBuilder.java:68-79`); `SQLBaseExpression.toGenericStatement` (`SQLBaseExpression.java:107-120`, string branch `:112-113`); `SQLBinaryCondition.toGenericStatement` (`SQLBinaryCondition.java:190-195`).
**Issue**: A3 keys the cache on the post-walk *generic statement*, and the design's structural-token rule keeps class names "inline" (not parameterized) so a structural token bound as a param cannot serve a wrong plan. That rule is necessary but not sufficient, and the missing half is the actual failure. `classEquals("Person")` builds `@class = <string 'Person'>` via `stringExpression` (`MatchWhereBuilder.java:74-77`, `:459-462`), i.e. the right operand is a `SQLBaseExpression` holding a raw `String`. `SQLBaseExpression.toGenericStatement` renders a string operand as `PARAMETER_PLACEHOLDER` (`:112-113`), so the generic statement of `@class = 'Person'` is `@class = ?` — the class name is erased from the fingerprint. Consequence in non-polymorphic mode (where `HasStepRecogniser` adds the `@class = 'L'` WHERE, `HasStepRecogniser.java:161-163`): `hasLabel("Person")` and `hasLabel("Company")` produce the **same** cache key, and the cached plan carries the class name as an inline literal that the reusing query does not rebind — so the second label reuses the first label's plan and returns the first label's vertices. This is the exact silent-empty / silent-wrong outcome A3 was written to prevent, reached from the token direction rather than the value direction. The polymorphic default routes the class into the pattern node instead (`HasStepRecogniser.java:160` re-types the boundary), so the exposure there depends on whether the pattern node's class name is emitted inline in `SQLMatchExpression.toGenericStatement` — not confirmed this pass, so the non-polymorphic case is the certain one and the polymorphic case is an open question the plan must resolve. Reference-accuracy caveat: the pattern-node generic rendering and the full caller set of `classEquals` were not PSI-audited.
**Proposed fix**: Sharpen A3 from "structural tokens stay inline" to "structural tokens must remain *discriminating* in the fingerprint." Carry the class name in a form whose `toGenericStatement` emits it verbatim — `SQLIdentifier.toGenericStatement` appends `value` inline (`SQLIdentifier.java:141-142`), unlike the string-literal path — or fold the class name into the key outside the generic statement. Add an R6 case asserting `hasLabel(A)` and `hasLabel(B)` occupy **distinct** cache entries in both polymorphic and non-polymorphic modes, and confirm the polymorphic pattern-node path renders the class inline.

### R2 [should-fix]
**Certificate**: C2 (Assumption — the "post-walk generic-statement fingerprint" has an existing source)
**Location**: track-5.md `## Context and Orientation` (D5 paragraph), `## Interfaces and Dependencies` (modified: `GremlinToMatchStrategy` "cache get/put replaces the `useCache=false` eager build"; `SharedContext` cache field), `## Signatures`; `MatchExecutionPlanner.java:488-495` (additive-constructor caching precondition), `:549-556`, `:704-709`; `MatchPlanInputs.java` (record — no generic-statement method); `SQLStatement.java:38-39,164-165` (`getOriginalStatement` / `toGenericStatement` both dump `originalStatement`).
**Issue**: The plan speaks of "the post-walk generic-statement fingerprint" as if it exists, but the additive path produces no statement to fingerprint. `GremlinToMatchStrategy.buildPlan` runs `new MatchExecutionPlanner(inputs).createExecutionPlan(ctx, false, /*useCache*/ false)` (`GremlinToMatchStrategy.java:387-392`), and the additive constructor's javadoc is explicit: "the inherited `statement` field stays `null` … callers MUST invoke `createExecutionPlan` with `useCache=false`. Otherwise the cache lookup … dereferences `statement` and throws a NullPointerException. Plan caching for non-SQL front-ends would need a **separate cache-key mechanism keyed off the front-end's own fingerprint** … intentionally out of scope here" (`MatchExecutionPlanner.java:488-495`). The YQL cache keys on `statement.getOriginalStatement()` (`:556`, `:709`) — a raw statement string that the Gremlin path never sets. So `GremlinPlanCache` cannot ride `MatchExecutionPlanner.useCache`; it must be a genuinely separate cache (which the plan says: a new `SharedContext` field mirroring `YqlExecutionPlanCache`) keyed on a fingerprint Track 5 must *construct*. `MatchPlanInputs` has no serializer, and the planner constructor deliberately skips building an `SQLMatchStatement`. Two viable sources — (a) build an `SQLMatchStatement` from the translated inputs and call `SQLMatchStatement.toGenericStatement` (`SQLMatchStatement.java:451-503`, which does serialize the AST), or (b) a bespoke walk over the `MatchPlanInputs` fields — are both unbuilt work the plan does not name, and both must guarantee determinism across walks: `aliasFilters` is a `LinkedHashMap` (`WalkerContext.java:40`, insertion-ordered, good) but the planner copies it into a plain `HashMap` (`MatchExecutionPlanner.java:505`), so a fingerprint taken *after* the planner runs would read a nondeterministically-ordered map and two walks of one shape could key differently (cache never hits — the single-use thrash the track wants to avoid). Reference-accuracy caveat: `SQLMatchStatement.toGenericStatement` behavior read from source, not exercised.
**Proposed fix**: Commit the decomposition to one fingerprint source and pin *where* in the pipeline it is taken (before the planner's `HashMap` copy, or off an insertion-ordered structure). State that the fingerprint is computed from the pre-plan translated AST, not from planner-internal state, and add an R6 test that the same shape fingerprints identically across two independent walks (stability), separate from the value-independence test.

### R3 [should-fix]
**Certificate**: C3 (Exposure — partial positional-parameter switch bakes values into a shared plan)
**Location**: track-5.md `## Context and Orientation` (D5 paragraph: "switches `GremlinPredicateAdapter` to emit `SQLPositionalParameter` for **predicate comparison values**"), `## Plan of Work` step 5; `SQLBaseExpression.toGenericStatement` (`:108-116`); `SQLNumber.toGenericStatement` (`SQLNumber.java:33`); `SQLPositionalParameter` (`:44-55` — `toGenericStatement` → placeholder, `getValue` → `params.get(paramNumber)`); `CommandContext.setInputParameters` (`CommandContext.java:106`).
**Issue**: Value-independence has two independent requirements that the plan states as one. (1) The *key* must not vary with the value — `toGenericStatement` already collapses number literals (`SQLNumber.java:33`) and string literals (`SQLBaseExpression.java:112-113`) to `?`, so the key is value-blind even for Track 4's inline literals. (2) The cached *plan* must actually rebind the value per execution — and this is the requirement inline literals fail: a plan built from `eq(30)` with an inline `SQLNumber(30)` filters `age = 30` regardless of the reusing query's value. The two requirements together define a trap: any predicate value that `toGenericStatement` placeholder-collapses **but** that the adapter leaves as an inline literal (a missed form in the "switch to `SQLPositionalParameter`" work) fingerprint-collides same-shape/different-value queries onto one entry *and* serves them a plan with the first value baked in — silent wrong results, no decline. So the switch to positional params must be *total* over every value form whose generic rendering is a placeholder, not merely the common `Compare` path. Second, execution correctness rests on slot alignment: the cached plan holds `?` nodes with fixed `paramNumber`s (from the walk that built it); a reusing walk's value map is installed via `ctx.setInputParameters` and read by `paramNumber` (`SQLPositionalParameter.java:49-55`), so the reusing walk must allocate identical `paramNumber`s for the identical shape — i.e. `bindParam` slot numbering must be a pure function of walk order, independent of the values. A value-dependent allocation order (e.g. a branch that emits a slot for one value form and not another *within the same fingerprint*) misaligns the map against the plan and binds value A into value B's slot.
**Proposed fix**: State in step 5 that the positional-parameter switch covers **every** comparison value the adapter can emit (scalar `Compare`, `Contains` list elements, `Text`/`TextP` operands, the `between`/`inside`/`outside` decompositions), and that structural tokens (class names, RIDs, `~label` values) are explicitly excluded (they are the R1 discriminators). Add an R6 assertion that `bindParam` slot numbering is identical across two same-shape/different-value walks (deterministic walk-to-slot ordering), and one that a reused plan bound with a second value returns the second value's multiset (not the first's).

### R4 [should-fix]
**Certificate**: C4 (Assumption — the OR-child capture rationale matches HEAD's `putAliasFilter`)
**Location**: track-5.md `## Context and Orientation` (A4 bullet: "`putAliasFilter` is captured per-combinator … never committed to `aliasFilters` (which override rather than compose)"), `## Plan of Work` step 1-2; `WalkerContext.putAliasFilter` (`WalkerContext.java:292-305`), `setSingleReturnColumn` (`:320-330`), `AliasSequence` (`:339-351`).
**Issue**: The sub-walker alias trap (A4) is real and correctly diagnosed against HEAD — `AliasSequence` is per-context and its counter restarts at 0 on each fresh `WalkerContext` (`:336-337`, `:344-350`), so a naive per-child nested context mints `$g2m_anon_0` for both children of `and(__.out("a"), __.out("b"))`, collapsing two MATCH bindings into one; and `setSingleReturnColumn` clears and re-adds the projection (`:320-330`), so a hop child would clobber the outer return column. The mitigations the plan names (delegate minting to the parent `AliasSequence`; swallow `pinBoundary`/`setSingleReturnColumn` on the sub-context) are sound. The defect is the stated *reason* for the third bullet: the plan says OR children must not commit to `aliasFilters` because those entries "override rather than compose." HEAD contradicts this — Track 4 made `putAliasFilter` **AND-compose** a second same-alias contribution (`:292-305`: an existing entry is merged via `WHERE.and(...)`, not replaced). So the actual reason an OR child cannot route through `putAliasFilter` is that it AND-composes (the wrong connective for OR), not that it overwrites. The conclusion (OR children collect composable `SQLBooleanExpression`s for `MatchWhereBuilder.or`) is right; the justification is stale, and an implementer reading "override rather than compose" against a `putAliasFilter` that visibly AND-composes will either distrust the plan or build on the wrong model. (The `WalkerContext` class comment at `:37-39` still says aliasFilters entries "override builder entries on the same alias" — a separate stale line about the *builder*-merge boundary, worth checking in the same pass.)
**Proposed fix**: Correct the A4 bullet to state that `putAliasFilter` AND-composes same-alias contributions in HEAD, so OR-child WHERE fragments must be collected out-of-band and combined with `MatchWhereBuilder.or`; committing them individually would AND them. Keep the alias-minting and boundary-swallow bullets as-is (they verify correct).

### R5 [suggestion]
**Certificate**: C5 (Assumption — the eager-build net degrades a disqualifying NOT to native, and the cache preserves it)
**Location**: track-5.md `## Decision Log` (R4 entry), `## Context and Orientation` (NOT paragraph, ≈44), `## Signatures` (`manageNotPatterns`); `GremlinToMatchStrategy.apply` (`GremlinToMatchStrategy.java:199-254`), `applyTranslation`/`buildPlan` (`:372-392`); `MatchPlanInputs.notMatchExpressions` (`MatchPlanInputs.java:48,154`); `MatchExecutionPlanner` additive constructor copies `notMatchExpressions` (`:516`).
**Issue**: R4's claim — the edge-bearing-NOT decline is defense-in-depth, and the real barrier is the eager-build net — holds against HEAD. `apply` wraps `applyOrDecline` in a `try` catching `RuntimeException` (`:208-225`), `applyOrDecline` calls `translator.translate` then `applyTranslation` (`:249-253`), and `applyTranslation` runs `buildPlan` → `createExecutionPlan` eagerly (`:376`, `:387-392`) *inside* that try. So a `manageNotPatterns` throw on a disqualifying NOT shape (origin alias absent from the positive pattern, or origin item carrying an inline filter) is caught and degraded to a native decline, exactly as R4 states — provided the throw is a `RuntimeException` subtype (the plan and the Track 4 risk review assert `CommandExecutionException` is; not re-audited this pass). The consuming sink is ready: `MatchPlanInputs.notMatchExpressions` exists (`:48`, builder `:154`) and the additive constructor copies it (`:516`). So the net still protects, but only if get → build → put all sit inside `applyOrDecline`. Reference-accuracy caveat: `manageNotPatterns`'s throw type and its two precondition sites were read via the plan's line refs, not re-opened this pass.
**Proposed fix**: Add one line to the D5 step (or the R4 entry) that the cache lookup, plan build, and cache store all remain inside `apply`'s `RuntimeException` net, so a disqualifying NOT shape still degrades to native on the first (uncached) walk and is never stored.

### R6 [suggestion]
**Certificate**: C6 (Assumption — the R6 determinism suite covers the fingerprint's real collision surface)
**Location**: track-5.md `## Validation and Acceptance` (the D5/A3/R6 bullets: "`eq(null)` and `eq(v)` occupy distinct cache entries; collection-size classes do not collide"), `## Decision Log` (R3 cache-bypass); `StartStepRecogniser.buildRidInExpression` (`StartStepRecogniser.java:251-268`), `HasStepRecogniser.translateHasId` (`HasStepRecogniser.java:197-221`); `SQLRid.toGenericStatement` (`SQLRid.java:56-62`).
**Issue**: The stated R6 cases pin the *value-fork* collisions (`eq(null)` vs `eq(v)`, collection-size classes) but not the *structural-token* collisions the fingerprint most easily gets wrong — the R1 case (distinct `hasLabel` classes must land in distinct entries, both modes) belongs in this suite. Separately, R3's RID cache-bypass is load-bearing for **correctness**, not only thrash-avoidance: RIDs flow inline through `buildRidInExpression` (`:251-268`, a record-attribute `@rid IN [...]` with `SQLRid` literals) on both the `g.V(id)` start path and the `hasId` path (`HasStepRecogniser.translateHasId:214-220` calls the same builder), and `SQLRid.toGenericStatement` can itself render `PARAMETER_PLACEHOLDER` for the legacy / collection-position form (`SQLRid.java:57-58`).
