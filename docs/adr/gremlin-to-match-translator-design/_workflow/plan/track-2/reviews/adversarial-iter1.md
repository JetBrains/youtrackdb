## Findings

### A1 [blocker]
**Certificate**: CH-D4 (Decision challenge — D4 strategy ordering vs `StartStepRecogniser` gate)
**Target**: Decision D4 (translator-first ordering) and the track-2 Decision Log `StartStepRecogniser` gate
**Challenge**: The gate and the ordering cannot both be right. D4 (plan lines 155-165) and design §Overview point 1 (lines 42-58) both fix the translator to run **first** — `GremlinToMatchStrategy.applyPrior()={}` / `applyPost()={}`, and each of the three half-measures (including `YTDBGraphStepStrategy`) lists `GremlinToMatchStrategy` in its own `applyPrior()`, so TinkerPop's topological sort runs the translator before all three. But the track-2 Decision Log ("simplification") pins the start-step gate as `stepIndex == 0 && step instanceof YTDBGraphStep`. `YTDBGraphStep` is produced **only** by `YTDBGraphStepStrategy.apply()` (`new YTDBGraphStep<>(graphStep)`, `YTDBGraphStepStrategy.java:114` — the sole constructor call site in the whole main checkout). A plain `g.V()` reaches TinkerPop's optimization phase as an upstream `GraphStep`, and under translator-first ordering `YTDBGraphStepStrategy` has not yet run when the translator walks the list. So the start step is a plain `GraphStep`, `instanceof YTDBGraphStep` is `false`, `StartStepRecogniser` declines, and under D3 all-or-nothing the whole traversal declines. Track 2's sole deliverable — translate `g.V()` / `g.V(ids)` end to end — never fires; every recognized shape falls through to the native pipeline exactly as today.
**Evidence**: `YTDBGraphStepStrategy.java:106-118` (GraphStep -> YTDBGraphStep replacement, gated on `current instanceof GraphStep`, sole producer); grep confirms `new YTDBGraphStep` appears only at `YTDBGraphStepStrategy.java:114` across `core/src` (excluding worktrees). No traversal-source / graph-provider builds a `YTDBGraphStep` at construction time (`YTDBGraphImplAbstract.java:68-79` registers strategies but installs no start-step subclass). `YTDBGraphCountStrategy.applyPrior()` (`:114-116`) and `YTDBGraphMatchStepStrategy.applyPrior()` (`:146-149`) both return `singleton(YTDBGraphStepStrategy.class)` and both gate on `instanceof YTDBGraphStep` — confirming `YTDBGraphStep` is a *post-`YTDBGraphStepStrategy`* artifact, i.e. it exists only on the decline path relative to a translator that runs first. Reference-accuracy caveat: the constructor-caller enumeration is grep-based (the PSI `ReferencesSearch` call timed out on the ~60 s MCP limit); the claim rests on a whole-`core/src` `new YTDBGraphStep` grep, which for a package-visible constructor with a single literal call form is reliable.
**Proposed fix**: Pick one and record it as an immutable-DR revision before Phase B (this touches D4 and D9, so it needs ESCALATE, not an inline step edit):
(a) Gate `StartStepRecogniser` on the plain TinkerPop `GraphStep` (keying the D9 registry on `GraphStep.class`, see A2), keeping translator-first ordering. The recogniser then reads `graphStep.getIds()` / return class off the vanilla `GraphStep` and pins `polymorphic` via `YTDBStrategyUtil.isPolymorphic` — no dependence on `YTDBGraphStepStrategy` having run. This is the ordering the design actually argues for.
(b) Order the translator **after** `YTDBGraphStepStrategy` (translator lists `YTDBGraphStepStrategy.class` in its `applyPrior()`), so the start step is already a `YTDBGraphStep`. This contradicts D4's "translator first / half-measures are the pure fallback" and the "decline preserves the original step list **verbatim**" clause (see the second half of CH-D4), so it is the weaker option and needs D4 rewritten, not just reordered.

### A2 [should-fix]
**Certificate**: CH-D9 (Decision challenge — `getClass()`-keyed dispatch key identity)
**Target**: Decision D9 (`Map<Class<? extends Step>, StepRecogniser>` keyed on `step.getClass()`)
**Challenge**: D9's dispatch is `map.get(step.getClass())` on the **concrete runtime class**, and its stated virtue is "safe failure on unknown subclasses" (design lines 377-389). That same exactness makes the registry key load-bearing and coupled to A1: the key for the start recogniser must be the exact concrete class TinkerPop hands the walker for `g.V()`. Under translator-first ordering that class is the upstream `GraphStep` (or its concrete TinkerPop subtype), **not** `YTDBGraphStep`. If Track 2 registers `StartStepRecogniser` under `YTDBGraphStep.class` (the natural reading of the Decision Log gate), `map.get(GraphStep.class)` returns `null` and declines — the same failure as A1, but now baked into the registry key rather than an `instanceof` guard. Track 2's "simplification" to an ordered `StepRecogniser` list with an `instanceof YTDBGraphStep` gate masks the key-identity question (a list has no key), so the class-keyed map that D9 mandates and Track 3 restores inherits an unresolved question: which concrete class does `g.V()` present, and does the map key match it exactly.
**Evidence**: design §"Recogniser dispatch" lines 367-389 (concrete-class key, no `instanceof` fallback, deliberate `null`-on-unknown-subclass); `YTDBGraphStepStrategy.java:106` gates its own rewrite on `current instanceof GraphStep<?,?>` (accepting any GraphStep subtype) — the map-keyed translator gets no such polymorphic tolerance, so its key must be exact. track-2.md Decision Log ("simplification") swaps the map for a list to defer this, but the deferral carries the ambiguity into Track 3.
**Proposed fix**: Resolve jointly with A1. Once the ordering is fixed (A1 option (a)), state the exact key in the track file — register `StartStepRecogniser` under the concrete class of `g.V()`'s start step as it exists at translator time, and add a smoke assertion that `map.get(traversal.getStartStep().getClass())` is non-null for a bare `g.V()`. If TinkerPop's `g.V()` start step is an exact `GraphStep` (not a subclass), key on `GraphStep.class`; verify empirically in the first step's test rather than assuming.

### A3 [should-fix]
**Certificate**: AS-CACHE (Assumption test — additive ctor `statement` field and `useCache`)
**Target**: Assumption — the additive `MatchExecutionPlanner(MatchPlanInputs)` ctor is safe to route through the unchanged `createExecutionPlan`
**Challenge**: `createExecutionPlan` dereferences `statement` unconditionally when `useCache && !enableProfiling` (`MatchExecutionPlanner.java:478`: `statement.executinPlanCanBeCached(session)`). The additive `MatchPlanInputs` ctor (D5-deferred in Track 2) leaves `statement` null. The design's own §Workflow sequence diagram shows the strategy calling `createExecutionPlan(ctx, prof, useCache=true)` (design line 470), which would NPE on line 478. The track's scope-down correctly pins `useCache=false`, so the two documents disagree, and nothing structural prevents the `useCache=true` call the design diagram depicts — the guard is caller convention only.
**Evidence**: `MatchExecutionPlanner.java:478-479` (`statement` deref under `useCache && !enableProfiling`); the two existing non-SQL ctors (`:385`, `:398`) also leave `statement` null and never set it — so today's GQL callers already rely on the same "pass `useCache=false`" convention, which means the additive ctor inherits an established, working pattern (this is why the finding is should-fix, not blocker). design.md:470 shows `useCache=true` in the translated-path sequence, contradicting the deferred cache.
**Proposed fix**: Either (a) update the design §Workflow diagram to `useCache=false` for the additive-ctor path (matching the D5 deferral the track already records), or (b) add a cheap guard in `createExecutionPlan` / the additive ctor path so a null `statement` short-circuits the cache branch instead of NPE-ing — turning the convention into an enforced invariant. Option (a) is the minimal doc fix; option (b) hardens the shared engine for every null-`statement` caller.

### A4 [suggestion]
**Certificate**: AS-RIDS (Assumption test — `g.V(ids)` translation path)
**Target**: Assumption — the "single-RID fast path" applies to `g.V(ids)`
**Challenge**: The Context section (track-2.md:67) advertises "The `Pattern` single-RID fast path resolves `aliasRids[a]` to `SELECT FROM #X:Y`", and the minimal-translation list puts `g.V(id)` on `aliasRids` but `g.V(id1, id2, …)` on `aliasFilters[boundary] = WHERE @rid IN [...]`. The planner's `aliasRids` is `Map<String, SQLRid>` — exactly **one** RID per alias (`MatchExecutionPlanner.java:307`), consumed as `fromItem.setRids(singletonList(targetRid))` in the synthetic-SELECT builder. So multi-id `g.V(ids)` genuinely cannot use the RID seek and correctly falls to a `WHERE @rid IN` filter — which is a class scan plus row filter, not a RID lookup. The translation is **correct** (survives), but a reader could infer from the Context sentence that all id-bearing `g.V` shapes get the fast path, and that the translated `g.V(id1,id2)` is index-optimal. It is not.
**Evidence**: `MatchExecutionPlanner.java:307` (`private Map<String, SQLRid> aliasRids`); the synthetic-SELECT builder (~`:4549-4560`) sets a single RID via `fromItem.setRids(Collections.singletonList(targetRid))`; `aliasRids` is consumed by `estimateRootEntries` (`:498`) and the hash-join path (`:716`). The multi-id WHERE-IN path is the correct and only available route.
**Proposed fix**: One clarifying clause in the track file — note that multi-id `g.V(ids)` uses a `WHERE @rid IN` full-scan filter (not the single-RID seek), so a future index-backed multi-RID path is a Phase-2 optimization, not a Track-2 regression. No code change.

### A5 [suggestion]
**Certificate**: SC-PACK (Scope challenge — cross-cutting scaffolding vs mid-track deferrals)
**Target**: Scope — Track 2's packed footprint
**Challenge**: The scope indicator claims ~19 files (plan line 361) covering the record + ctor, strategy skeleton, walker + registry + context + start recogniser, plan cache, anon-alias generator, and boundary. But the track's own Decision Log then defers **two** of those cross-cutting pieces mid-track: `GremlinPlanCache` (D5) and the standalone `AnonAliasGenerator` both move to Track 3. So the real Track-2 footprint is materially smaller than the indicator, and two of the "scaffolding every later track extends" items are not actually built here. The challenge: is the boundary step (`YTDBMatchPlanStep`, step 2, `risk: high`) even exercised in Track 2? The Context section says "`ELEMENT` is wired in Track 3; Track 2's `g.V()` translation produces a vertex-source pattern that Track 3's boundary later emits" and design line 552-554 says "Track 3 is the first track that wires a boundary step at all". If the boundary emits nothing until Track 3, building the full `YTDBMatchPlanStep` lifecycle (lazy open, AutoCloseable close, clone re-bind) in Track 2 with no output type wired is speculative scaffolding whose correctness cannot be end-to-end tested here.
**Evidence**: track-2.md:24-41 (three scope-down / simplification deferrals); track-2.md:69 + design.md:547-554 (boundary output type `ELEMENT` deferred to Track 3); Concrete Step 2 (`YTDBMatchPlanStep`, `risk: high`) vs Validation lines 106-107 (the acceptance tests assert translate-and-return parity, which needs the boundary to emit — so either the boundary IS exercised in Track 2 and the "Track 3 wires it" prose is wrong, or the parity tests cannot run in Track 2). This internal tension is the finding; the decision to pack survives because the deferrals are documented and reduce risk, but the boundary/output-type wiring story is contradictory across the track and design.
**Proposed fix**: Reconcile whether Track 2's `g.V()` acceptance tests (translator-on vs -off multiset parity, Validation lines 106) actually run in Track 2 — if they do, the boundary emits `ELEMENT` in Track 2 and the "Track 3 wires the first boundary" prose (design line 552, track line 69) is stale and should move to Track 2; if they don't, the Validation section's parity lines belong to Track 3 and Track 2's acceptance shrinks to "produces the expected `MatchPlanInputs` / declines correctly". Pick one so Phase B knows whether it is testing end-to-end results or just IR shape.

## Evidence base

#### CH-D4 (Decision D4 — translator-first ordering vs YTDBGraphStep gate) — WRONG
- **Chosen approach**: Translator registers first (`applyPrior={}`, `applyPost={}`; each half-measure lists it in its own `applyPrior()`); `StartStepRecogniser` gates on `stepIndex == 0 && step instanceof YTDBGraphStep`.
- **Best rejected alternative**: Gate on the plain TinkerPop `GraphStep` (option (a) in A1's fix), which is the only class present at translator time under translator-first ordering.
- **Counterargument trace**:
  1. In the scenario `g.V().toList()` with the translator registered first, TinkerPop's `applyStrategies` runs `GremlinToMatchStrategy` before `YTDBGraphStepStrategy` (D4 topological order). At that moment the start step is the upstream `GraphStep` TinkerPop built for `g.V()` — evidence: `YTDBGraphStepStrategy.java:106-118` is the only code that turns a `GraphStep` into a `YTDBGraphStep`, and `new YTDBGraphStep` occurs only at `:114` in the whole `core/src`.
  2. The recogniser evaluates `step instanceof YTDBGraphStep` → `false`, returns `false`, and under D3 the walker declines the whole traversal (design §Workflow lines 461-463).
  3. Outcome: `g.V()` is never translated; it declines to the native path and `YTDBGraphStepStrategy` handles it as today. Track 2 ships no observable behavior change — its Validation lines 106-107 (translate-and-return parity) can never pass.
- **Codebase evidence**: `YTDBGraphStepStrategy.java:114` (sole `YTDBGraphStep` producer); `YTDBGraphCountStrategy.java:57-59` and `YTDBGraphMatchStepStrategy.java:79-80` both gate on `instanceof YTDBGraphStep` **and** both `applyPrior()` = `{YTDBGraphStepStrategy}` — i.e. `YTDBGraphStep` is by construction a post-`YTDBGraphStepStrategy` artifact, so a translator that runs before `YTDBGraphStepStrategy` cannot observe one.
- **Second half — does "decline preserves the original step list verbatim" hold?** Yes, but only for the decline branch, which is the branch that always fires here. On decline the translator touches nothing, `YTDBGraphStepStrategy` then converts `GraphStep`→`YTDBGraphStep` and folds `has`, and Count/Match strategies see the `YTDBGraphStep` list — internally consistent. The contradiction is that the *recognize* branch is unreachable for the shapes Track 2 targets, because recognition demands an artifact (`YTDBGraphStep`) that only the decline path's downstream strategy produces. Chicken-and-egg.
- **Survival test**: NO. The chosen gate is incompatible with the chosen ordering; one of the two immutable decisions (D4 or the D9-registered gate) must change.

#### CH-D9 (Decision D9 — concrete-class registry key identity) — WEAK
- **Chosen approach**: `Map<Class<? extends Step>, StepRecogniser>` keyed on `step.getClass()`, deliberately declining unknown subclasses.
- **Best rejected alternative**: An ordered `instanceof` chain (which D9 rejected) — or, more narrowly, keying on the correct concrete class once A1 is resolved.
- **Counterargument trace**:
  1. Under the corrected translator-first ordering (A1 option (a)), `g.V()`'s start step at translator time is the upstream `GraphStep`, whose exact runtime class is what `map.get(step.getClass())` looks up.
  2. If Track 2 (or Track 3, which restores the map) registers `StartStepRecogniser` under `YTDBGraphStep.class`, `map.get(GraphStep.class)` misses and declines — reproducing A1 at the key level.
  3. Outcome: the "safe failure on unknown subclasses" virtue turns into "silent failure on the one class we must handle", unless the key exactly matches TinkerPop's concrete start-step class.
- **Codebase evidence**: design lines 367-389 (exact-class key, no `instanceof` tolerance); `YTDBGraphStepStrategy.java:106` accepts any `GraphStep` subtype via `instanceof`, but the map-keyed translator has no such tolerance.
- **Survival test**: WEAK. D9's design survives as a mechanism, but its correctness for the start recogniser is entirely contingent on A1's ordering fix and on stating the exact key; the track file leaves the key unstated (list-with-`instanceof` simplification), so the rationale needs strengthening to name the concrete class and assert it in a test.

#### AS-CACHE (Assumption — additive-ctor `statement` null is safe through `createExecutionPlan`) — FRAGILE
- **Claim**: The `MatchPlanInputs` ctor can leave `statement` null and route safely through the unchanged `createExecutionPlan`.
- **Stress scenario**: The strategy calls `createExecutionPlan(ctx, prof, useCache=true)` (exactly what design §Workflow line 470 depicts) with a `MatchPlanInputs`-constructed planner.
- **Code evidence**: `MatchExecutionPlanner.java:478` dereferences `statement.executinPlanCanBeCached(session)` whenever `useCache && !enableProfiling`; `statement` is null for the additive ctor. NPE. The two pre-existing non-SQL ctors (`:385`, `:398`) already leave `statement` null, so today's GQL path proves the "pass `useCache=false`" convention works — but it is convention, not an enforced guard.
- **Verdict**: FRAGILE. Holds as long as every additive-ctor caller passes `useCache=false`, which the track's scope-down pins; but the design diagram contradicts it (`useCache=true`), so the documents disagree and a future caller reading the design could NPE.

#### AS-RIDS (Assumption — `g.V(ids)` uses the single-RID fast path) — HOLDS
- **Claim**: id-bearing `g.V` shapes resolve through `aliasRids` / the single-RID `SELECT FROM #X:Y` fast path.
- **Stress scenario**: `g.V(id1, id2, id3)` — three ids on one alias.
- **Code evidence**: `aliasRids` is `Map<String, SQLRid>` (`MatchExecutionPlanner.java:307`) — one RID per alias; the synthetic-SELECT builder sets `fromItem.setRids(singletonList(targetRid))`. Multiple ids cannot fit, so the track correctly routes multi-id to `aliasFilters[boundary] = WHERE @rid IN [...]` (a class scan + filter). Single-id `g.V(id)` does use `aliasRids` and the RID seek.
- **Verdict**: HOLDS. The translation is correct; only the Context section's phrasing invites the wrong inference that multi-id also seeks by RID. Doc clarity, not a defect.

#### SC-PACK (Scope — packed scaffolding with mid-track deferrals and deferred boundary output) — HOLDS
- **Chosen approach**: Land all cross-cutting scaffolding in Track 2 (record+ctor, strategy, walker/registry/context, start recogniser, cache, anon-gen, boundary).
- **Best rejected alternative**: Split — land only what `g.V()` needs end to end in Track 2, defer cache + anon-gen + boundary-lifecycle to the track that first exercises them.
- **Counterargument trace**:
  1. The track defers `GremlinPlanCache` (D5) and standalone `AnonAliasGenerator` to Track 3 (Decision Log), and the boundary's only output type (`ELEMENT`) is "wired in Track 3" (Context line 69, design line 552).
  2. So the ~19-file indicator overstates the real footprint, and the `risk: high` boundary step (Concrete Step 2) is built with no output type and — per the Track-3-wires-it prose — no end-to-end emission in Track 2.
  3. Outcome: either the Validation parity tests (lines 106-107) do run in Track 2 (so the boundary DOES emit `ELEMENT` here and the "Track 3 first wires it" prose is stale), or they don't (so those acceptance lines belong to Track 3). The scaffolding decision itself is fine; the boundary/output-type/acceptance story is internally contradictory.
- **Codebase evidence**: track-2.md:24-41 (deferrals), :69 + design.md:547-554 (boundary output deferred), :106-107 (parity acceptance).
- **Survival test**: HOLDS. Packing scaffolding into the wiring track is a defensible planner choice (documented deferrals reduce risk), so the decision survives; but the contradiction over whether Track 2 emits from the boundary at all needs one clarifying reconciliation so Phase B tests the right layer.
