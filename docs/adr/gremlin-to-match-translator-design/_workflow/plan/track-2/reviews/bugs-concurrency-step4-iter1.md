## Findings

### BC1 [should-fix] Repeated ids in `g.V(id, id)` translate to `@rid IN [id, id]`, which emits the vertex once vs native's twice

**File:** `core/src/main/java/com/jetbrains/youtrackdb/internal/core/gremlin/translator/strategy/StartStepRecogniser.java` (line 429; `buildRidInExpression`, line 531)

**Issue.** For a multi-id source the recogniser builds `@rid IN [...]` from the full normalised id list without de-duplicating (`buildRidInExpression` iterates `rids` verbatim, line 537-540). MATCH evaluates `@rid IN [...]` as set membership: the planner scans the class once and emits each vertex whose `@rid` is in the set exactly once, regardless of how many times the id appears in the list. Native Gremlin resolves the id array element-by-element: `YTDBGraphImplAbstract.elements` (line 185-196) does `Stream.of(elementIds)...map(load).filter(nonNull)` with no dedup, so a repeated id loads and emits the same vertex once per list entry. The two paths disagree on the result multiset for any traversal that passes a duplicate id.

**Failure scenario.** `g.V("#25:3", "#25:3")` against a graph where `#25:3` exists. Native returns `[v(#25:3), v(#25:3)]` (two traversers â€” `elements` maps each of the two array entries to a loaded `YTDBVertexImpl`). Translated returns `[v(#25:3)]` (one row â€” the MATCH root scan matches `#25:3` once against the IN-set). This violates the multiset-equality contract ("the same elements the same number of times for every recognized shape"). The single-id path (`rids.size() == 1`) is unaffected â€” one native emission, one `aliasRids` lookup.

**Refutation considered.** I checked whether native dedups: `YTDBGraphImplAbstract.elements` (the `getElementsByIds` target wired in `YTDBGraphStep` line 118 / `vertices()` line 59-61) has no `distinct()` and no `Set` collection on the id path â€” it streams and maps one-to-one, so duplicate emission is native's actual behavior. I checked whether the folded `HasContainer` filter (`IteratorUtils.filter`, YTDBGraphStep line 117) could dedup â€” it filters element-wise, preserving cardinality. I checked whether MATCH `@rid IN [dup, dup]` could emit twice: the root is a single class-scan pattern node testing membership per scanned record, so each vertex is visited once; the IN list's duplicate entry does not create a second row. Verdict: CONFIRMED â€” the divergence is real. Rated should-fix rather than blocker because a duplicate literal id inside one `g.V(...)` call is an uncommon shape in practice; it is nonetheless a legal input that produces divergent multisets, and it is the kind of corner the Step-5 equivalence suite must not paper over.

**Suggestion.** De-duplicate while preserving the first-seen set semantics before building the IN filter â€” e.g. collect `normaliseIds` output into a `LinkedHashSet<RecordIdInternal>` (or dedup in `buildRidInExpression`). Note this makes translated `g.V(id, id)` return one element; if the intended contract is instead to match native's duplicate emission, the shape cannot be expressed by `@rid IN` at all and must decline to native â€” decide which multiset is canonical and pin it with a Step-5 parity test over a repeated-id input. Either resolution is fine; silently emitting a different multiset than the current code does is not.

### BC2 [should-fix] Non-polymorphic bare `g.V()` narrows to `@class = 'V'` (exact class), excluding subclass instances native returns

**File:** `core/src/main/java/com/jetbrains/youtrackdb/internal/core/gremlin/translator/strategy/StartStepRecogniser.java` (line 430; `buildClassEqExpression`, line 564)

**Issue.** When `isPolymorphic` resolves to `false`, the recogniser adds a `@class = 'V'` filter (line 430-431). `@class` evaluates to the record's **exact** schema-class name (`SQLRecordAttribute.evaluate`, line 156: returns `schemaClass.getName()`), so `@class = 'V'` matches only records whose direct class is `V`, excluding instances of any subclass of `V`. Native non-polymorphic `g.V()` does not narrow this way for a **bare** source: `YTDBGraphStep.elements`' no-id branch (line 131-158) builds a class query that is polymorphic at the storage level, and the non-poly path only applies `stream.filter(HasContainer.testAll(element, labelContainers))` (line 146-150). For a bare `g.V()` there is no `hasLabel`, so `labelContainers` is empty and `testAll(element, [])` is vacuously true â€” the non-poly flag has no filtering effect. Native bare non-poly `g.V()` therefore returns all vertices, including subclass instances; translated non-poly `g.V()` returns only exact-`V` instances.

**Failure scenario.** A schema with a user vertex class `Person extends V` and some `Person` rows, session flag `QUERY_GREMLIN_POLYMORPHIC_BY_DEFAULT = false`, no per-traversal override. Native `g.V()` returns every vertex including all `Person` rows (the empty-labelContainers filter passes everything). Translated `g.V()` plans `SELECT FROM V WHERE @class = 'V'`, which excludes every `Person` row and typically returns nothing (users almost always instantiate subclasses, not the abstract root). The multisets diverge â€” translated massively undercounts. The single-`@class`-narrowing is exactly what `walk_nonPolymorphic_emitsClassEqualsFilter` asserts is emitted, but that test only checks the filter string, not parity against the native pipeline, so it locks in the diverging behavior.

**Refutation considered.** I verified `@class`'s exactness by reading `SQLRecordAttribute.evaluate` (line 156, `@class` â†’ `getImmutableSchemaClass(session).getName()`) â€” it is the concrete class, not a polymorphic `instanceof` test. I verified native non-poly bare behavior by reading the whole `YTDBGraphStep.elements` no-id branch: the non-poly filter is by-`labelContainers` only, and the source's own comment (line 146-149) documents that true non-poly collection filtering is an unimplemented TODO ("we should propagate a flag to the query engine ... consider only non-polymorphic collection ids"). So today native non-poly bare `g.V()` returns the full polymorphic set; the translated `@class = 'V'` is stricter. I considered that this might be intended design â€” the plan-slim concern #1 explicitly asks to "check the polymorphic-vs-non-polymorphic `@class` narrowing," and one could argue the translation implements the *documented-intended* non-poly semantics while native has a latent gap. That framing does not rescue the multiset contract: the contract is translator-on == translator-off for recognized shapes, and native-off is the current observable behavior. Verdict: CONFIRMED as a divergence from the native pipeline. Rated should-fix (not blocker) because it only bites when the non-poly flag is off (non-default) and the schema has subclass instances; but that combination is the normal case for any real vertex schema under non-poly mode, so it needs resolution before the shape ships.

**Suggestion.** Match native's actual non-poly-bare semantics: for a bare `g.V()` (no id, no folded label) the non-poly flag currently has no effect natively, so do not emit `@class = 'V'` â€” leave the polymorphic class-`V` scan. Reserve `@class`-narrowing for the shapes where native non-poly actually filters (folded `hasLabel`, which arrives in Track 4). Alternatively, if the design deliberately intends non-poly `g.V()` to mean exact-`V`, that is a semantics change relative to native and must be called out in the design and gated behind a Step-5 parity test that documents the intentional divergence â€” not left as a silent multiset mismatch. Confirm the chosen direction against native by adding a translate-vs-native parity test with a `Person extends V` fixture under `polymorphic=false`.

### BC3 [should-fix] Reflection to set `SQLInCondition.operator` is unnecessary and its Javadoc rationale is false â€” a public setter exists

**File:** `core/src/main/java/com/jetbrains/youtrackdb/internal/core/gremlin/translator/strategy/StartStepRecogniser.java` (lines 605-647; `SQL_IN_OPERATOR_FIELD`, `resolveInOperatorField`, `setInOperator`)

**Issue.** `setInOperator` writes the `SQLInCondition.operator` field through a cached reflective `Field` (resolved at class-load in `resolveInOperatorField`, line 615). The Javadoc (line 627-639) justifies this: "The `SQLInCondition.operator` field is `protected` with no public setter ... This mirrors the same trick `MatchWhereBuilder.setInOperator` uses." Both claims are false. `SQLInCondition` exposes a **public** `setOperator(SQLBinaryCompareOperator operator)` (SQLInCondition.java line 431), and `SQLInOperator` **implements** `SQLBinaryCompareOperator` (SQLInOperator.java line 14), so `condition.setOperator(new SQLInOperator(-1))` compiles and is exactly what the claimed mirror does: `MatchWhereBuilder.in` (line 71-77) builds the identical `@... IN [...]` condition and sets the operator via `condition.setOperator(new SQLInOperator(-1))` (line 75) â€” no reflection anywhere in that class. There is no `MatchWhereBuilder.setInOperator` method at all.

**Failure scenario.** No runtime misbehavior on the happy path â€” the reflective write produces the same result as the public setter, so `g.V(id1, id2)` translates correctly today. The defect is fragility and a false invariant: if `SQLInCondition.operator` is ever renamed or the parser regenerated with a different field name, the failure mode is a class-load `ExceptionInInitializerError` from the static initializer (line 613, `resolveInOperatorField` throws `IllegalStateException`), taking down every load of `StartStepRecogniser` â€” which, via the walker's `PRODUCTION_RECOGNISERS`, is every load of the translator. The equivalent break with the public setter would be a compile error caught at build time. Reflection also opts the field write out of the IDE-refactoring and find-usages guarantees that a `setOperator` call site would participate in (the project explicitly routes symbol edits through IntelliJ refactors for exactly this reason). The self-referential Javadoc ("mirrors `MatchWhereBuilder.setInOperator`") points at a method that does not exist, so a future maintainer cannot verify the claim.

**Refutation considered.** I read `SQLInCondition` directly and confirmed `setOperator` is `public` (line 431, `public void setOperator(SQLBinaryCompareOperator operator)`) with a matching `getOperator` (line 428). I read `SQLInOperator` and confirmed `public final class SQLInOperator extends SimpleNode implements SQLBinaryCompareOperator` (line 14), so it is assignable to the setter's parameter. I read `MatchWhereBuilder.in` (line 71-77) and confirmed it uses the public setter, and grepped the class for any reflection (`getDeclaredField`/`reflect`) â€” none. I confirmed the reflection helpers (`setInOperator`, `SQL_IN_OPERATOR_FIELD`, `resolveInOperatorField`) are `private static` and referenced only inside `StartStepRecogniser` (grep: zero external callers), so replacing them is a purely local change with no ripple. I considered whether some subtlety of the generated parser blocks the public setter (e.g. a package-visibility quirk) â€” none: `SQLInCondition` is a public class and the setter is public, and the in-package `MatchWhereBuilder` (actually a different package, `...match.builder`) already calls it successfully. Verdict: CONFIRMED â€” the reflection is dead weight built on a false premise. Rated should-fix (not suggestion) because the false Javadoc actively misleads and the class-load failure mode is strictly worse than the compile-time one the public setter gives, even though there is no functional bug today.

**Suggestion.** Delete `SQL_IN_OPERATOR_FIELD`, `resolveInOperatorField`, and `setInOperator`, and in `buildRidInExpression` replace `setInOperator(condition)` with `condition.setOperator(new SQLInOperator(-1))` â€” matching `MatchWhereBuilder.in` exactly. Remove the false "protected field / no public setter / mirrors `MatchWhereBuilder.setInOperator`" Javadoc. This drops ~35 lines, removes the class-load failure surface, and restores compile-time safety and IDE refactor participation for the operator write.
