## Findings

### BC1 [should-fix] Throw-safety net catches `Throwable` and silently swallows `Error` / `AssertionError`

**File:** `core/src/main/java/com/jetbrains/youtrackdb/internal/core/gremlin/translator/strategy/GremlinToMatchStrategy.java` (line 216)

**Issue.** `apply` wraps its whole body in `try { applyOrDecline(traversal); } catch (Throwable t) { declineOnThrow(traversal, t); }` (lines 214-221), and `declineOnThrow` (lines 253-256) is an empty no-op — it neither logs nor rethrows anything. So every `Error` — `OutOfMemoryError`, `StackOverflowError`, and, decisively, `AssertionError` — is caught and silently discarded, downgrading to "decline to native." R3's intent is "a recogniser throw must never break native Gremlin," which is fully served by catching `Exception`; catching `Throwable` additionally swallows failures that are not translator bugs and that the JVM is entitled to treat as fatal or as invariant violations.

The `AssertionError` arm is the load-bearing one for this feature. YTDB runs its unit/integration suites and CI with assertions enabled (`-ea`), and the code the net is meant to protect uses Java `assert` on the hot path it wraps — e.g. `DatabaseSessionEmbedded.getConfiguration()` opens with `assert assertIfNotActive();` (DatabaseSessionEmbedded.java:2598), reached from `resolveSessionIfEnabled` inside `applyOrDecline`. From Step 4 the real `GremlinStepWalker` + recognisers run under this same net; any `assert` an internal invariant guards (a walker that mis-counts consumed steps, a recogniser that violates the no-mutation-on-decline contract, a malformed `MatchPlanInputs`) will throw `AssertionError`, be swallowed here, and vanish. The very test that pins this behavior — `apply_translatorThrows_declinesWithoutPropagating` (test lines 706-720) — deliberately throws `AssertionError` and asserts it is swallowed, codifying the mask into the contract.

This also diverges from an existing codebase convention: `Streams.java:165-168` catches `Throwable` and explicitly rethrows `Error` (`if (throwable instanceof Error error) { throw error; }`) before wrapping the rest. The strategy's blanket swallow is the opposite discipline.

**Failure scenario.** With assertions enabled (every test and CI run), a Step-4 walker or recogniser trips a genuine `assert` on some traversal shape — a real correctness bug the assert exists to catch. Instead of the query failing loudly with an `AssertionError` that points a developer at the broken invariant, `apply` swallows it, the traversal declines to native execution, and the shape "just works via native Gremlin." The translator bug is invisible in the green suite and only surfaces later as a silent non-translation (a missed optimization) or, worse, masks a real state-corruption assert. Separately, on `OutOfMemoryError` the net swallows the error and hands the traversal to the native pipeline, which re-attempts the same allocation in a JVM already at its memory ceiling, with no log record that the first attempt OOM'd.

**Why should-fix, not blocker.** The current facade declines every shape, so no recogniser code runs under the net yet and no live `AssertionError` from a walker can be produced in this step. R3's stated invariant (never break native Gremlin) is upheld. The defect is that the net is wider than R3 requires and its landing point is silent; it becomes actively harmful the moment Step 4 wires the walker under it, which is the very next step and reuses this net unchanged.

**Refutation considered.** I checked whether R3 requires catching `Throwable` specifically — the track Decision Log (`review-sequencing (R3)`) and the class Javadoc say only "a recogniser throw must never break native Gremlin," which `catch (Exception)` satisfies; nothing requires swallowing `Error`. I checked whether assertions are realistically on: the project builds/tests under `-ea` (module `argLine` flags, and `DatabaseSessionEmbedded` is riddled with `assert assertIfNotActive()` guards that only fire under `-ea`), and `getConfiguration()` on the wrapped path is one such assert, so `AssertionError` is reachable under the net today. I checked for a codebase norm and found the opposite convention at `Streams.java:165`. The one counter-argument — that declining is always "at least as safe as native" — holds for translator *logic* bugs (Exceptions) but not for `Error`/invariant-assert failures, where silent masking is the harm. Verdict: CONFIRMED as a should-fix.

**Suggestion.** Narrow the catch to `catch (Exception e)` (or catch `RuntimeException` if only unchecked translator failures are expected), so `Error` and `AssertionError` propagate. If a broad `Throwable` catch is genuinely wanted for defense in depth, mirror `Streams.java`: rethrow `Error` (`if (t instanceof Error err) throw err;`) before declining, and give `declineOnThrow` at least a debug/warn-level log so a swallowed translator failure leaves a trace instead of vanishing. Update `apply_translatorThrows_declinesWithoutPropagating` to throw a `RuntimeException` (the realistic walker-bug case) rather than an `AssertionError`, so the test stops codifying the swallow-invariant-errors behavior.

### BC2 [suggestion] `session.getConfiguration()` is `@Nullable` and dereferenced without a guard

**File:** `core/src/main/java/com/jetbrains/youtrackdb/internal/core/gremlin/translator/strategy/GremlinToMatchStrategy.java` (line 283)

**Issue.** `resolveSessionIfEnabled` reads the kill-switch via `session.getConfiguration().getValueAsBoolean(GlobalConfiguration.QUERY_GREMLIN_TO_MATCH_TRANSLATOR_ENABLED)` (lines 281-284). `DatabaseSessionEmbedded.getConfiguration()` is annotated `@Nullable` (DatabaseSessionEmbedded.java:2597) and returns `storage.getContextConfiguration()` — a null return dereferences into an NPE at the `.getValueAsBoolean(...)` call. The value-level null is handled (`ContextConfiguration.getValueAsBoolean` returns `false` for a null value, ContextConfiguration.java:151-153, so a genuinely-unset flag declines cleanly), but a null *ContextConfiguration object* is not guarded.

**Failure scenario.** A `YTDBGraph` whose backing storage returns a null `ContextConfiguration` (the `@Nullable` contract permits it) reaches `resolveSessionIfEnabled`; the chained call NPEs. Today the NPE is caught by the BC1 throw-safety net and degrades to a decline, so the observable effect is "translation silently disabled for that graph" rather than a crash — but that only holds while the net swallows `RuntimeException`, and it relies on the net rather than on the resolver being correct.

**Refutation considered.** This exactly mirrors an already-shipped precedent: `YTDBStrategyUtil.isPolymorphic` (YTDBStrategyUtil.java:43-45) does the identical unguarded chain `tx.getDatabaseSession().getConfiguration().getValueAsBoolean(...)` on every traversal via the existing `YTDBGraphStepStrategy`. If a null `ContextConfiguration` were reachable in normal operation, that pre-existing strategy would already NPE on every Gremlin query, which it does not — so in practice `getConfiguration()` is non-null for an open YTDB session. That is why this is a suggestion, not a should-fix: it is a latent nullability gap consistent with existing code, not a new live defect. The `@Nullable` annotation is nonetheless load-bearing (asserted-active accessor), and the new strategy would be strictly more robust with an explicit null-check that declines. This is PLAUSIBLE — I could not construct an in-tree path that actually returns null from an active session, only the annotation permitting it.

**Suggestion.** Capture the config once and null-check it before reading the flag: `var cfg = session.getConfiguration(); if (cfg == null) { return null; }` then read `cfg.getValueAsBoolean(...)`. This makes the decline explicit rather than routed through the exception net, and does not depend on BC1's resolution.

## Evidence base

#### C1 — `catch (Throwable)` swallows `Error`/`AssertionError` — verdict CONFIRMED

Survived refutation: R3 needs only `catch (Exception)`; assertions run under `-ea` in test/CI and are reachable under the net (`DatabaseSessionEmbedded.getConfiguration` assert on the wrapped path); codebase convention at `Streams.java:165` rethrows `Error`; the pinning test codifies the swallow. Live harm activates when Step 4 wires the walker under the unchanged net. Reference basis: source reads of the diff + `Streams.java` + `DatabaseSessionEmbedded.java`; no symbol-usage search required.

#### C2 — `getConfiguration()` nullable-deref — verdict PLAUSIBLE

Survived as a suggestion: the `@Nullable` annotation permits a null `ContextConfiguration`, and the chain dereferences it without a guard; if null, the call NPEs and is only saved by the BC1 net. Not elevated because the identical unguarded chain already ships in `YTDBStrategyUtil.isPolymorphic` (run on every traversal by `YTDBGraphStepStrategy`) and does not NPE in practice, so an active YTDB session's config is non-null in the field. I could not exhibit an in-tree path returning null from an active session — only the annotation allowing it — so the deref-NPE arm is PLAUSIBLE, not CONFIRMED.

#### Refuted claims (checked, not reported)

**Test-seam production-safety (orchestrator concern #2) — REFUTED.** The two package-private constructors (`Function` translator seam, `BiFunction` planBuilder seam) are reachable only from the `translator.strategy` package; both `GremlinToMatchStrategy` and `GremlinToMatchTranslator` are package-scoped there, and the production `INSTANCE` (lines 168-170) wires `GremlinToMatchTranslator::translate` (the declining facade) and `GremlinToMatchStrategy::buildPlan` (the real planner). External callers reach the strategy only via `instance()`. No production behavior is weakened or mutable from outside the package. Not a finding.

**Gating-cascade ordering / short-circuit (concern #3) — REFUTED.** `applyOrDecline` (lines 229-245) runs the gates in the required order: session-resolution + kill-switch first (`resolveSessionIfEnabled`, cheapest per-session read, returns before any walk), then the whole-list idempotency scan (`containsBoundaryStep` iterates *all* steps, lines 294-301 — not start-step-only, matching D7), then the plain-`GraphStep` vertex-start gate (`hasVertexGraphStart`, lines 310-313), and only then `translator.apply`. The translator (potentially the expensive walk) is never consulted until all three cheap gates pass, verified by the `consulted[0]`/`neverBuilt[0]` counters in the kill-switch, edge-start, and detached-traversal tests. Not a finding.

**Decline-leaves-list-verbatim (concern #4) — REFUTED.** Every decline path in `applyOrDecline` is a bare `return` before any mutation; the only step-list mutation is `replaceAllStepsWithBoundary` (lines 361-375: `TraversalHelper.removeAllSteps` + `addStep`), reached exclusively from `applyTranslation`, which runs only after a non-empty translation *and* a successful plan build. Because the plan is built (line 334) before the boundary is constructed and the list mutated, a planner throw is caught by the net with the list still intact (pinned by `apply_planBuilderThrows_declinesWithStepListIntact`). Not a finding.

**Empty step list / `getStartStep()` NPE (concern #5) — REFUTED.** `DefaultTraversal.getStartStep()` returns `EmptyStep.instance()` (not null, no exception) when the step list is empty (decompiled from gremlin-core), so `hasVertexGraphStart`'s `instanceof GraphStep` check simply fails and the strategy declines. `graph.tx()` is guarded by `instanceof YTDBGraph` then `instanceof YTDBTransaction` (lines 273-276), so a detached `__.V()` (TinkerPop `EmptyGraph`, whose `tx()` throws `UnsupportedOperationException`) declines at the first gate without touching `tx()` — pinned by `apply_anonymousDetachedTraversal_declines`. Not a finding.

**`readWrite()` transaction-open side effect during compilation — REFUTED as a regression.** `resolveSessionIfEnabled` calls `tx.readWrite()` (line 279), which under the default `READ_WRITE_BEHAVIOR.AUTO` opens a DB transaction as a side effect if none is open (decompiled `AUTO.accept` → `open()` → `doOpen()` → `activeSession.begin()`). This runs during `applyStrategies()` on the compilation path. It is not a new hazard: the pre-existing `YTDBStrategyUtil.isPolymorphic` (used by `YTDBGraphStepStrategy` on every traversal) performs the identical `graph.tx(); tx.readWrite()` sequence, so opening the transaction at strategy time is the established YTDB behavior, not something this strategy introduces. Under `MANUAL` read-write mode with a closed transaction, `readWrite()` throws `IllegalStateException` (decompiled `MANUAL.accept`), which the BC1 net degrades to a decline — acceptable, and again matching the pre-existing pattern's exposure. Not a finding; noted for context.

**Resource leak on plan-build-then-splice — REFUTED for this step.** `buildPlan` → `MatchExecutionPlanner.createExecutionPlan(ctx, false, false)` can execute prefetch sub-plans at build time, but the returned `InternalExecutionPlan`'s own `ExecutionStream` opens lazily in the boundary step (Step-2 design), so a plan built and installed does not leave a dangling stream the boundary fails to close. The only leak window — plan built, then a throw before the boundary (which owns `plan.close()`) is installed — requires the `YTDBMatchPlanStep` constructor or `TraversalHelper.removeAllSteps` to throw between lines 334 and 373; the constructor only re-`requireNonNull`s already-non-null fields and the TinkerPop calls do not throw for a well-formed boundary. Moreover this whole splice path is unreachable through the production declining facade (per the step's scope note), so any residual leak is dead-code-only in this step. Not a finding; the boundary-step plan lifecycle is Step 2's settled concern.

**Singleton publication / static-init ordering — REFUTED.** `INSTANCE` is a `static final` initialized in the class initializer with method references to `GremlinToMatchTranslator::translate` and `GremlinToMatchStrategy::buildPlan`; both target classes are already loaded when the initializer runs (same package, and `buildPlan` is a static method of the class being initialized, invoked only later at call time, not during init). The strategy is stateless apart from the two final function fields, safely published via the `static final` + class-init happens-before. No unsafe-publication or init-cycle hazard. Not a finding.
