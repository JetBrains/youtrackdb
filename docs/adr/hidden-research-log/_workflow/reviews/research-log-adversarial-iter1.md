## Findings

### A1 [blocker]
**Certificate**: CD1 (Challenge: Decision D1 — §1.7(k) criterion-(2) classification of `create-plan/SKILL.md`)
**Target**: Decision D1 — Use the §1.7(k) prose-rule opt-out, not §1.7 staging
**Challenge**: D1's whole validity hinges on `create-plan/SKILL.md` qualifying under §1.7(k) criterion (2) as judgment-layer. The log resolves the classification by reading "the *specific edit*" (Phase-0 narration prose, "not a mechanical gate sequence" — research-log.md:88-91, :172-175). But §1.7(k) states the test in the opposite terms. `conventions.md:1231` heads the criteria "consumer class, **not author intent**"; `conventions.md:1243-1245` repeats "Both criteria are about what consumes the edited file, **not what the planner meant to do**. A plan that satisfies (1) but edits an execution-procedure file fails (2) and must stage that file." Criterion (2) at `conventions.md:1237-1241` names the disqualifier as "Files **a running phase reads as executable procedure** (the implementer rulebook's gate sequence, the step-implementation orchestration loop, the migrate replay)." `create-plan/SKILL.md` is exactly that: `user-invocable: true` (SKILL.md:5), "Read and follow the workflow for Phase 0 (Research) and Phase 1 (Planning)" (SKILL.md:21), with numbered Steps the planner executes mechanically — Step 1.5 drift gate (SKILL.md:48-60), stamp computation, resume-state reads, Step 4/4b plan-and-track authoring. By the file-level consumer-class test the log itself names in Open Question 2, this is an execution-procedure file, and the rule's own words say it "fails (2) and must stage." D1 picked the edit-level reading the rule explicitly rejects.
**Evidence**: `conventions.md:1231` ("consumer class, not author intent"); `conventions.md:1237-1241` (criterion 2 + execution-procedure exemplars); `conventions.md:1243-1245` ("must stage that file"); `create-plan/SKILL.md:5,21,48-60` (the file is the executable /create-plan orchestration procedure). The log's own Open Question 2 (research-log.md:166-175) states the file-level test yields "execution procedure → must stage → killing the opt-out."
**Proposed fix**: Re-decide D1 in research before any artifact derives. Either (a) accept the file-level consumer-class verdict and take §1.7(b) staging for the plan (the issue's stated default), or (b) if the opt-out is still wanted, restructure the change so the only edited execution-procedure file is staged while the judgment-layer file opts out — but see A2: §1.7(k) as written gives no clean per-plan path for that, so this likely collapses back to (a). Self-application of the opacity rule is then lost for this branch; if that loss is the decisive reason to keep the opt-out, the rule's "consumer class, not author intent" wording must itself be amended on this branch (an in-scope workflow-prose edit), and that amendment, not a per-branch reinterpretation, is what should carry the decision.

### A2 [should-fix]
**Certificate**: CD1 (same challenge, mixed-staging corollary)
**Target**: Decision D1 — opt-out marker carried instead of the §1.7(b) marker
**Challenge**: Even granting that one edited file fails criterion (2), §1.7(k) offers no clean state for D1's situation. `conventions.md:1244-1245` says the failing file "must stage that file" (implying per-file staging), but `conventions.md:1265-1269` says "The two markers are mutually exclusive on **one plan**... never both: the opt-out's whole purpose is to default the staging consumers to live, which is exactly the workflow-modifying marker's absence." A plan cannot simultaneously carry the opt-out marker (defaults every staging consumer to live, no staged subtree) and stage one file (needs the write-routing and staged subtree the opt-out disables — `conventions.md:1211-1215`, `:1271-1274`). So a plan that contains any execution-procedure file has no valid opt-out configuration; the only consistent state is the whole plan stages under §1.7(b). This makes D1's choice **unavailable**, not merely a weaker-but-legal alternative the log frames it as. D1 never surfaces this mutual-exclusion-vs-must-stage tension; it treats the opt-out as freely selectable once it labels the edit judgment-layer.
**Evidence**: `conventions.md:1244-1245` (per-file "must stage"); `conventions.md:1265-1269` (markers mutually exclusive per plan); `conventions.md:1211-1215` and `:1271-1274` (opt-out disables write-routing and the staged subtree wholesale). No clause reconciles per-file staging with the per-plan marker exclusivity.
**Proposed fix**: Add an explicit step to D1's reasoning that resolves the mutual-exclusion-vs-must-stage tension before choosing a marker. If A1 is accepted, this resolves to "stage the whole plan." If the opt-out is retained, the resolution must name which clause wins and why — and that is a workflow-prose rule clarification (an edit §1.7(k) itself should carry), not a silent per-branch reading.
