- Cost: latent false-positive `should-fix` finding — a footer cross-reference like `D1 (supersedes D2)` would flag `D2` as a dangling bare citation even though it is plainly inside another code's rationale, not cited in its own right.
- Issue: `_iter_footer_dcode_citations` runs `_FOOTER_DCODE_RE.finditer` over the *whole* joined `D-records:` entry, so it yields a citation for every `D<n>` token, including codes that appear inside another code's parenthetical rationale. The `has_rationale` heuristic then decides bare-ness from the character immediately after the nested code: a code followed by `)` or `,` is judged bare. So in `- D-records: D1 (see D2)`, `D2`'s rest is `)` → `rest[0] in ",)"` → `has_rationale=False`, and `D2` is emitted as a bare citation. If `D2` has no full record elsewhere in design.md (`num not in full_intros`), `check_decision_cited_without_rationale` fires a `should-fix` on a code the author never independently cited — a false positive. The comma form (`D1 (see D2, D3)`) flags both `D2` and `D3`. I confirmed this empirically by importing the module and exercising the function directly: `D1 (see D2)` yields `[(10, 1, True), (10, 2, False)]`, and `D1 (see D2, D3)` yields `[(10, 1, True), (10, 2, False), (10, 3, False)]`. The reason the new test still passes — and the reason this is Minor, not should-fix — is that the frozen `design.md` happens to avoid the pattern: every footer code carries its own top-level parenthetical (`D2 (two-gate model …), D3 (timing …)`) and no code is nested inside another's parenthetical (`grep -nE "D-records:.*\([^)]*\bD[0-9]+"` over the seed returns nothing). So the seed is correct by content, not by the check excluding the case.
