**Same-transaction schema-validation gap (design-level; escalated from Track 4 completion review).** A class, property type, or constraint rule created or changed inside an open transaction is not enforced on that same transaction's own entities.

Mechanism: `EntityImpl.validate()` (`EntityImpl.java:3923-3952`) resolves the entity's class through `getImmutableSchemaClass(session)` → `getImmutableSchemaSnapshot()` → `SchemaProxy.makeSnapshot()`, which is committed-only by explicit design (`SchemaProxy.java:76-80`, "the immutable snapshot is taken from the committed instance, not the tx-local copy"). During a schema tx the shared `SchemaShared` is not mutated until commit promotes it (`design.md:241-242`), so the snapshot has no entry for a class/rule/type created in the open tx. `validate()` guards all constraint checks behind `if (immutableSchemaClass != null)`, so for a new-in-tx class the class resolves to `null` and strict-mode, mandatory, notnull, type, min/max, and regex checks are all silently skipped; serialization likewise falls back to schemaless. The new schema becomes visible only after commit promotes the tx-local schema (`fromStream` + `forceSnapshot`, `AbstractStorage.java:2690-2691`), i.e. for subsequent transactions.

The transaction has two schema read paths (`design.md:269-277`): structural reads through `SchemaProxy` are tx-aware (read-your-writes), but the validation/serialization path — `EntityImpl` on the immutable snapshot — is not. So read-your-writes is honored for structure and broken for the schema contract. This is consistent with Step 6's explicit decision to leave `EntityImpl.getSchemaClass()` on the committed snapshot ("by-design-exempt: cold callers + tx-local semantics") — but its consequence for validation is not written down as a user-facing semantic, and no test pins it.

Not a Track 4 regression: Track 4 built the commit half and sits on top of the snapshot tier; the snapshot is committed-only deliberately (lock-free fast reads). The fix is a design decision with a real performance/architecture trade-off: either make the immutable snapshot tx-aware during a schema tx, or route `EntityImpl` schema resolution through the tx-local proxy, or accept the limitation and document it as an explicit semantic. That is what inline replanning decides.
