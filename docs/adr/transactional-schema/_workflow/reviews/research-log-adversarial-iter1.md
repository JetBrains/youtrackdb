## Findings

### A1 [should-fix]
**Certificate**: C1 (assumption test — the transactionality-enabler de-guarding has no dedicated invariant)
**Target**: Invariant list completeness — the de-guarding of the six self-commit / throw guards (F3 `saveInternal`, F4 `dropClass`/`dropClassInternal`, F21 `createIndex`, F26 `dropIndex`, F46 `addCollectionToIndex` / `removeCollectionFromIndex`)

**Challenge**: The single most foundational transformation of the whole feature is removing the constructs that *today actively forbid* a schema mutation inside a user transaction. PSI confirms all six are present in current code (`SchemaShared#saveInternal` throw+self-commit, `SchemaEmbedded#dropClass`/`dropClassInternal` throw, `IndexManagerEmbedded#createIndex` throw, `#dropIndex` throw+self-commit, `#addCollectionToIndex` / `#removeCollectionFromIndex` self-commit). D1's inversion is impossible unless every one is de-guarded so the mutation rides the user tx. Yet no entry in `## Invariants and Test Requirements` names this contract, and no test constructs the dangerous failure: the **self-commit** guards (`addCollectionToIndex` / `removeCollectionFromIndex`, reached transitively from `createClass` / `addSuperClass` via the polymorphic ripple) do not throw — they silently commit a subclass-collection-membership change in a nested micro-tx that escapes the user tx, leaking to other sessions and breaking rollback-freedom. F46 tagged exactly this an isolation BLOCKER.

The throw-guards (F3/F4/F21) are implicitly covered: left in place they throw on any DDL, so I-A5's `tx1 creates a class` test (and every other DDL test) fails loud. But the self-commit guards are the silent ones. The closest coverage is I-P2's clause "collection-membership changes apply commit-only and never mutate a shared `Index` object mid-tx," but I-P2's *Test* sentence does not build the leak scenario (a concurrent session reading an uncommitted membership change, or a rollback leaving the shared `Index` mutated), and the mechanism line buries "in-place collection-membership" as one of four overlay categories without flagging that the self-commit must be removed. An implementer working the invariant list could route the membership change through the overlay yet leave `executeInTxInternal` in `addCollectionToIndex`, and no listed test would fail.

**Evidence**: PSI find-methods over current code returned `throwGuard`/`selfCommit` true for all six (eid `adv-research-log-guards`). F46 §(a) names the fifth/sixth self-commit guard and grades the self-commit + isolation facets BLOCKER; F46's collectionCounter pass shows the membership ripple feeds a `null` collection name under a provisional id, so it *cannot* run mid-tx at all — commit-only deferral is a correctness requirement, not just isolation. F3/F4/F21/F26 are each `tx.isActive()` throw-guards. None of I-A1 / I-A5 / I-P2 states "the N self-commit/throw guards are removed" as an invariant, nor tests the membership self-commit leak.

**Proposed fix**: Add one invariant — call it the transactional-enablement contract — stating that every schema/index mutation entry point that today throws on an active tx or opens its own micro-tx (enumerate the F3/F4/F21/F26/F46 sites by method, not by F-label) is reworked to ride the user tx, deferring its structural effect to commit. Give it a *Test* that constructs the silent failure: inside one tx, trigger a polymorphic membership ripple (`addSuperClass` or alter-add-collection on a class with an indexed subclass), assert a concurrent session does not see the membership change before commit, and assert rollback leaves the shared `Index`'s `collectionsToIndex` untouched. This is the testable form of F46's isolation BLOCKER and pins the de-guarding an implementer would otherwise miss.

### A2 [should-fix]
**Certificate**: C2 (violation scenario — a committed class DROP whose collection is never removed passes every listed invariant test)
**Target**: Invariant I-A1 (and the D9/F43 drop-detection contract)

**Challenge**: I-A1's test exercises three paths — rollback (no orphan), crash-before-commit (clean state), and "a committed drop survives a crash (redone from the WAL)." None of them is a *positive* test that commits a class DROP and asserts the collection and engine files are actually removed and the data is gone. The drop-detection mechanism is load-bearing and non-obvious: per D9/F43 a dropped class's record is deleted, so it carries no per-property change signal, and the drop must be detected from the **collection-id set difference over the committed vs tx-local in-memory structures** — explicitly NOT from D6's changed-record set. The natural misreading of D6 ("commit reads the tx's changed records and derives the create/drop set") builds the diff from the changed-record set, in which a dropped class is invisible, so the commit drops nothing: the collection and its data survive a committed drop-class. "Redone from the WAL" presupposes the drop was applied at commit; it does not test that the drop was detected and applied in the first place.

**Evidence**: D9 "Drop = real collection ids in the old schema absent from the new... computed over the committed in-memory `SchemaShared` (old) vs the tx-local `SchemaShared` (new) collection-id sets." F43 / F37: "a dropped class's record is deleted, so it has no per-property change to inspect," and "the *structural* collection/index create/drop set is D9's set difference over the committed vs tx-local in-memory structures, not the record-level property diff." Grep of the invariant section (lines 5946-6405) for drop-application test scenarios returns only I-A1's rollback/crash cases and I-U3's same-name drop-and-recreate file-collision case — neither asserts a committed drop removes the collection via the set-difference path.

**Proposed fix**: Add a positive *Test* case to I-A1 (or a short dedicated drop invariant): create a class with data and an index, commit; in a second tx drop the class, commit; assert the collection files, the engine files, and the registry entries are gone and the data is unreadable; restart and confirm the same. State in the invariant that the structural create/drop set is the in-memory collection-id set difference (D9), not the changed-record set (D6) — the property the test defends.

### A3 [suggestion]
**Certificate**: C3 (assumption test — F51's lazy-invalidation performance invariant is unpinned)
**Target**: Invariant I-P2 (and the Delegated section completeness)

**Challenge**: F51 is a MAJOR finding tagged "implementation invariant": F35's "force a tx-local snapshot rebuild on every mid-tx `createIndex`/`dropIndex`" must be *lazy invalidation* (null the snapshot, rebuild on next read, O(1)), never *eager reconstruction* (O(current schema size) per call → O(N^2) for the F48 4,000-index batch — the exact write-amplification class YTDB-382 exists to kill, reappearing on the read side). I-P2 carries the force-rebuild trigger but states it as a correctness requirement ("without the rebuild, same-tx inserts into the new index are silently untracked") and its *Test* (create index mid-tx, insert rows, commit, assert rows present) passes identically under eager reconstruction. So the O(N^2) cliff has no invariant test, and F51 appears in neither the invariant Provenance lines nor the Delegated section.

This is graded a suggestion, not a should-fix, because `Performance hot path` is not among the matched categories and the consolidation may have judged a pure-performance implementation detail as legitimately out of the design-altitude invariant set. But the consolidation's own altitude rule ("an invariant is a property a test can check") and F51's MAJOR grade argue it should at least appear in `## Delegated to implementation` tied to I-P2, the way F50's sibling O(N^2) caveat already does.

**Evidence**: F51 "Read as eager reconstruction, that is O(current schema size) per createIndex → O(N^2) for a 4,000-index batch. It must be lazy invalidation." Grep of the invariant+delegated sections (line >=5920) for `F51` / `lazy invalidat` / `eager reconstruct` / `O(N` returns only F50's monolithic-link-set entry, not F51. The uncited-F-finding diff confirms F51 is cited nowhere in the two new sections.

**Proposed fix**: Add a one-line entry to `## Delegated to implementation` under I-P2: "Snapshot-rebuild invalidation must be lazy (null-and-rebuild-on-read, O(1)), never eager reconstruction (O(N^2) on a batch); serves I-P2; F35, F51; folds the read-side amplification next to F50's write-side entry." No new test is required — note it as a performance property the implementer owns, mirroring F50's treatment.

## Evidence base

#### C1 — Assumption test: the six self-commit/throw guards' de-guarding is pinned by an invariant with a catching test
- **Claim**: The invariant list's completeness covers every design-load-bearing failure mode, so the de-guarding of the F3/F4/F21/F26/F46 transactionality enablers (the constructs that forbid schema mutation inside a user tx) is captured by an invariant whose test catches a missed de-guard.
- **Stress scenario**: An implementer routes the F46 collection-membership change through the D15 overlay (satisfying I-P2's "apply commit-only" clause for the *create/drop/rename* categories) but leaves `IndexManagerEmbedded#addCollectionToIndex`'s `executeInTxInternal` self-commit in place. On `addSuperClass` of an indexed subclass, the membership change self-commits mid-tx, leaks to concurrent sessions, and survives a rollback.
- **Code evidence**: PSI (eid `adv-research-log-guards`) over current code — `SchemaShared#saveInternal@817 throwGuard=true selfCommit=true`; `SchemaEmbedded#dropClass@369`, `#dropClassInternal@414 throwGuard=true`; `IndexManagerEmbedded#createIndex@270 throwGuard=true`; `#dropIndex@456 throwGuard=true selfCommit=true`; `#addCollectionToIndex@99 selfCommit=true`; `#removeCollectionFromIndex@131 selfCommit=true`. F46 §(a)/(b) grade the self-commit + isolation facets BLOCKER; the collectionCounter pass shows commit-only deferral is also a correctness requirement (provisional id yields a `null` collection name mid-tx). I-P2's *Test* sentence asserts "collection-membership changes apply commit-only and never mutate a shared `Index` object mid-tx" but does not construct the concurrent-leak or rollback-leak scenario; I-A5's test catches only the throwing guards (any DDL throws if a throw-guard remains).
- **Verdict**: BREAKS — the throw-guards are caught loudly but incidentally; the self-commit guards (the silent, BLOCKER-graded ones) have no test that builds the leak. The contract is implied, never stated, and the catching test is missing.

#### C2 — Violation scenario: a committed class DROP whose collection is never removed
- **Invariant claim**: I-A1 — "A rolled-back or crashed-before-commit schema transaction leaves the storage byte-for-byte unchanged: no orphaned collection, no orphaned engine, no lost data on a drop... A committed drop survives a crash (redone from the WAL)."
- **Violation construction**:
  1. Start state: a committed class `Foo` with one collection holding data and one index engine.
  2. Action sequence: implementer builds the commit-time structural diff from D6's tx changed-record set (the natural reading of "commit reads the changed records and derives the create/drop set"). Tx2 issues `DROP CLASS Foo`; the schema-record link-set loses `Foo`'s per-class record, which is deleted.
  3. Intermediate state: the changed-record set contains the deleted per-class record and the modified root, but *no per-property signal that a collection should be dropped* (F37/F43: a dropped class has no per-property change to inspect).
  4. Violation point: the commit's create/drop reconciliation, fed by the changed-record set instead of D9's in-memory collection-id set difference, finds no collection to drop. The collection and its engine survive.
  5. Observable consequence: a committed `DROP CLASS` leaves the collection files, the engine files, and the data on disk; the schema says the class is gone, storage disagrees — silent storage leak, recoverable orphan after restart.
- **Feasibility**: CONSTRUCTIBLE — D6's framing invites the changed-record-set reading; the safeguard is D9's set difference over in-memory structures, which is delegated mechanism. No listed invariant test commits a drop and asserts removal (I-A1 tests rollback/crash; I-U3 tests same-name-recreate file collision). The test gap is real even though the design's *intent* (D9) is correct.

#### C3 — Assumption test: F51's lazy-invalidation performance property is pinned
- **Claim**: Every load-bearing implementation invariant from the F-trail is either an invariant entry or a Delegated entry; F51 (lazy invalidation, not eager reconstruction) is one of them.
- **Stress scenario**: the F48 batch — 400 classes, 4,000 indexes, one commit. An implementer reads I-P2's "the tx-local snapshot is force-rebuilt on every mid-tx `createIndex`/`dropIndex`" and implements eager reconstruction (rebuild the snapshot in full per call). For a composite-index or data-interleaved batch this is O(N^2), reintroducing the read-side analog of the write amplification YTDB-382 removed.
- **Code evidence**: F51 — "It must be lazy invalidation: null the tx-local snapshot (today's `forceSnapshot` shape, `SchemaShared:218`, O(1)) and rebuild only on the next snapshot read." I-P2's *Test* (create index, insert rows, commit, assert present) is a correctness test that passes under eager reconstruction; grep of lines >=5920 for `F51`/`lazy invalidat`/`eager reconstruct`/`O(N` returns only F50's entry, not F51.
- **Verdict**: WEAK — the property is genuinely load-bearing and unpinned, but `Performance hot path` is not a matched category and a Delegated note (not a full invariant) is the proportionate fix; hence a suggestion rather than a should-fix.

#### C4 — Assumption test: I-freezer-1's site-classification is testable (control)
- **Claim**: I-freezer-1's tests catch a mis-classified freeze site (a transient quiesce tagged operator-kind, or an operator freeze tagged transient).
- **Stress scenario**: implementer mis-tags `freeze(db,false)` (operator filesystem-snapshot, `AbstractStorage` freeze park-mode) as transient, or tags `doSynch` as operator.
- **Code evidence**: PSI freeze-registration census (eid `adv-freeze-sites3`/`adv-freeze-sites4`) — distinct `freezeWriteOperations` callers are `doSynch@3749`, `freeze@3901` (throw), `freeze@3905` (park), `DiskStorage#copyWALToBackup@356`, `DiskStorage#storeBackupDataToStream@1248`: the F87 five-site inventory is complete. I-freezer-1's test cases drive the *real* freeze entry points ("operator park-mode freeze pre-engaged → throw by type"; "in-flight `doSynch`/backup segment cut → brief park, commit succeeds"), so an operator-as-transient mis-tag is caught by the park-then-outage case and a transient-as-operator mis-tag by the "commit succeeds" case.
- **Verdict**: MATCHES — site classification is testable through the real freeze callers; the five-site inventory is complete; no finding.

#### C5 — Assumption test: I-A4's concurrent-registrar test scenario is real (control)
- **Claim**: I-A4's test ("a non-commit engine registrar — `rebuild`, `loadExternalIndexEngine`, `recreateIndexes` — running under `stateLock.write` alone does not collide with a schema commit's id allocation") rests on a constructible scenario.
- **Code evidence**: PSI (eid `adv-registrars`) — `AbstractStorage#addIndexEngine` callers are `IndexAbstract#create@196` and `IndexAbstract#rebuild@305`; `rebuild` is reachable from the user rebuild API and the `recreateIndexes` crash-recovery thread, running under `stateLock.write` without the D7 schema mutex. The F88 race (a concurrent registrar drawing an engine id from the shared registry while a schema commit allocates) is real, so I-A4's "seed the commit-local allocator inside the write-lock window" invariant and its concurrent-registrar test are sound.
- **Verdict**: MATCHES — the F88 pin and I-A4's test are well-grounded; no finding.
