## Findings

### A4 [should-fix]
**Certificate**: C6 (assumption test — the post-commit positive coverage of a polymorphic membership change is pinned by a catching test)
**Target**: Invariant-list completeness — F46 facet (c), the membership-only changed-index category, split across I-A7 (de-guarding) and I-P2 (overlay), tested by neither

**Challenge**: The A1 resolution is sound for the two facets it pins, but it routes the reader's attention onto isolation and rollback and leaves F46's third facet with no catching test. F46 names three distinct failure modes for the polymorphic collection-membership ripple: (a) the self-commit micro-transaction, (b) in-place mutation of a committed shared `Index`, and (c) a membership-only change to an existing index that is none of D15's create / drop / rename categories, so if the changed-index entity is not tracked the commit persists no membership delta and the committed index silently fails to cover the new subclass collection — a polymorphic query under-returns after commit. I-A7's new *Test* asserts (a) through "a concurrent session does not observe the membership change before commit" and (b) through "a rollback leaves the shared `Index`'s `collectionsToIndex` untouched." Both are negative / safety properties. I-P2's clause "collection-membership changes apply commit-only and never mutate a shared `Index` object mid-tx" is likewise a negative no-mid-tx-mutation property. Neither states the *positive* post-commit property: after committing an `addSuperClass` (or alter-add-collection) on a class whose parent carries an index, the committed parent index covers the new subclass collection and a polymorphic query through that index returns the subclass rows. An implementer can de-guard the self-commit (pass I-A7's isolation+rollback test), route the change through the overlay without touching a shared object mid-tx (pass I-P2's negative clause), yet forget to add the membership-only change to the changed-index / changed-class set (F46's resolution: "add collection-membership change to the changed-index set as an in-place-mutation category alongside rename"). Commit then persists nothing, and a polymorphic query silently under-returns. No listed test fails. This is exactly the silent-test-gap pattern A1 itself was raised to close, displaced one facet over by the resolution that closed A1.

**Evidence**: F46 §(c) — "A membership-only change to an existing, non-created/non-dropped index that is not a rename is none of D15's enumerated categories... If the changed index entity is not tracked, commit does not persist the membership delta and the committed index silently fails to cover the new subclass collection — a polymorphic query under-returns." F46 Resolution — "add collection-membership change to the changed-index set as an in-place-mutation category alongside rename (F40)... route the mutation through the tx-local index overlay and apply it commit-only." PSI (eid `adv-iter2-verify`) over current code confirms the de-guarding targets exist and have the claimed shape: `IndexManagerEmbedded#addCollectionToIndex` / `#removeCollectionFromIndex` self-commit via `executeInTxInternal` with no throw-guard (the facet-(a) sites I-A7's test catches); `SchemaShared#saveInternal`, `SchemaEmbedded#dropClass`/`dropClassInternal`, `IndexManagerEmbedded#createIndex`/`#dropIndex` throw on active tx (caught loudly). A grep of the Invariants + Delegated sections (lines >=5938) for membership / polymorphic / subclass coverage returns only I-A7's isolation+rollback test (lines 162-164), I-P2's negative commit-only clause (line 336), and I-P2's mechanism listing of the overlay category (line 341) — no positive post-commit coverage test, and no entry naming the F46 facet (c) "membership-only is its own changed-index category" requirement as a property a test checks.

**Proposed fix**: Add a positive *Test* case for the membership-coverage property, on whichever of I-A7 or I-P2 owns it (I-P2 is the natural home, since the overlay / changed-index tracking is I-P2's mechanism and F46's resolution routes the persistence through D15). Concretely: create a parent class with an index and a subclass with its own collection; in one transaction `addSuperClass` (or alter-add-collection) so the subclass collection joins the parent index's membership; commit; assert a polymorphic query over the parent that targets a subclass row resolves *through the index* (not a scan fallback) and returns the row, and restart and confirm the same. State in the owning invariant that a membership-only change to an existing, non-created / non-dropped, non-renamed index is a tracked changed-index category in its own right (F46 facet (c) / the generalized D15 third category), so the implementer cannot satisfy the de-guarding (I-A7) and the no-mid-tx-mutation (I-P2) tests while omitting the changed-index tracking. This closes the third facet with the same testable-property discipline A1 applied to the first two.

## Evidence base

#### C6 — Assumption test: the post-commit positive coverage of a polymorphic membership change is pinned by a catching test
- **Claim**: After the A1 resolution, every load-bearing facet of F46 (the self-commit guard, the in-place shared-`Index` mutation, and the missing membership-only changed-index category) is captured by an invariant whose test catches the failure.
- **Stress scenario**: An implementer de-guards `addCollectionToIndex` / `removeCollectionFromIndex` so the membership change rides the user tx and routes it through the tx-local overlay (so no shared `Index` is mutated mid-tx). I-A7's isolation+rollback test passes; I-P2's "never mutate a shared `Index` mid-tx" clause passes. But the implementer treats the membership change as covered by the overlay's read-side effect alone and never adds the affected index to the commit-time changed-index set (F46 facet (c)'s exact omission). At commit, the create/drop/rename reconciliation finds no create, no drop, no rename for this index, so it persists no membership delta. After commit, the parent index's on-disk `collectionsToIndex` still excludes the new subclass collection; a polymorphic query through the parent index under-returns the subclass rows. No listed invariant test exercises this positive post-commit coverage, so the regression ships green.
- **Code evidence**: PSI (eid `adv-iter2-verify`) — `IndexManagerEmbedded#addCollectionToIndex(DatabaseSessionEmbedded,String,String,boolean)` and `#removeCollectionFromIndex(DatabaseSessionEmbedded,String,String)` both `executeInTxInternal` with no `isActive()` throw-guard, matching I-A7's facet-(a) enumeration; the throw-guard sites (`saveInternal`, `dropClass`/`dropClassInternal`, `createIndex`, `dropIndex`) all carry the `isActive()`-throw shape I-A7 claims. F46 §(c) and its Resolution define facet (c) as a separate changed-index category whose omission yields a silent polymorphic under-return at commit. Grep of the Invariants + Delegated sections (lines >=5938) for membership / polymorphic / subclass coverage returns only the I-A7 isolation+rollback test, I-P2's negative commit-only clause, and I-P2's mechanism listing — no positive coverage test and no entry naming facet (c) as a checkable property.
- **Verdict**: GAP — facets (a) and (b) are now caught by I-A7's added test (A1 resolved); facet (c)'s positive post-commit coverage is testable, load-bearing (a silent data-correctness regression: polymorphic queries under-return after a committed `addSuperClass`), and pinned by neither invariant's test. The fix is one positive test case plus a one-line invariant clause, not a re-decision.

#### V-A1 — Verdict: A1 (I-A7 de-guarding contract) — VERIFIED
- **Prior finding**: No invariant named the de-guarding of the six self-commit / throw transactionality enablers (F3 `saveInternal`, F4 `dropClass`/`dropClassInternal`, F21 `createIndex`, F26 `dropIndex`, F46 `addCollectionToIndex`/`removeCollectionFromIndex`), and no test built the silent self-commit membership leak.
- **Resolution checked**: I-A7 ("Every schema and index mutation entry point rides the user transaction", lines 6086-6112) states the de-guarding contract, enumerates every throw site and every self-commit site by method, and flags the self-commit sites as the dangerous silent ones ("left in place they do not throw, they silently commit a collection-membership change in a nested micro-transaction that escapes the user transaction"). Its *Test* constructs the silent failure: trigger a polymorphic membership ripple (`addSuperClass` or alter-add-collection on a class with an indexed subclass), assert a concurrent session does not observe the change before commit, assert rollback leaves the shared `Index`'s `collectionsToIndex` untouched, and distinguishes the throw-guard (fails any DDL loudly) from the self-commit guard (passes a naive DDL test, fails this one). Provenance: D1, D8, D15; F3, F4, F21, F26, F46.
- **Code evidence**: PSI (eid `adv-iter2-verify`) confirms all enumerated sites exist with the claimed shape — `saveInternal` throws on `tx.isActive()` and self-commits via `executeInTx`; `dropClass`/`dropClassInternal`, `createIndex`, `dropIndex` throw on active tx; `addCollectionToIndex`/`removeCollectionFromIndex` self-commit via `executeInTxInternal` with no throw-guard. The de-guarding contract is grounded and the test would fail if a self-commit guard were left in place (the concurrent-leak and rollback-leak assertions catch it).
- **Verdict**: VERIFIED — the contract is pinned and the catching test is present. (The facet-(c) gap is a new, distinct issue raised as A4, not a defect in this resolution.)

#### V-A2 — Verdict: A2 (I-A1 positive committed-drop test + set-difference detection source) — VERIFIED
- **Prior finding**: I-A1 had no positive test that commits a class DROP and asserts the collection / engine is actually removed; the set-difference drop-detection path (D9/F43) was undefended against the natural changed-record-set (D6) misreading, under which a committed drop silently removes nothing.
- **Resolution checked**: I-A1 (lines 5966-5994) now states the create/drop set "is detected from the collection-id set difference over the committed versus tx-local in-memory structures (D9), never from the transaction's changed-record set (D6)," and explains that a changed-record-set diff "would silently drop nothing" because a dropped class's record is deleted and carries no per-property change signal. The *Test* adds the positive committed-drop case (create a class with data and an index, commit; in a second tx drop the class, commit; assert collection files, engine files, and registry entries are gone and the data is unreadable; restart and confirm) and explicitly states a changed-record-set drop-detection "passes the rollback and crash cases but fails this positive drop, which is the property it defends." Provenance now cites F37, F43 alongside D9.
- **Verdict**: VERIFIED — the positive drop test is present and the set-difference-not-changed-record-set property is now defended in both the invariant statement and the test's failure clause.

#### V-A3 — Verdict: A3 (lazy snapshot-rebuild invalidation Delegated entry) — VERIFIED
- **Prior finding**: F51's lazy-invalidation requirement (eager rebuild = O(N^2) on a batch) appeared in neither the invariant list nor the Delegated section; I-P2's correctness test passes under the O(N^2) misimplementation.
- **Resolution checked**: A `## Delegated to implementation` entry "Snapshot-rebuild invalidation is lazy, not eager" (lines 6544-6551) now exists, tied to I-P2 and citing F35, F51. It pins lazy invalidation (null the snapshot, rebuild on next read, O(1)) versus eager reconstruction (O(current schema size) per call → O(N^2) on the F48 4,000-index batch, "the write-amplification class YTDB-382 exists to kill, reappearing on the read side"), notes I-P2's correctness test passes under either, and frames it as a performance property the implementer owns, folded next to F50's write-side entry — exactly the proportionate Delegated-not-invariant fix A3 proposed.
- **Verdict**: VERIFIED — present, correctly tied to I-P2, and scoped as the suggestion intended.
