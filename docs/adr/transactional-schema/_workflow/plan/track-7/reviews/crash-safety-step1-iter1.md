# Crash-safety / durability review — Track 7 Steps 1-2 + Step-1 review fixes (iter 1)

**Artifact:** `git diff e2605c8ba3..1063e1d987`, commits `cb2d4d3b79`, `fafac7e8b3`, `1063e1d987` (docs-only commits in range ignored).
**Charter:** persistence ordering, WAL/atomic-op state, fsync boundaries, recovery invariants, failure/exception-path state consistency.
**Method:** decision criteria + numbered premises per claim; every trace cites file:line of code actually read; each defect carries a concrete counterexample, each no-issue a justification; hypothesis log at the end. Read-only review; no Maven executed (another thread owns test execution in this worktree).

## Decision criteria

1. **Replay/registry agreement.** After every failure shape (and after any crash point inside it), the state WAL replay reconstructs on reopen and the state the in-memory registries advertise must either agree, or the disagreement must be contained (storage error state / fail-fast) and self-heal on reopen.
2. **No durable destruction from cleanup.** A failure-path cleanup that runs in its own committing atomic operation must never durably delete a structure the rolled-back commit did not create.
3. **Undo completeness.** "A failed commit leaves the in-memory state exactly as before the commit" must cover *all* in-memory registries mutated synchronously by the reconcile phases, not only the ones the undo arms name.
4. **Snapshot validity.** Every fresh-committed-read must ride a snapshot that is (a) coherent under concurrent apply and (b) vacuum-protected for its whole lifetime.
5. **Containment honesty.** A comment that justifies leaving divergent state standing must accurately describe what remains reachable while the divergence exists.

## Findings

### CS20 [should-fix] The drop-restore arm never restores the collection's link-bag B-tree into `fileIdBTreeMap`; the restored collection NPEs on its first link-bag operation

**Files:** `AbstractStorage.java:3470-3500` (drop-restore arm of `undoReconciledCollections`), `AbstractStorage.java:3300` (`reconcileCollections` calling `deleteComponentByCollectionId`), `LinkCollectionsBTreeManagerShared.java:100-110, 112-141, 143-152`.

**Premises.**
1. `reconcileCollections` drops a committed collection through `dropCollectionInternal` + `configuration.dropCollection` + `linkCollectionsBTreeManager.deleteComponentByCollectionId(atomicOperation, committedId)` (`AbstractStorage.java:3283-3301`).
2. `deleteComponentByCollectionId` (`LinkCollectionsBTreeManagerShared.java:100-110`) does `fileIdBTreeMap.remove(intFileId)` **eagerly** (in-memory, not rollback-aware) and buffers the file delete on the atomic operation.
3. On a failed commit the rollback restores the durable `global_collection_<id>.grb` file (the delete was buffered — `AtomicOperationBinaryTracking.deleteFile:922-937` adds to `deletedFiles`, applied only in `commitChanges`), and the drop-restore arm re-registers the collection object via `registerCollection` (`AbstractStorage.java:3478`, body at `6789-6812`) — which touches only `collectionMap` + `collections`.
4. Nothing re-puts the removed `SharedLinkBagBTree` into `fileIdBTreeMap`.
5. Later link-bag access resolves through that map: `loadIsolatedBTree` (`LinkCollectionsBTreeManagerShared.java:143-152`) does `fileIdBTreeMap.get(intFileId)` and constructs `IsolatedLinkBagBTreeImpl` around the result **without a null check**; `doCreateRidBag`'s `fileId >= 0` branch (`:133-141`) does the same (the durable file is present again, so `fileIdByName >= 0` and the create-fresh branch that would re-put is not taken).

**Counterexample.** Class `C` (committed, has vertices with edge link-bags). Tx: `dropClass("C")`; commit fails with a retry-family fault (in-window hook, or the new shape-1 `endTxCommit` catch — `CommandInterruptedException` per the new tests, so the storage stays open). Undo restores `C` in the registry; the test-style probes (`getCollectionNames`, plain-row insert) all pass. The user then reads an existing vertex's edge bag in `C` → `loadIsolatedBTree` → `fileIdBTreeMap.get` → `null` → NPE inside `IsolatedLinkBagBTreeImpl`; creating a new link bag NPEs the same way through `doCreateRidBag`'s existing-file branch. Broken until reopen (open-time load pass rebuilds the map — self-heals, like the accepted config-cache gap, but this one throws on user data operations, not on a passive config read).

**Interaction with the diff.** The map-removal and restore arms pre-date this diff, but (a) the diff's new `endTxCommit` shape-1 catch adds a whole new class of failures that now route through this incomplete restore (previously they propagated with no undo at all — worse overall, but the *map* gap was then subsumed in general registry breakage), and (b) the diff's slot-reuse analysis makes the omission visible: on the **slot-reuse** path the gap is incidentally masked, because `createComponent` for the reusing create put a replacement `SharedLinkBagBTree` into the map under the resurrected (identical) fileId, and `SharedLinkBagBTree` carries no state beyond `fileId` (`SharedLinkBagBTree.java:30-46`), so the replacement object serves the restored survivor's durable pages correctly. Only the **pure-drop** (no reuse) failure path is exposed. The new tests both happen to drop-and-create (first-null-slot reuse), so neither exercises the exposed path, and neither probes link bags.

**Crash matrix.** A hard crash anywhere in this window is safe (durable state is pre-commit; reopen rebuilds the map from files). The defect is live-process-only, matching decision criterion 3, not 1.

**Suggestion.** In the drop-restore arm, re-register the link-bag component: reconstruct a `SharedLinkBagBTree(storage, FILE_NAME_PREFIX + id, FILE_EXTENSION)`, `load` it on a read-only/fresh operation, and re-put it into `fileIdBTreeMap` (a `restoreComponentByCollectionId` mirror of `deleteComponentByCollectionId`); extend `assertClassWritableAndReadable`-style probes with an edge/link-bag write on the restored class.

### CS21 [should-fix] The shape-2 arm's containment claim "every later operation fails fast" is inaccurate: reads never consult the error state, and the durably-committed sub-shape serves post-commit data under the unpromoted schema

**Files:** `AbstractStorage.java:3086-3115` (shape-2 comment + arm), `AbstractStorage.java:6060-6070` (`checkErrorState`), `AtomicOperationsManager.java:524` (its only caller), `AbstractStorage.java:6043-6054` (`checkOpennessAndMigration` — status stays `OPEN` in error state).

**Premises.**
1. `setInError` (`AbstractStorage.java:2063`) sets `error`; the only production gate that consults it on the operation path is `AtomicOperationsManager.acquireExclusiveLockTillOperationComplete:524` — i.e., component-**write** operations. `checkOpennessAndMigration` checks `status`, which remains `OPEN`.
2. Record reads (`DatabaseSessionEmbedded.executeReadRecord` → `storage.readRecord`), schema-snapshot queries, `getCollectionNames`, and index lookups take no error-state gate. The new test itself demonstrates this: after poisoning, it happily calls `getCollectionNames()` (`SchemaCommitReconciliationTest.endTxCommitFailureWithoutRollbackKeepsPublicationAndPoisonsStorage`).
3. Sub-shape analysis of "no-internal-rollback" throws inside `endAtomicOperation` (`AtomicOperationsManager.java:279-464`):
   - **commitChanges throw** (the realistic family): `atomicOperationsTable.commitOperation` is never reached, the entry stays in-progress, so SI keeps the failed commit's record versions invisible — readers keep a *coherent pre-commit* logical view. Divergence is limited to low-level registry surfaces (a phantom collection in `getCollectionNames`, a dropped collection whose class still resolves and then errors in `doGetAndCheckCollection`). Contained; acceptable.
   - **post-`commitOperation` throw** (`writeAheadLog.addEventAt` failing, `releaseLocks`/`deactivate` anomalies — narrow but real, and exactly what `endTxCommitPostDurabilityFailureTestHook` simulates): the table entry is COMMITTED, `flushSnapshotBuffers` ran (`AtomicOperationBinaryTracking.commitChanges:1113-1120`), pages are applied — the commit's data is **visible to every new snapshot**. But the rethrow skips promotion (`AbstractStorage.java:3160-3178` runs only after a successful `endTxCommit`), so the committed `SchemaShared` still carries the pre-commit class graph and the pre-commit `globalProperties` table. Other sessions now deserialize records written with new global-property ids against a schema that lacks them, and query classes through a stale class graph — wrong results / parse anomalies, not fail-fast errors, until someone reopens.
4. The comment's justification ("every later operation fails fast and the reopen rebuilds the registries") therefore overstates the containment on which the leave-standing decision rests (decision criterion 5).

**Counterexample.** Session A commits a schema-carry tx adding property `p` (new global property id) plus rows using it; `addEventAt` throws (WAL shutting down under memory pressure). Storage poisoned, publication stands, promotion skipped. Session B (already open, read-only analytics) queries the class: SI shows the new rows; the immutable schema snapshot (built from the unpromoted `SchemaShared`) lacks global property `p` → property values silently missing or deserialization error, depending on serializer path. No operation on session B ever throws the `checkErrorState` wrapper.

**What is right about the arm.** Leaving the publication standing is the correct half: undoing could de-register a durably committed collection (criterion 2), and the reopen recovery story is sound (`DiskStorage.clearStorageDirty:612-616` skips clearing when `isInError()`, so the next open runs `recoverIfNeeded` and replays WAL — C6). The defect is only that the fail-fast claim doesn't cover reads.

**Suggestion.** Either (a) gate the session read path (or session operation entry) on `checkErrorState` so the poison actually fails fast for readers too, or (b) in the shape-2 arm, when `atomicOperationsTable` reports the operation COMMITTED, continue into promotion instead of rethrowing pre-promotion (durability is *certain*, not in-doubt, in exactly the sub-shape that diverges), or minimally (c) correct the comment to state that reads keep serving a divergent view until reopen, so the trade-off is documented honestly.

### CS22 [suggestion] `setInError` silently skips `AssertionError`, so under `-ea` the shape-2 arm leaves an in-doubt commit's publication standing on a fully usable storage

**Files:** `AbstractStorage.java:3103` (shape-2 `setInError(e)`), `AbstractStorage.java:2063-2088` (`setInError` AssertionError skip), catch signature at `:3099` (`IOException | RuntimeException | AssertionError`).

**Premises.**
1. The catch deliberately includes `AssertionError` and the arm's comment acknowledges the skip ("acceptable: asserts never fire in production, and the rethrow below still fails the commit loudly").
2. `commitChanges` contains multiple asserts (`AtomicOperationBinaryTracking.java:1020-1021, 1102-1106`) reachable only under `-ea` — i.e., in every CI/test run and in any production deployment that enables assertions.
3. When one fires: rollbackInProgress is false → shape-2 arm → `setInError(AssertionError)` is a **no-op** → the storage stays open and writable, the registry publication stands, durable state is in-doubt (pre-end-record assert: nothing durable; the table entry is stuck in-progress), and the *committed schema is unpromoted* — yet subsequent commits proceed against registries containing a phantom collection whose files were never applied.
4. "The rethrow fails the commit loudly" is true but does not restore consistency: unlike every other shape, no containment (undo *or* poison) happens.

**Counterexample (test-env).** `-ea` run; a regression makes `hasPendingOperations` assert trip in `commitChanges` for a schema-carry commit that created class `N`. The commit fails; storage stays healthy-looking; the next test operation writes into phantom collection `N` → file-not-found / cache errors far from the root cause, cascading a single assert into a confusing failure chain — precisely the outcome the error-state containment exists to prevent.

**Suggestion.** In the shape-2 arm, wrap before poisoning: `setInError(e instanceof AssertionError ? new StorageException(name, "...", e) : e)` — mirroring how `endAtomicOperation` wraps AssertionError into `StorageException` before rethrow (`AtomicOperationsManager.java:428-458`). This keeps the dev/test-guard philosophy of `setInError` intact for stray asserts elsewhere while making *this* arm's containment unconditional.

### CS23 [suggestion] After a failed slot-reuse commit the config cache holds the failed create's entry at the survivor's slot (accepted gap changed shape; durable behavior strictly improved — not worse)

**Files:** `AbstractStorage.java:3535-3596` (slot-reuse branch skips `configuration.dropCollection`), `CollectionBasedStorageConfiguration.java:1485-1512` (`updateCollection` mutates `cache` eagerly), `:1570-1584` (`dropCollection` nulls `cache` slot eagerly).

**Premises.**
1. Failed drop-C/create-N-at-slot-5 timeline: reconcile's drop sets `cache[5]=null` + buffers the durable delete; the create sets `cache[5]=N-config` + buffers the durable store; rollback reverts both durable writes (C's config record intact) but the cache map is not rollback-aware (its own comment at `:1956` says so).
2. The **fixed** code correctly refuses the durable id-keyed deletes on the slot-reuse path (pre-fix, the cleanup's committing fresh operation durably destroyed the survivor's config entry and grb component — the blocker this commit fixes; decision criterion 2 now satisfied — see C1).
3. Residual: `cache[5]` retains N's config (name `c_<counter>`, wrong) while registry + durable state say C. This is a *shape change* of the known-accepted "config-cache slot gap for rolled-back drops": the accepted gap is stale-**null** (drop rolled back, no reuse); the new path leaves stale-**wrong**. Both self-heal on reopen via `preloadCollections` (`:1526-1567`). Live consumers of `getCollections()` are thin (tools/info paths; runtime record operations resolve through the registry), so impact stays in the accepted gap's class.
4. Checked per the tasking: **the fixes did not make the durable gap worse** — they eliminated a durable-destruction bug and left an in-memory-cache staleness of comparable (arguably marginally worse: wrong vs missing) severity.

**Suggestion.** In the drop-restore arm (or the slot-reuse branch), repair the cache from the captured survivor: `configuration` cache-put of `entry.collection().generateCollectionConfig()` (`PaginatedCollectionV2.generateCollectionConfig:309-323`), which would close both the stale-wrong (reuse) and stale-null (plain drop) shapes without any durable write.

### CS24 [suggestion] Shape-1 undo on an already-poisoned storage: structural arms degrade silently, and the `-ea` assert-in-catch of the engine drop-restore can replace the primary exception and skip the membership undo

**Files:** `AbstractStorage.java:3097-3101` (shape-1 catch → undo), `AtomicOperationsManager.java:330-336` (Hook A poisons via `moveToErrorStateIfNeeded` before rethrow), `:524` (`checkErrorState` in component-lock acquisition), `AbstractStorage.java:3792-3840` (`restoreReconciledDroppedIndexEngines` catch containing `assert false`), `:3395-3405` (arm order: restore before `undoAppliedMembership`).

**Premises.**
1. A *real* shape-1 failure (lifecycle persist-hook throw) calls `storage.moveToErrorStateIfNeeded(persistFailure)` (`AtomicOperationsManager.java:333`) before rethrowing. For a non-retry-family failure the storage is therefore **already in error state when the new endTxCommit catch runs its undo**. (The new tests inject `CommandInterruptedException` — `NeedRetryException` per `CommandInterruptedException.java:28` — so they only exercise the unpoisoned variant.)
2. On the poisoned variant, the undo's structural arms run fresh atomic operations whose component operations hit `checkErrorState` (`AtomicOperationsManager.java:524`) and throw:
   - `revertCreatedCollectionStructure` catches and warn-logs (bounded in-memory-profile leak; fine — the storage is poisoned anyway and reopen heals).
   - `restoreReconciledDroppedIndexEngines` catches, error-logs, then executes `assert false : ...` **inside the catch** (`AbstractStorage.java:3835-3838`). Under `-ea` this throws `AssertionError` out of `undoSchemaCarryRegistryPublication`, which (a) replaces the primary shape-1 exception propagating from the catch (`throw e` at `:3117` never runs) and (b) skips the trailing `indexManager.undoAppliedMembership(indexPlan)` (`:3404`).
3. The method's own Javadoc frames the assert as surfacing "a broken invariant" — but on this path the reconstruction failure is the *expected* consequence of the deliberate poisoning, not a broken invariant, so the assert misfires by design.
4. Production (`-ea` off) is correct: logged, undo continues, poison contains everything, reopen heals. The exposure is test-environment diagnosability plus the pre-existing same-family exposure on the old pre-endTxCommit failure branch (which has the identical poisoned-undo interaction through `rollback` → `endAtomicOperation` → `moveToErrorStateIfNeeded`); the diff extends that family's reach rather than creating it.

**Counterexample.** `-ea`; schema tx drops index I; the lifecycle persist hook throws a plain `RuntimeException` (non-retry) → storage poisoned + rollback → endTxCommit catch → undo → engine reconstruction hits `checkErrorState` (if `engine.load` routes through a component operation) → logged, then `AssertionError` escapes; the test observes an `AssertionError` about slot reconstruction instead of the injected persist failure, and the shared index's in-memory membership deltas remain applied on a poisoned storage.

**Suggestion.** Gate the structural/reconstruction arms (or at least the `assert false`) on `!isInError()`: when the storage is already poisoned, reopen is the designated healer and the arms' failures are expected, so skip them (or log-only) instead of assert-guarding.

## Evidence base

#### C1 [SAFE] Slot-reuse undo branch performs no wrong durable delete on either engine — VERIFIED
The Javadoc's three load-bearing claims all check out against code: (1) `addFile` un-deletes rather than eagerly installing when the name was buffered-deleted in the same operation (`AtomicOperationBinaryTracking.addFile:855-862`: `deletedFileNameIdMap.removeLong` + `deletedFiles.remove`, `isNew=false`, and the eager in-memory install at `:885-888` is gated on `isNew`), so the resurrected `global_collection_<id>.grb` reverts with the rollback on **both** engines and the file present afterwards is provably the restored survivor's; (2) config-entry writes are buffered page writes reverted on both engines (`CollectionBasedStorageConfiguration.dropCollection/updateCollection` route through `dropProperty`/`storeProperty` on the atomic operation); (3) `collection.exists` is name-keyed (`PaginatedCollectionV2.exists:304-306` → `isFileExists(atomicOperation, getFullName())`) and `collection.delete` (`:467-481`) deletes exactly the collection's own four name-derived files — never the id-keyed config entry or grb component. On the disk profile `exists` is false (rollback reverted the create), so the whole cleanup is a durable no-op there; the only durable writes the undo can ever make on disk are none. Decision criterion 2 satisfied. Crash *inside* the slot-reuse undo (before/after the name-keyed deletes, before/after the drop-restore re-registration) is safe at every point: durable state is pre-commit throughout (the failed unit has no `AtomicUnitEndRecord`; the fresh cleanup op is disk-no-op), and all remaining mutations are process-local, rebuilt on reopen.

#### C2 [SAFE] `droppedIds` classification in `undoReconciledCollections` is correct for all drop/create slot permutations — VERIFIED
Traced `AbstractStorage.java:3432-3467`. The `nextFreeCollectionId` free-slot assert (`:3326-3330`) pins the allocator premise the classification depends on.

#### C3 [SAFE] The endTxCommit catch's `isRollbackInProgress()` discrimination matches every throw source in `endAtomicOperation` — VERIFIED

#### C4 [SAFE] Fresh-committed-read scope vacuum-safety premise — VERIFIED
The scope registers nothing (`AtomicOperationsManager.startAtomicOperation:102-125` — snapshot capture only; no table entry, no freezer, no tsMin) and `deactivate()` (`AtomicOperationBinaryTracking.java:1434-1436`, `active=false`) is complete teardown, so no leak on success or throw (finally at `DatabaseSessionEmbedded.java:3645-3648`). Vacuum: eviction removes only visibility entries whose *death* ts is `< lwm` (`evictStaleSnapshotEntries:8498-8528` — headMap over `visibilityIdx`, i.e., superseded/dead versions only; a live latest version has no visibility entry and is never evicted), and `lwm ≤` the enclosing transaction's `tsMin` pinned at `startStorageTx` (`AbstractStorage.java:6460-6489`) until `resetTsMin` at transaction close (`:6510-6534`). Any version the fresh snapshot needs is alive at scope start; if superseded at T1 > scope-start ≥ tx-begin ≥ tsMin ≥ lwm, its death-ts exceeds lwm — protected. All four scope consumers run inside an active frontend transaction (copyForTx asserts it, `SchemaShared.java:280-282`; the three commit-time scopes run inside `commit`, and the promotion scope runs before the tx closes — status COMMITTING is `isActive()` per `FrontendTransactionImpl.java:1507-1511`), so the tsMin pin is live for the promotion scope too even though the commit's own atomic operation has ended. Additionally the three commit-time scopes hold `stateLock.writeLock()`, excluding superseding commits entirely.

#### C5 [SAFE] Fresh-read scope coherence during a concurrent commit's cache apply — VERIFIED
