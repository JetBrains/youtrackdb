## Findings

### A1 [should-fix]
**Certificate**: C1 (Scope challenge — the ~7-file footprint omits the second raw-record schema parser)
**Target**: Scope indicator (`> **Scope:** ~7 files …`) and the `## Interfaces and Dependencies` in-scope list
**Challenge**: The footprint lists `SchemaShared`/`SchemaClassImpl` serialization, the link set, the version constant, the open gate, and serializer tests — but NOT `DatabaseCompare`. PSI confirms exactly two main-code readers of the raw schema record's `"classes"` field: `SchemaShared` (in scope) and `DatabaseCompare.convertSchemaDoc` (out of scope). `convertSchemaDoc` reads `"classes"` as `Set<EntityImpl>` of embedded class docs and iterates each `classDoc.getProperty("properties")`. After this track, the root record's `"classes"` is a LINKSET of per-class record RIDs, so the `entity.<Set<EntityImpl>>getProperty("classes")` cast and the per-element `"properties"` access operate on links, not embedded docs — the schema-root comparison breaks. The byte/entity record-walk in `compare` will still visit the new per-class records as ordinary records (content is verified), but the `convertSchemaDoc` special-case for the root record is now malformed. This is reached only through `DatabaseCompare.compare`, which PSI shows is invoked from 15+ standing tests/ITs (`StorageBackupTest` ×11, `StorageBackupMTIT`, `StorageBackupMTRestoreIT`, `LocalPaginatedStorageRestoreFromWALIT`, `DbImportExportTest`, `DbImportStreamExportTest`, `LocalPaginatedStorageRestoreTx`, …) that assert `differences == 0`. The track will not compile-or-test green without touching `DatabaseCompare`, so the footprint is undercounted and the in-scope list omits a file the decomposition must include. (The risk review independently flagged this parser; this certificate confirms it is a hard test-green blocker for the track, not merely advisory, and that `DatabaseExport` does NOT share the hazard — it reads the logical schema API, not the raw record.)
**Evidence**: `DatabaseCompare.java:945-953` (`convertSchemaDoc` reads `"classes"` as EMBEDDEDSET of `EntityImpl`); `DatabaseCompare.java:783-787` (special-case fires only for the schema root record); `DatabaseCompare.java:694-728` (the all-collections record walk visits per-class records as ordinary records); `SchemaShared.java:530,657` (the only other raw-record `"classes"` reader, in scope); `DatabaseExport.java:453,461` (export reads `getImmutableSchemaSnapshot().getClasses()`, logical API — not affected). PSI usage search lists the 15+ `compare`/constructor call sites.
**Proposed fix**: Add `DatabaseCompare` to the in-scope list and raise the footprint sketch to ~8 files. Decompose a step (or fold into the format-change step) that updates `convertSchemaDoc` to handle the LINKSET root record — either resolve the links to the per-class records before coercing, or drop the now-redundant root-record special-case since the per-class records are already content-compared by the record walk. Either way the standing backup/restore/import ITs that assert `differences == 0` must stay green; name that as an acceptance line.

### A2 [suggestion]
**Certificate**: C2 (Cross-track-episode reality — Track 1's "clean replay" delivers nothing Track 2 functionally consumes)
**Target**: `## Interfaces and Dependencies` → "depends on Track 1 (clean replay underneath)"; Checklist "Depends on: Track 1"
**Challenge**: Track 1's episode fixes a missing-FILE redo during WAL replay — its helper `ensureFileForReplay` scans the unit for a matching `FileCreatedWALRecord` and materializes the file. Track 2 runs under today's storage-leads model (per its `## Context and Orientation`) and writes the schema through `saveInternal` → `session.executeInTx(... toStream ...)`. The per-class records it introduces are new metadata RECORDS in the existing schema collection, not new collection FILES. A new record allocates a record position; it does not emit a `FileCreatedWALRecord`. So Track 2's own write path never exercises Track 1's fix, and a crash mid-schema-write would replay through the ordinary record-update path, which Track 1 did not change. The declared dependency is therefore a conservative sequencing convenience (clean test ground, and Track 4 — which does create collection files at commit — is the real consumer of Track 1's fix), not a functional prerequisite. The dependency is harmless and the ordering survives; the rationale is just over-stated, which could mislead a reader into thinking Track 2 has crash-recovery coupling it does not.
**Evidence**: `SchemaShared.java:817-829` (`saveInternal` writes the schema record inside a tx via `toStream`, no file creation); `SchemaShared.java:644-674` (`toStream` writes properties on the schema EntityImpl, no `addFile`); plan D10 risk note and Track 1 episode (the F55 fix is about `FileCreatedWALRecord` missing-file redo, consumed by Track 4's I-A1); track-4 scope line ("structural delta … creates or drops collections"). No PSI reference accuracy caveat needed — the claim rests on the absence of any file-creation call in the schema-write path, which the read of `SchemaShared.saveInternal`/`toStream` confirms directly.
**Proposed fix**: Soften the dependency rationale to "ordering convenience — Track 2 creates records, not files, so it does not itself exercise Track 1's replay fix; the dependency keeps the per-class format change on a clean-replay base and is sequenced before Track 4, the real consumer of D10." Keep the `Depends on: Track 1` edge (it costs nothing and is the right stack order).

### A3 [should-fix]
**Certificate**: C3 (Invariant/scope challenge — the version bump collides with the still-accepted legacy V5 marker, so "reject-and-redirect" is not what the open gate does)
**Target**: Decision D14 ("bump the schema version into a reject-and-redirect gate"); plan `### Constraints` "New binaries reject an old-format database on a version check"; Track 2 Validation/Acceptance bullet 4
**Challenge**: The open-time gate at `SchemaShared.fromStream` line 501 reads `schemaVersion != CURRENT_VERSION_NUMBER && VERSION_NUMBER_V5 != schemaVersion`. Today `CURRENT_VERSION_NUMBER == 4` and `VERSION_NUMBER_V5 == 5` (a legacy 2.0-M1/M2 compatibility marker that the gate still accepts). The natural "bump the version" move is `CURRENT_VERSION_NUMBER = 5`. Trace: an OLD-format database written by current binaries carries `schemaVersion == 4`; with `CURRENT == 5` the predicate is `4 != 5 && 5 != 4` → true → it correctly throws the export/import redirect. But a genuine legacy 2.0-M1/M2 database carrying `schemaVersion == 5` evaluates `5 != 5 && 5 != 5` → false → the gate ACCEPTS it and `fromStream` proceeds to parse its `"classes"` as the new LINKSET format. A legacy V5 record stores `"classes"` as an EMBEDDEDSET of embedded class docs, so the new link-set-aware parse reads embedded entities where it expects links — a silent mis-parse / corruption instead of the loud reject-and-redirect the constraint and Validation bullet 4 promise. The decision SURVIVES with rationale strengthening: bumping to 6 (or any value ≠ 4 and ≠ 5), or removing the `VERSION_NUMBER_V5` acceptance arm as part of the gate flip, makes every pre-bump version reject cleanly. But "bump `CURRENT_VERSION_NUMBER`" stated without that constraint is a real implementation trap that breaks the acceptance test it is meant to satisfy. (The technical review flagged this; this certificate confirms the exact failing predicate and that the failure surfaces at the cross-track open path, not just at write.)
**Evidence**: `SchemaShared.java:62-66` (`CURRENT_VERSION_NUMBER = 4`, `VERSION_NUMBER_V5 = 5`); `SchemaShared.java:501-509` (the dual-accept gate predicate and the export/import redirect throw); Track 2 Validation bullet 4 ("Opening an old-format … is rejected with a redirect").
**Proposed fix**: In D14's risk note and the Plan of Work, specify the bump target as a value distinct from BOTH `VERSION_NUMBER_V4` (4) and `VERSION_NUMBER_V5` (5) — e.g. 6 — and either remove the `VERSION_NUMBER_V5 != schemaVersion` acceptance arm or fold it so the gate is a strict `schemaVersion != CURRENT_VERSION_NUMBER` reject. Add a regression test that opens a synthetic `schemaVersion == 5` record with the new binaries and asserts the export/import redirect throw (not a silent parse), making the "reject-and-redirect" acceptance line actually cover the legacy-V5 case.

### A4 [suggestion]
**Certificate**: C4 (Invariant challenge — I-U1's "root written exactly when its payload changes" is not observable under Track 2's own write path)
**Target**: Invariant I-U1 (Track 2 — "per-class records remove write amplification, and the root is written exactly when its payload changes")
**Challenge**: I-U1's second clause ("root written exactly when its payload changes") and the F59 regression (null `globalRef` / colliding collection name) depend on SELECTIVE per-class writes — only the changed class records plus the root when its non-link payload changed. But Track 2 runs under today's storage-leads model, where the schema is serialized wholesale by `toStream` (every class re-serialized, the root always written) on each `saveInternal`. The selective write set that could omit the root — and thus the F59 failure — is driven by D6 per-property dirty tracking, which is Track 4 work. So under Track 2 in isolation the root is ALWAYS written, the omission failure cannot occur, and the I-U1 "exactly when" clause has no observable counterexample to test here. The invariant HOLDS, but the test that proves the load-bearing half (root-omission → F59) cannot be written against Track 2's storage-leads path; it must wait for Track 4's selective write. Track 2's Validation bullet 2 (the F59 regression: "committed property-create … no null globalRef … fresh collection name") therefore over-promises what this track can actually demonstrate.
**Evidence**: `SchemaShared.java:644-674` (`toStream` re-serializes all classes and always sets `globalProperties`/`collectionCounter` on the root); `SchemaShared.java:817-826` (`saveInternal` calls `toStream` wholesale); plan D14 risk note and D6 ("per-property dirty tracking … limits the write to the classes that actually changed", implemented in Track 4); Track 2 `## Decision Log` D14 risk note ("D6's dirty tracking puts the root in the write set"); the sibling technical review's note that the F59-root-omission behavior is Track-4-owned.
**Proposed fix**: Re-scope Track 2's Validation bullet 2: the F59 root-omission regression is a Track-4 acceptance line (where selective per-class writes first make the omission possible). Track 2 should assert the weaker, observable property — a `toStream`→`fromStream` round-trip preserves the global-property table, collection counter, blob-collections, and each class's record RID — and leave the "root written exactly when its payload changes" / F59 assertion to Track 4. Note the dependency in I-U1's provenance so the test home is unambiguous.

## Evidence base

#### C1 [Scope challenge] DatabaseCompare.convertSchemaDoc is a second raw-record schema parser outside the footprint — WEAK
- **Chosen approach**: Track 2 scopes ~7 files: `SchemaShared`/`SchemaClassImpl` serialization, the link set, the version constant, the open gate, serializer/round-trip tests.
- **Best rejected alternative**: include every raw-record consumer of the schema `"classes"` field in the footprint.
- **Counterargument trace**:
  1. After the format change, the schema root record stores `"classes"` as a LINKSET of per-class record RIDs (D14 rationale: "a schema record that links to per-class entity records").
  2. `DatabaseCompare.convertSchemaDoc` (`DatabaseCompare.java:945`) reads `"classes"` as `Set<EntityImpl>` and iterates each `classDoc.getProperty("properties")` — it expects embedded class docs, not links. It fires only for the schema root record (`DatabaseCompare.java:783-787`).
  3. PSI confirms `convertSchemaDoc` and the surrounding `compare` are reached from 15+ standing tests/ITs (StorageBackupTest family, MT/restore ITs, DbImportExport*) that assert `differences == 0`. The per-class records ARE visited by the all-collections record walk (`DatabaseCompare.java:694-728`), so their content is still compared; the root-record special-case is the broken part.
- **Codebase evidence**: `DatabaseCompare.java:945-953`, `:783-787`, `:694-728`; PSI usage list (15+ `compare` callers); `DatabaseExport.java:453,461` shows export is NOT affected (logical API).
- **Survival test**: WEAK — the scope sketch is a soft signal, but omitting `DatabaseCompare` from the in-scope list will leave standing ITs red; the decomposition must include it, so the footprint and in-scope list need correcting.

#### C2 [Assumption test] Track 2 depends on Track 1's replay fix — but creates records, not files
- **Claim**: Track 2 "depends on Track 1 (clean replay underneath)".
- **Stress scenario**: a crash mid-schema-write under Track 2's storage-leads model.
- **Code evidence**: `SchemaShared.java:817-829` and `:644-674` — `saveInternal`/`toStream` write the schema EntityImpl record inside a tx; no `addFile` / `FileCreatedWALRecord` is emitted. Track 1's `ensureFileForReplay` only repairs a missing-FILE page redo. Track 4 is the track that creates collection files at commit and thus consumes D10/Track 1.
- **Verdict**: HOLDS as harmless ordering (YES, keep the edge), but the rationale is over-stated — Track 2 does not itself exercise Track 1's fix.

#### C3 [Invariant/Scope challenge] The version bump collides with the still-accepted legacy V5 marker — WEAK
- **Chosen approach**: "bump `CURRENT_VERSION_NUMBER` and turn the version check into a reject-and-redirect gate" (Plan of Work; D14).
- **Best rejected alternative**: bump to a value disjoint from all accepted legacy markers and tighten the gate to a strict single-version accept.
- **Counterargument trace**:
  1. Gate predicate `SchemaShared.java:501`: `schemaVersion != CURRENT_VERSION_NUMBER && VERSION_NUMBER_V5 != schemaVersion`. `VERSION_NUMBER_V5 == 5` is still an accepted legacy marker.
  2. Bumping `CURRENT_VERSION_NUMBER` to 5 makes a legacy `schemaVersion == 5` record evaluate `5 != 5 && 5 != 5` → false → ACCEPTED, then parsed as the new LINKSET format.
  3. A legacy V5 record holds `"classes"` as an EMBEDDEDSET of embedded docs, so the new parse mis-reads it: silent corruption instead of the loud reject the constraint promises.
- **Codebase evidence**: `SchemaShared.java:62-66`, `:501-509`; Track 2 Validation bullet 4.
- **Survival test**: WEAK — the decision survives if the bump target excludes 4 and 5 (e.g. 6) or the V5 acceptance arm is removed; stated as a bare "bump", it breaks its own acceptance test.

#### C4 [Invariant challenge] I-U1 "root written exactly when its payload changes" is unobservable under Track 2's wholesale toStream — HOLDS
- **Invariant claim**: per-class records remove write amplification AND the root is written exactly when its payload changes (F59 regression: no null globalRef, no colliding collection name).
- **Violation construction**:
  1. Start state: Track 2 in isolation, storage-leads model, schema persisted via `saveInternal` → `toStream`.
  2. Action: a property-create that advances the global-property table and an alter-add-collection that advances the counter.
  3. Intermediate state: `toStream` (`SchemaShared.java:644-674`) re-serializes ALL classes and ALWAYS writes `globalProperties`/`collectionCounter` on the root.
  4. Violation point: there is none under Track 2 — the root is always written, so the omission path (root NOT in the write set → F59) cannot be reached. Selective per-class writes (the only way to omit the root) are D6 dirty tracking, which is Track 4.
  5. Observable consequence: the F59 failure (null globalRef / colliding collection name) cannot be produced here; Validation bullet 2 has no constructible counterexample under this track.
- **Feasibility**: INFEASIBLE under Track 2 (the selective-write precondition does not exist until Track 4) — so the invariant HOLDS, but its load-bearing test belongs to Track 4.
