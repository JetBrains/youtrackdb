## Findings

### R1 [should-fix]
**Certificate**: Exposure — collection-primitive registry publication; Assumption — failed commit leaves no phantom registration (I-A4 / D10)
**Location**: Track 4 `## Plan of Work` steps 3 and 6, `## Interfaces and Dependencies` ("the `doAddIndexEngine`/`doDeleteIndexEngine` extraction") — and `AbstractStorage.doAddCollection` (`AbstractStorage.java:5002`) → `registerCollection` (`4963`) → `setCollection` (`4944`).
**Issue**: The track reconciles "through lock-free inner primitives (`doAddCollection`/`dropCollectionInternal`)" (Decision Log D3) and separately defers "shared-registry publication to the post-`commitChanges` success path" (Plan of Work step 6, D10). These two collide on the collection path. `doAddCollection` calls `registerCollection`, which writes the live `collectionMap` (put, line 4977) and the live `collections` array via `setCollection` (line 4983) — both *inside* the atomic-operation lambda, before `commitChanges`. The WAL atomic-operation revert reverts disk pages, not these in-memory Java maps; today the public `addCollection` self-commit tolerates this because a failure just throws and the storage is per-op consistent. Inside the user commit, reusing `doAddCollection` verbatim publishes the new collection into the shared registry before `commitChanges`, so a commit that fails at or after `commitChanges` leaves the collection registered in `collections`/`collectionMap` with no on-disk backing — exactly the phantom registration D10/I-A4 forbids ("a failed commit leaves no phantom registration … leaves its ids free to reuse"). The same shape exists on the engine path (`addIndexEngine` lambda does `indexEngineNameMap.put` + `indexEngines.add`, lines 2811-2812), but the track at least names the engine-primitive extraction; the collection-primitive restructuring is unnamed. Likelihood: high that a naive `doAddCollection` reuse ships the phantom-registration bug; impact: a failed schema commit corrupts the in-memory registry until the next reopen.
**Proposed fix**: Add an explicit Plan-of-Work item (and Decision Log note under D10) that the inner primitives must split file/engine *creation + id allocation* (runs inside the atomic op, reverted by the WAL) from *registry publication* into `collections`/`collectionMap`/`indexEngines`/`indexEngineNameMap` (runs only on the post-`commitChanges` success path). For collections this means `doAddCollection` (or a new lower primitive it calls) must stop calling `registerCollection`/`setCollection` during the atomic op when invoked from the commit, with publication moved to step 6's success path. Make I-A4's acceptance test assert the registry is clean (`collectionMap`/`collections` and `indexEngineNameMap`/`indexEngines` show no entry) after a forced commit failure, not merely that the next commit reuses the id.

### R2 [should-fix]

**Location**: Track 4 `## Decision Log` D6 (Risks/Caveats), `## Plan of Work`, `## Validation and Acceptance` — versus Track 2 file `_workflow/plan/track-2.md` gate item G2.
**Issue**: Track 2's Phase-A gate record explicitly carried the root-omission work forward: G2 reads "plan invariant I-U1 still bundles the Track-4 'root written exactly when its payload changes' clause — carried to Track 4's Phase A as the home for that clause and its F59 test." The design fully specifies the regression (design.md lines 449-471, invariant I-U1): if the root record is left out of the write set when its non-link payload changes, a committed property-create restarts into a null `globalRef` and the reverted `collectionCounter` regenerates colliding collection names. Track 4 is the owning track (per-property dirty tracking lands here, D6). The selective per-class write (D6's write-amplification win) is named but its F59 guard is missing. Likelihood: medium that decomposition lands the diff/reconciliation without the F59 guard, since the clause is absent from this track's authoritative sections; impact: a known silent-corruption regression (null `globalRef` / colliding collection names across restart) ships unguarded.
**Proposed fix**: During this Phase A, seed Track 4's Decision Log D6 Risks/Caveats (or a new note) with the root-record-write-on-payload-change rule and the F59 root-omission regression, add a Plan-of-Work item for the selective per-class write plus the root-dirtiness rule, and add the F59 regression to `## Validation and Acceptance` (property-create commit → restart → assert non-null `globalRef` resolution and no colliding collection name). Anchor the new I-U1 sub-clause home here, closing the Track-2 G2 forward-handoff.

### R3 [suggestion]
**Certificate**: Assumption — only two lock-based read sites remain on hot paths
**Location**: Track 4 `## Plan of Work` step 8, Decision Log D19, `## Interfaces and Dependencies` (the two snapshot-first conversions: `YTDBGraphImplAbstract.createVertexWithClass`, `SQLMatchStatement.getLowerSubclass`).
**Issue**: D19 takes `stateLock.writeLock()` for the whole schema-carrying commit and asserts the lock-based reader set "blocks once for the whole commit"; the mitigation is converting the two named hot lock-based read sites to snapshot-first so the hot paths stay unaffected. PSI confirms both targets are genuinely hot: `createVertexWithClass` is the per-vertex-creation path and `getLowerSubclass` is the per-MATCH-step path, and both enter the lock-based read via `getSharedContext().getSchema()` / `getMetadata().getSchema()`. The residual risk is the completeness of the "only two" enumeration: "hot path" is a semantic judgment grep cannot settle. There are ~25 production `getSchema()` call sites, the large majority off the per-record/per-query hot path (admin / import / security / genesis / reload), but if a third genuinely-hot lock-based reader exists that the design's enumeration missed, the whole-commit write lock turns into a read outage for it for the commit's duration. The two-site claim was validated by the design's adversarial passes and Track 3's prior work, so this is a low-probability residual, not a fresh defect.
**Proposed fix**: No structural change required. During decomposition, add one verification sub-step that re-confirms the two-site enumeration is exhaustive for the hot paths — i.e., that every other production `getSchema()` reader is either already snapshot-routed, off the commit-contended hot path, or itself a schema-write path — and record the result so the I-U5 / read-outage acceptance test (which already asserts a data commit runs concurrently on the read-lock path) covers the assumption.

### R4 [suggestion]
**Certificate**: Testability — I-A4 force-commit-to-fail-at-apply
**Location**: Track 4 `## Validation and Acceptance` ("Force a commit to fail at apply; no collection or engine is registered afterward and the next commit reuses the ids (I-A4)").
**Issue**: There is no built-in failure-injection hook in `AbstractStorage.commit` / `commitChanges` / `atomicOperationsManager` (grep for `injectFailure`/`testHook`/`@VisibleForTesting` crash points returns nothing on the commit path). The I-A4 test must force a failure precisely at apply — after the structural reconciliation has run but at/after `commitChanges` — to exercise the post-`commitChanges` publication-and-id-reuse guarantee. Without a hook this needs a Mockito spy on the storage or a reflective fault point, which is achievable (Track 1 settled its replay regression on a Mockito harness, `RestoreAtomicUnitPageOperationTest`) but is the kind of test that quietly degrades into a weaker variant (e.g., failing before `commitChanges`, which does not exercise the same window) if the injection point is not pinned. The I-A1 crash-recovery half has an established harness (the `LocalPaginatedStorageRestoreFromWALIT` close-copy-restore pattern), so it is lower-risk.
**Proposed fix**: In `## Idempotence and Recovery` / the I-A4 step, name the injection mechanism and the exact window (fault at/after `commitChanges`, not before) so the test verifies the registry-clean + id-reuse guarantee at the right point. If a reusable fault hook does not exist, decompose a small testability step that adds one (a `@VisibleForTesting` seam or a Mockito-spyable boundary) rather than leaving each recovery test to improvise its own crash point.

## Evidence base

#### Exposure: collection-primitive registry publication inside the atomic op
- **Track claim**: reconcile "through lock-free inner primitives (`doAddCollection`/`dropCollectionInternal`)" (D3) while deferring "shared-registry publication to the post-`commitChanges` success path" (Plan of Work step 6, D10).
- **Critical path trace**:
  1. Entry: `commit(FrontendTransactionImpl, allocated)` @ `AbstractStorage.java:2221` — takes `stateLock.readLock()` today (D19 changes this to writeLock for schema commits) and runs inside one atomic operation.
  2. (Track-4-added) reconciliation calls `doAddCollection(atomicOperation, name)` @ `AbstractStorage.java:5002`.
  3. `doAddCollection` → `collection.create(atomicOperation)` (file create buffered as intent, reverted by WAL) THEN `registerCollection(collection)` @ `AbstractStorage.java:5026`.
  4. `registerCollection` → `collectionMap.put(...)` @ `4977` and `setCollection(id, collection)` @ `4983` → `collections.set/add(...)` @ `4950/4952` — live in-memory maps, NOT reverted by the WAL atomic-operation rollback.
  5. `commitChanges` (later, inside `endTxCommit`) is the step rollback skips; the in-memory publication at step 4 already happened before this point.
- **Blast radius**: a schema commit that fails at/after `commitChanges` (or in the catch→`rollback` path) leaves the new collection registered in `collections`/`collectionMap` with no durable backing; the registry diverges from disk until reopen; the engine path (`addIndexEngine` lambda `indexEngineNameMap.put`/`indexEngines.add` @ `2811-2812`) has the identical shape.
- **Existing safeguards**: the WAL atomic operation reverts the on-disk file create (D10 — file create is buffered intent applied only in `commitChanges`); `deleteIndexEngine` already defers its in-memory map mutation to *after* the atomic op (comment @ `AbstractStorage.java:3053-3056`), proving the codebase already knows in-memory publication must trail the atomic op — but `doAddCollection`/`addIndexEngine` do NOT yet follow that discipline on the create path.
- **Residual risk**: MEDIUM — bounded and recoverable on reopen, but it is a live in-memory registry corruption on a failed schema commit unless the primitive is restructured; the fix is well-understood (mirror `deleteIndexEngine`'s defer-publication pattern).

#### Assumption: a failed commit leaves no phantom registration (I-A4)
- **Track claim**: "Force a commit to fail at apply; no collection or engine is registered afterward and the next commit reuses the ids (I-A4)."
- **Evidence search**: Read `doAddCollection`/`registerCollection`/`setCollection` and the `addIndexEngine` lambda; PSI find-usages on `addCollection`/`addIndexEngine`/`dropCollection`/`deleteIndexEngine` (each ≤2 production callers).
- **Code evidence**: `AbstractStorage.java:4977,4983` (eager collection publication); `AbstractStorage.java:2811-2812` (eager engine publication); `AbstractStorage.java:3053-3057` (the deferred-publication pattern the create paths must adopt).
- **Verdict**: UNVALIDATED
- **Detail**: I-A4 holds only if the inner primitives are restructured to defer registry publication to the post-`commitChanges` path; with the current primitives reused verbatim, a failed commit leaves a phantom in-memory registration. Drives R1.

#### Assumption: blast radius of extracting lock-free inner primitives is small
- **Track claim**: extract `doAddIndexEngine`/`doDeleteIndexEngine(atomicOperation, …)` from the public methods; reconciliation calls the inner primitives, the public methods stay as thin wrappers.
- **Evidence search**: PSI `ReferencesSearch` on `AbstractStorage.addCollection` (2 overloads), `addIndexEngine`, `dropCollection` (2 overloads), `deleteIndexEngine`, `loadExternalIndexEngine`.
- **Code evidence**: `addCollection` ← `DatabaseSessionEmbedded` only; `addIndexEngine`/`deleteIndexEngine`/`loadExternalIndexEngine` ← `IndexAbstract` only; `dropCollection` ← `DatabaseSessionEmbedded` + one internal `AbstractStorage` site. All production, no polymorphic surprises.
- **Verdict**: VALIDATED
- **Detail**: The public structural methods have a tight, non-polymorphic caller set, so extracting the inner primitives and re-pointing the commit at them is a low-blast-radius refactor. This is the reassuring counterweight to R1: the extraction itself is safe; only the publication-timing of the extracted bodies is the risk.

#### Assumption: the commit-local id allocator must be seeded inside the stateLock.write window (F88 pin)
- **Track claim**: D10 — "The commit-local allocator's seed read must sit inside the `stateLock.writeLock()` window so it excludes the non-commit engine registrars (`rebuild`, `loadExternalIndexEngine`, `recreateIndexes`) that run under `stateLock.write` alone."
- **Evidence search**: grep + Read for the live allocators and the background engine-registration path.
- **Code evidence**: engine-id allocator is `indexEngines.size()` (`AbstractStorage.java:2786`, `2691`); collection-id allocator is the first-null-slot scan in `doAddCollection` (`AbstractStorage.java:4990-4997`); `recreateIndexes` runs on a background thread (`IndexManagerEmbedded.java:608,619` — `RecreateIndexesTask` / "YouTrackDB rebuild indexes" thread) and calls `addIndexEngine`/`deleteIndexEngine`, both `stateLock.writeLock()`-guarded.
- **Verdict**: VALIDATED
- **Detail**: A background rebuild registering an engine concurrently would advance `indexEngines.size()` mid-commit; seeding the commit-local allocator under the same `stateLock.writeLock()` the schema commit holds (D19) excludes it. The two named live allocators are the only ones; the F88 pin is accurate.

#### Assumption: the D2 provisional-id predicate split is real and bounded (collectionId < 0 vs <= -2)
- **Track claim**: D2/D9 — "the schema layer tests `collectionId < 0` (not `== -1`) in 11+ places"; the in-memory map sites must treat provisional `<= -2` as pending-real while file/storage sites keep skipping negatives.
- **Evidence search**: grep for `collectionId < 0` / `getCollectionIds()[0] < 0` across `core/src/main` excluding tests.
- **Code evidence**: 30 `< 0` collection-id guard sites total; schema-layer subset confirmed at `SchemaShared.java:417,1068`, `SchemaEmbedded.java:371,550,566,590`, `SchemaClassImpl.java:1417`; abstract marker is the single id `-1`; storage-layer sites (`AbstractStorage.java:1526,1570,3793,5456`) are file/range checks that legitimately treat all negatives uniformly.
- **Verdict**: VALIDATED
- **Detail**: The "11+ places" claim is borne out (30 sites), the `<= -2` sub-range is disjoint from `-1`, and the split (schema in-memory maps vs file/storage sites) is the correct boundary. Concrete and well-scoped; not a finding on its own, but the five-item patch list (D2) is the highest-density correctness surface in the track — the property-value re-point (item 5) is the F58 silent-corruption case if skipped.

#### Exposure: D3 ordering — engine creation before lookup-by-id
- **Track claim**: D3 — engine creation must land before `lockIndexes`, which "resolves engines by id and throws on a missing one"; collection creation before the record-position-allocation loop.
- **Critical path trace**:
  1. `commit` calls `lockIndexes(indexOperations, atomicOperation)` @ `AbstractStorage.java:2297`.
  2. `lockIndexes` @ `5926` → `changes.getIndex().acquireAtomicExclusiveLock(atomicOperation)` — the lock acquire resolves the engine by id; `applyTxChanges`→`doPut` (`2480-2487`) resolves the engine by `index.getIndexId()` and throws `InvalidIndexEngineIdException` (`checkIndexId` @ `3077`) on a missing/null engine.
  3. Record-position allocation loop @ `2300-2359` → `doGetAndCheckCollection(collectionId)` → `collection.allocatePosition(...)` requires the collection to exist.
- **Blast radius**: if reconciliation does not create the engine before `lockIndexes`/`doPut`, the commit throws `InvalidIndexEngineIdException` and rolls back; if it does not create the collection before allocation, `doGetAndCheckCollection` fails. These are loud failures (rollback), not silent corruption.
- **Existing safeguards**: the `InvalidIndexEngineIdException` guard (`checkIndexId`) and `doGetAndCheckCollection` are the existing fail-loud checks; the WAL rollback path catches the resulting throw.
- **Residual risk**: LOW — the ordering constraint is load-bearing but its violation fails loudly and rolls back; D3 names it explicitly and the Plan of Work step 4 sequences it correctly (engines before `lockIndexes`, collections before allocation).

#### Assumption: the two read-site conversions target genuinely hot paths
- **Track claim**: D19 — convert `YTDBGraphImplAbstract.createVertexWithClass` and `SQLMatchStatement.getLowerSubclass` to snapshot-first so the whole-commit write lock is not a read outage on the hot paths.
- **Evidence search**: PSI `ReferencesSearch` on both methods + the read entry (`getSharedContext().getSchema()` / `getMetadata().getSchema()`); grep for the production `getSchema()` caller set.
- **Code evidence**: `createVertexWithClass` @ `YTDBGraphImplAbstract.java:121` is the per-vertex-creation path (reads `getSharedContext().getSchema()` @ `123`); `getLowerSubclass` @ `SQLMatchStatement.java:365` is the per-MATCH-step path (reads `session.getMetadata().getSchema()` @ `368`). ~25 production `getSchema()` sites total, the majority off the per-record/per-query hot path.
- **Verdict**: VALIDATED (for the two named sites) / partially UNVALIDATED (for the "only two" exhaustiveness)
- **Detail**: Both targets are confirmed hot and lock-based, so the conversions are warranted. The exhaustiveness of "only two remain" is a semantic hot-path judgment that grep cannot fully settle; drives the R3 suggestion. The design's adversarial passes already exercised this enumeration, so the residual is low-probability.

#### Testability: I-A1 crash-recovery vs I-A4 commit-fail-at-apply
- **Coverage target**: 85% line / 70% branch.
- **Difficulty assessment**: I-A1's crash-recovery half (crash after the transaction body, before commit; recovery leaves a clean state) has an established harness — the `LocalPaginatedStorageRestoreFromWALIT` close-copy-restore pattern plus Track 1's Mockito replay harness (`RestoreAtomicUnitPageOperationTest`), and Track 1 already landed the F55 replay fix this rests on. I-A4 (force a commit to fail *at apply*) has no built-in failure-injection hook on the commit path; the test must spy/fault at/after `commitChanges`, a narrow window easy to miss.
- **Existing test infrastructure**: `RestoreAtomicUnitPageOperationTest` / `RestoreAtomicUnitNonDurableSkipTest` (Mockito replay harness, `core/src/test/.../storage/impl/local/`); `LocalPaginatedStorageRestoreFromWALIT` and siblings (`.../storage/impl/local/paginated/`); Track-3 schema-tx fixtures `CopyForTxTest`, `SchemaProxyRoutingTest`, `MetadataWriteMutexTest`.
- **Feasibility**: ACHIEVABLE (I-A1) / DIFFICULT (I-A4)
- **Detail**: I-A4 is achievable but needs an explicit, pinned fault point; without one the test risks degrading into a weaker variant that does not exercise the post-`commitChanges` publication-and-id-reuse window. Drives R4.
