## Findings

### R1 [should-fix]
**Certificate**: E1 (Exposure — tx-aware snapshot cache invalidation on the commit path), E7 (Assumption — reconcile re-keys the tx-local class before the working-set build)
**Location**: `## Plan of Work` "Guard the commit-path read" (track-5.md:119-123); D21 Risk (1) (track-5.md:60); `SchemaShared.resolveProvisionalCollectionIds` and `SchemaShared.makeSnapshot` (`core/.../metadata/schema/SchemaShared.java`); `AbstractStorage.computeCommitWorkingSet` (`AbstractStorage.java:2363-2422`, called at 2528).
**Issue**: D21 Risk (1) frames the commit-path hazard as "hand back the reconciled real collection id, never a provisional id," and the plan proposes to "confirm the tx-local class carries its reconciled real collection id by that point, or guard the read." The trace shows the class-array patch is already done before the working-set build — `AbstractStorage.commit` calls `txLocalSchema().resolveProvisionalCollectionIds(getResolvedCollectionIds())` at line 2489, before `computeCommitWorkingSet` at line 2528 — and `resolveProvisionalCollectionIds` patches every class's collection-id arrays and rebuilds `collectionsToClasses`. So the tx-local `SchemaShared` *class object* is provisional-free by line 2528.

The residual, sharper hazard the plan does not name is the **snapshot cache**, not the class object. `SchemaShared.makeSnapshot(session)` memoises `this.snapshot` and only rebuilds when `forceSnapshot()` nulls it. Once D21 makes `SchemaProxy.makeSnapshot()` resolve the tx-local `SchemaShared`, the tx-local copy owns its own `snapshot` field. If any read during the transaction built that snapshot (a query, a MATCH, an `EntityImpl.getImmutableSchemaClass` cache-fill — the D21 read-your-writes path is the whole point of the change), the `ImmutableSchema` captured the tx-created class with its provisional id (`<= -2`). `resolveProvisionalCollectionIds` at line 2489 patches the mutable `SchemaShared.classes` arrays but does not call `forceSnapshot()` on the tx-local copy. At line 2528 `computeCommitWorkingSet` reads `entity.getImmutableSchemaClass(session)` → `session.getMetadata().getImmutableSchemaSnapshot()` → the still-cached tx-aware snapshot → `SchemaImmutableClass.getCollectionForNewInstance` returns the provisional id → `doGetAndCheckCollection` → `checkCollectionSegmentIndexRange` throws `IllegalArgumentException("Collection segment #-2 does not exist")` for any id `< 0`.

`EntityImpl.getImmutableSchemaClass` caches `immutableClazz` and only invalidates when `immutableSchema.getVersion()` changes, so even a fresh snapshot object does not guarantee re-resolution unless the version bumps. Likelihood: high for the exact target scenario of D21 — a transaction that creates a class, inserts an entity, and commits (the "commit succeeds and the commit-path collection-id resolution does not fail on a provisional id" acceptance line, track-5.md:184-185). This is the scenario D21 exists to enable, so it is on the primary happy path, not a corner case.
**Proposed fix**: Add an explicit step (and its ordering constraint) that force-rebuilds the tx-local snapshot immediately after `resolveProvisionalCollectionIds` at line 2489 and before `computeCommitWorkingSet` at line 2528, and clears the per-operation refcount-pinned snapshot (`MetadataDefault.immutableSchema` via `forceSnapshot` / snapshot re-pin) so the working-set read re-resolves against the patched, real-id tx-local class. Add a decomposition note that `EntityImpl.getImmutableSchemaClass`'s version-keyed cache must see a version change or a null-and-rebuild across this point. The D21 commit-path acceptance test (create class + insert + commit) is the regression guard; assert the committed entity landed in the real collection, not that commit merely did not throw.

### R2 [should-fix]
**Certificate**: E2 (Exposure — the 94-production-caller snapshot read tier), E8 (Assumption — a read never seeds the tx-local copy)
**Location**: `SchemaProxy.makeSnapshot` (`SchemaProxy.java:75-80`, the `delegate.makeSnapshot(session)` line the design cites as line 78); D21 (track-5.md:57-62); Integration Points (implementation-plan.md:305-307).
**Issue**: D21 changes one method but flips read semantics for a large tier. `ReferencesSearch` on `MetadataDefault.getImmutableSchemaSnapshot` returns 174 references, of which 94 are production (80 test) — the design's "174 call sites" figure is exact and confirmed. Today `SchemaProxy.makeSnapshot()` deliberately reads the committed `delegate` ("Tier 1: the immutable snapshot is taken from the committed instance, not the tx-local copy"); D21 makes it resolve the tx-local `SchemaShared` when a schema/index tx is active. Every one of those 94 production readers — the query planner, entity serializers, and the security stack — begins seeing tx-local classes/properties/rules mid-transaction. Two concrete exposures:

(1) The security path reads the snapshot to resolve class/collection identity; a tx-created class now appears there mid-tx, and its physical collection does not exist yet (provisional id, D2). D21 Risk (2) covers the query/MATCH planner via the D13 skip extension, but the plan does not enumerate the non-planner readers (serializers, security) that also resolve a collection or engine off the snapshot and could hit the same provisional-id / unbuilt-engine surface the planner guard is meant to catch.

(2) The change must be a strict no-op outside a schema/index tx (the pure-data-read hot path must not pay for tx-awareness or lose the committed fast read). The safeguard exists: `SchemaProxedResource.resolve()` returns the committed `delegate` unchanged when `session.getTxSchemaState() == null`, and a read never seeds the tx-local copy (only `resolveForWrite()` seeds). So an outside-tx `makeSnapshot` can keep the committed fast path for free, and an in-tx read of a tx that has taken no schema write (no `TxSchemaState`) also stays committed-only. This is a real safeguard, but the plan does not state the D21 change must route through `resolve()`-style null-check logic rather than unconditionally consulting the tx-local copy, and must not seed.
**Proposed fix**: In `## Plan of Work` for the D21 change, state the invariant explicitly: `makeSnapshot()` returns the committed snapshot unchanged when no `TxSchemaState` exists (mirroring `resolve()`), consults the tx-local `SchemaShared` only when one exists, and never seeds a tx-local copy from a read. Add a decomposition step to enumerate the non-planner snapshot readers (serializer, security) and confirm each either tolerates a tx-created (provisional-collection, unbuilt-engine) class or is covered by the D13 skip extension; add a test that a tx-created class resolved through the serializer/security path in-tx does not fault. Add an acceptance line that a pure-data read/commit with no schema write takes the committed snapshot (no tx-local resolution, I-U5 fast-path parity).

### R3 [should-fix]
**Certificate**: E3 (Exposure — failed engine-creating commit, in-memory orphan)
**Location**: `## Plan of Work` gap (1) failed-commit engine cleanliness (track-5.md:128-134); acceptance line I-A4 engine arm (track-5.md:189-191); `AbstractStorage.undoReconciledCollections` (`AbstractStorage.java:2879`); `doAddIndexEngine`/`publishIndexEngine`/`doDeleteIndexEngine` (`AbstractStorage.java:3402`/`3444`/`3715`).
**Issue**: The commit-time engine build lands on a proven seam — Track 4 already split engine creation into `doAddIndexEngine` (files inside the atomic op, no registry publish), `publishIndexEngine` (deferred in-memory publish past `commitChanges`), and `doDeleteIndexEngine` (atomic-op delete, no registry mutation). The public `addIndexEngine`/`deleteIndexEngine` take `stateLock.writeLock()` and would self-deadlock under the held write lock, so the commit build must use the `do*`/`publish*` primitives — the plan should name this to avoid an implementer reaching for the public methods. The genuine residual risk is the same in-memory orphan Track 4 hit for collections: `undoReconciledCollections`'s own Javadoc records that the in-memory engine (`AtomicOperationBinaryTracking.addFile` / `DirectMemoryOnlyDiskCache`) does not revert `addFile` on rollback, so a created collection's id-named `.grb` component survives a failed commit and blocks the next id-reusing create. A tx-created index's engine files are created the same way, so a failed engine-creating commit on the default in-memory profile orphans the engine's id-named component; the next commit that reuses that engine id then hits "file already exists." The plan's I-A4 engine-arm acceptance line asserts "no entry in `indexEngines`/`indexEngineNameMap`" (the registry) but does not assert the on-disk/in-cache engine component is gone — the exact gap that made Track 4's `failedSchemaCommitLeavesNoPhantomRegistration` test red on the in-memory profile until the create-side structural-revert arm was added (see the Track 4 episode and YTDB-1175).
**Proposed fix**: Add to the failed-commit engine-cleanliness step a create-side structural-revert arm for engines that mirrors `undoReconciledCollections`: on a failed engine-creating commit, drop the orphaned engine component in a fresh atomic operation (guarded on the component still being present, no-op on disk). Extend the I-A4 engine-arm acceptance test to run on the default in-memory profile (not only disk) and assert both the registry is clean and the engine id is reusable on the next successful commit — mirroring the Track-4 lesson that the in-memory profile is the one that catches this.

### R4 [should-fix]
**Certificate**: E4 (Assumption — the D12 v1 boundary is a settled decision)
**Location**: D12 Risks/Caveats "The v1 behavior at the boundary ... is the open Phase-1 decision this track settles" (track-5.md:46); `## Plan of Work` "Settle the v1 populated-class boundary (loud reject vs documented heap envelope)" (track-5.md:114); acceptance line (track-5.md:175-176).
**Issue**: D12 explicitly carries an unmade decision into the track: whether a populated-class index build beyond the v1 bound is a loud rejection pointing at YTDB-1064 or an accept-with-documented-heap-envelope. This is not a code risk per se but a decomposition risk — the two arms produce different code (a size-check-and-throw predicate vs a build-anyway path with a heap-budget guard) and different acceptance tests (assert the exception vs assert the build completes within the envelope). Leaving it open means step decomposition cannot write the build step or its test deterministically, and an implementer will either guess or return DESIGN_DECISION_NEEDED mid-Phase-B (the Track 4 pattern that forced two mid-phase re-decompositions). The forward hazard is real: Track 8's genesis builds `OUser.name` on a class that is empty at genesis, so the v1 empty-class scope is enough for the smoke test — but the boundary predicate still has to be written, and "empty class" needs a precise definition (zero records in the source collection at commit, checked how, under which lock).
**Proposed fix**: Settle the boundary in Phase A (Track Pre-Flight amendment) before decomposition: pick loud-reject or documented-envelope, and pin the "empty class" predicate to a concrete check (e.g., the source collection's record count read lock-free inside the commit window). Record it in the Decision Log so the build step and its acceptance test are deterministic. If it cannot be settled without design input, flag it as a Track Pre-Flight escalation rather than letting it surface as a Phase-B DESIGN_DECISION_NEEDED.

### R6 [suggestion]
**Certificate**: E6 (Exposure — create-time provisional-collection index gap)
**Location**: `## Plan of Work` gap (2) create-time provisional-collection index gap (track-5.md:135-139); acceptance line (track-5.md:192-194); `IndexManagerEmbedded.createIndex` deferred branch (`IndexManagerEmbedded.java:435-443`); `findCollectionsByIds` and `DatabaseSessionEmbedded.getCollectionNameById` (`DatabaseSessionEmbedded.java:2842-2852`).
**Issue**: The plan describes this gap as resolved "at commit re-resolve," but the trace shows the throw fires earlier, at handle-build time inside the open tx, not at commit. The deferred branch of `createIndex` (line 435+) calls `findCollectionsByIds(collectionIdsToIndex, session)` at line 439 to populate the deferred handle's `IndexMetadata`; `findCollectionsByIds` calls `getCollectionNameById(collectionId)`, which returns null for any `collectionId < 0` and makes `findCollectionsByIds` throw `IndexException("Collection with id -2 does not exist.")`. So indexing a same-tx-created class fails immediately when the deferred handle is built (the `CREATE INDEX` statement), before any commit. The plan's fix ("resolve provisional ids via `TxSchemaState` ... so the deferred handle stores the right name and the commit-time engine build re-resolves it") is correct in direction but must intercept at the handle-build path (line 439), not only at commit — the commit-time re-resolve is a second, separate resolution.
**Proposed fix**: Sharpen the Plan-of-Work note: the provisional-id (`<= -2`) resolution via `TxSchemaState.getProvisionalCollectionNames()` must run in the deferred `createIndex` branch (or in a tx-aware variant of `findCollectionsByIds`) so the handle build does not throw, and again at commit to re-resolve the now-real id. State both resolution points. The existing acceptance line (track-5.md:192-194) covers the end-to-end case; add that the `CREATE INDEX` on a same-tx class must not throw at statement time.

  2. `schemaContext.txLocalSchema().resolveProvisionalCollectionIds(getResolvedCollectionIds())` @ AbstractStorage.java:2489 — patches every tx-local class's collection-id arrays and rebuilds `collectionsToClasses`; does NOT call `forceSnapshot()` on the tx-local copy.

  6. `doGetAndCheckCollection(collectionId)` @ AbstractStorage.java:2023 → `checkCollectionSegmentIndexRange(id)` @ AbstractStorage.java:6282 throws `IllegalArgumentException` for `id < 0`.

#### E2 Exposure: makeSnapshot tx-awareness flips read semantics for 94 production snapshot readers
- **Track claim**: D21 — "the snapshot is the single read tier the whole read/query/serialize/security stack consumes (174 call sites), so making it tx-aware gives consistent read-your-writes."
- **Critical path trace**: `SchemaProxy.makeSnapshot()` @ SchemaProxy.java:75-80 currently returns `delegate.makeSnapshot(session)` (committed). D21 routes it to the tx-local `SchemaShared` during a schema/index tx. Consumers reach it through `MetadataDefault.getImmutableSchemaSnapshot()` @ MetadataDefault.java:105 and `makeThreadLocalSchemaSnapshot()` @ MetadataDefault.java:77 (refcount-pinned via `immutableCount`).
- **Blast radius**: `ReferencesSearch` on `MetadataDefault.getImmutableSchemaSnapshot` = 174 refs (94 production, 80 test). Production readers span the query planner, entity serializers, and the security stack; all begin seeing tx-local classes/properties/rules mid-tx during a schema/index tx.

- **Residual risk**: MEDIUM — the change is one method, but the non-planner readers (serializer, security) are not enumerated against the provisional-collection / unbuilt-engine surface the planner guard handles.

#### E3 Exposure: failed engine-creating commit orphans the engine component on the in-memory profile
- **Track claim**: `## Plan of Work` gap (1) — assert `indexEngines`/`indexEngineNameMap` carry no entry after a failed engine-creating commit and the ids are reused next commit (I-A4 engine arm / TB2).
- **Critical path trace**: the commit build creates the engine via `doAddIndexEngine(atomicOperation, engineData)` @ AbstractStorage.java:3402 (files buffered as WAL-reverted intent; on disk they vanish on rollback), with the in-memory registry publish deferred to `publishIndexEngine(internalId, engine)` @ AbstractStorage.java:3444 past `commitChanges`. On a failed commit the deferred publish never runs, so the registry stays clean — but `AtomicOperationBinaryTracking.addFile` (the in-memory engine) does not revert the file install on rollback (documented in `undoReconciledCollections` @ AbstractStorage.java:2879), so the engine's id-named component survives as an orphan.
- **Blast radius**: on the default in-memory test profile, the next commit that reuses the freed engine id hits "file already exists"; the I-A4 engine-arm acceptance ("ids reused on the next successful commit") fails. Exactly the shape of the Track-4 collection-arm failure (YTDB-1175).
- **Existing safeguards**: Track 4's `undoReconciledCollections` already carries a create-side structural-revert arm for collections (fresh atomic op, component-present-guarded, no-op on disk) — the pattern to mirror for engines. The lock-free `doDeleteIndexEngine` @ AbstractStorage.java:3715 is the delete primitive available.
- **Residual risk**: MEDIUM — well-understood, with a proven mitigation pattern already in the tree; the risk is that the plan's acceptance line checks only the registry, not the on-disk/in-cache component, and the in-memory profile is the one that catches it.
