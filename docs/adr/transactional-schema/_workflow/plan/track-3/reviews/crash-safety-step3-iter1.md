## Findings

### CS1 [informational] The eager in-tx collection allocation reachable through the de-guard persists in its own durable atomic operation; a rolled-back in-tx create leaves a recovery-visible stray collection — correctly scoped to Track 4

- **File**: `core/src/main/java/com/jetbrains/youtrackdb/internal/core/metadata/schema/SchemaEmbedded.java` (line 457-475), reachable via the de-guard in `dropClass`/`dropClassInternal` and the create path
- **Crash scenario**: A session opens a user transaction, calls `createClass` (now permitted because the throw-guards are removed), and the transaction later rolls back — or the process crashes after the create but before the user transaction commits. The class metadata change is correctly confined to the tx-local copy (see refutation roster: all five de-guarded paths leave the shared schema and shared index registry untouched). But the collection that backs the new class was allocated eagerly during the create through a self-contained atomic operation that committed to the WAL independently of the user transaction. After rollback or crash-recovery the metadata change is gone while the collection file persists on disk.
- **Evidence**: Write-path trace, PSI-backed.
  - In-tx `createClass` → `createClassInternal` allocates collections via `createCollections` → `session.addCollection(collectionName)` (PSI: `SchemaEmbedded.createCollections`).
  - `DatabaseSessionEmbedded.addCollection` → `AbstractStorage.addCollection`, which calls `atomicOperationsManager.calculateInsideAtomicOperation(op -> doAddCollection(op, collectionName))` (PSI: `AbstractStorage.addCollection`). `calculateInsideAtomicOperation` runs and closes a complete atomic operation right then — it does not append to the user transaction's atomic unit — so the collection-create WAL record becomes durable immediately and the file is created independently of the user transaction's commit/rollback.
  - Consequently the user transaction's rollback (which reverts only the records that rode it) does not reclaim the collection, and a crash-before-commit leaves a committed collection-create atomic unit replayed by recovery into a real on-disk collection with no class pointing at it.
- **Recovery impact**: WAL replay reconstructs the eagerly-created collection as a committed durable structure (its atomic op end-record became durable at allocation time). The schema metadata that would have referenced it never committed, so recovery yields a consistent-but-orphaned collection — a stray file, not corruption: no index, collection, or metadata structure is left half-written, and no shared in-memory structure is left mutated.
- **Refutation considered**: (1) Is this introduced by *this* diff? No. The diff changes no collection-allocation, WAL, page-cache, or disk-cache code (the changed files are `IndexManagerEmbedded`, `SchemaEmbedded`, `SchemaShared`, and a test). The eager `createCollections`/`addCollection` path is pre-existing; the de-guard only *enables* an in-tx `createClass` to reach it (the throw-guards previously forced DDL top-level, where the same self-committing allocation was the only behaviour). (2) Does the diff make the intermediate state *worse* — a partially-applied shared mutation, a half-written record, a recovery-visible metadata inconsistency? No. Every de-guarded path was traced (C1): the membership ripple early-returns before the only shared `collectionsToIndex` mutator; `saveInternal` early-returns under `txLocal`; the tx-local `dropClass`/`dropClassInternal` defer all structural deletion and only record into an in-memory per-tx set; `createIndex`/`dropIndex` build no engine and touch no shared registry. The only durable side effect of an in-tx DDL on this step's paths is the pre-existing eager collection allocation. (3) Is it correctly scoped? Yes — the track file `## Idempotence and Recovery` "Eager-allocation intermediate state" states the in-tx `createClass` still allocates eagerly and "a rolled-back in-tx create can therefore leave a stray collection on disk — a known intermediate-state condition closed by Track 4 (D10 reconciliation / D2 provisional ids), not a Track-3 bug." D2 inverts this to a provisional negative id resolved at commit; D10 makes the structural create buffered intent applied only in `commitChanges`, which rollback skips. The condition is the documented, Track-4-owned inversion target. Reported as informational only to record the precise durability characterization (the stray collection is recovery-visible and durably committed, not merely an in-memory registration) so the Track-4 reviewer can confirm D2/D10 close exactly this window.
- **Suggestion**: None for this step (out of scope, correctly documented). Flag forward to the Track-4 crash-safety review: confirm the provisional-id (D2) and buffered-structural-intent (D10) changes eliminate the durable eager collection allocation so an in-tx create leaves no recovery-visible structure on rollback or crash-before-commit.

## Evidence base

#### C1 — Every de-guarded path leaves shared/durable state untouched in-tx; the only durable in-tx side effect is the pre-existing eager collection allocation (PSI-backed)

- **Shared `Index.collectionsToIndex` is the field the atomicity claim turns on.** PSI `ReferencesSearch` on `IndexAbstract.collectionsToIndex` returns its mutators as `create`, `load`, `rebuild`, `fillIndex`, `doFillIndex`, `addCollection`, `removeCollection`, `save`. The membership-ripple mutators are exactly `addCollection`/`removeCollection`; `create`/`load`/`rebuild`/`fillIndex` are engine-build paths, not the membership ripple.
- **The de-guarded membership sites are the only in-tx route to those mutators.** PSI: `IndexManagerEmbedded.addCollectionToIndex` production callers are `SchemaEmbedded.createClassInternal` and `SchemaClassEmbedded.addCollectionIdToIndexes`; `removeCollectionFromIndex` production caller is `SchemaClassImpl.removeCollectionFromIndexes`. All route through the de-guarded manager methods. The de-guard's `recordMembershipChangeIntoTxLocalView` early-`return`s *before* `acquireSharedLock()` and before `index.addCollection`/`removeCollection`, so the shared `collectionsToIndex` add/remove and the `save(transaction)` inside them never run in-tx. PSI confirms `IndexAbstract.addCollection`/`removeCollection` are the sites that do `collectionsToIndex.add/remove` + `save(transaction)`.
- **`recordMembershipChangeIntoTxLocalView` mutates no shared/durable state.** It calls `session.ensureTxSchemaState()` (seed) then a read-locked read of the index definition's class name, then `txState.markClassChanged(changedClass)`. PSI: `TxSchemaState.markClassChanged` does `changedClasses.add(className)` on an in-memory per-tx `Set`; nothing persists.
- **The seed is read-only w.r.t. durable/shared state.** PSI: `DatabaseSessionEmbedded.ensureTxSchemaState` builds `committed.copyForTx(this)` and stashes it in the transaction's custom data (dies with the tx). `SchemaShared.copyForTx` re-parses the committed root record read-only (`copy.fromStream(session, session.load(identity))`), sets `txLocal = true` on the private copy before re-parse, and opens/commits no transaction of its own. The committed shared instance keeps `txLocal == false`.
- **`saveInternal` `txLocal` early-return covers both callers.** PSI `MethodReferencesSearch` on `SchemaShared.saveInternal`: callers are `releaseSchemaWriteLock` and `fromStream`. The new `if (txLocal) return;` at the top short-circuits both, so a mutation against the tx-local copy never persists eagerly.
- **`createIndex` de-guard returns an engine-less, record-less handle.** PSI: the de-guard returns `Indexes.createIndexInstance(type, algorithm, storage)` — the overload `findFactoryByAlgorithmAndType(...).createIndex(indexType, storage)` that takes no `FrontendTransactionImpl` and no `RID`, so it builds no engine and allocates no record; the shared `indexes` registry is not mutated.
- **`dropIndex` de-guard is a records-only no-op.** It records `markClassChanged` under a shared read lock and returns; the shared registry and engine are intact in-tx.
- **The only durable in-tx side effect is pre-existing.** PSI: in-tx `createClass` reaches `createCollections` → `session.addCollection` → `AbstractStorage.addCollection` → `atomicOperationsManager.calculateInsideAtomicOperation(...)`, a self-contained durable atomic op (see CS1). Not in this diff; documented Track-4 condition.

#### Refuted candidates (survived the refutation check → not reported)

- **Membership ripple leaks into the shared `Index.collectionsToIndex` on rollback (the I-A7 silent-leak surface).** CONFIRMED-not-a-bug. The de-guard early-returns before `acquireSharedLock()` and before `index.addCollection`/`removeCollection`, the only membership-path mutators of the shared field (PSI). A rollback leaves `collectionsToIndex` untouched because it was never mutated. Test `membershipRippleInTransactionLeavesSharedIndexUntouchedOnRollback` pins both the during-tx and post-rollback invariant; `membershipRippleRecordsChangedClassIntoTxLocalState` pins the capture into the in-memory changed set.
- **`saveInternal` `txLocal` early-return half-persists a schema record.** CONFIRMED-not-a-bug. The early-return is the first statement; it returns before `tx.isActive()` is even consulted and before any record write, so no partial schema-record write occurs on the tx-local path. `txLocal` is set only on the `copyForTx` copy; the committed shared instance keeps the legacy `saveInternal` body (which still throws on an active tx on the top-level path, preserving the non-tx-local contract).
- **The tx-local `dropClass`/`dropClassInternal` durably deletes a collection/index before commit.** CONFIRMED-not-a-bug. PSI: the `txLocal` branch skips `deleteCollection`, `dropClassIndexes`, and (in `dropClass`) `localCache.freeCollection`, recording only `markClassChanged`. The structural deletion is deferred to commit (Track 4). A rolled-back in-tx drop leaves the shared structure and the indexes' collection membership intact. Test `dropClassInsideTransactionIsIsolatedAndRollsBack` pins the committed-schema isolation half.
- **`createIndex` de-guard builds a partial engine or registers a half-state index.** CONFIRMED-not-a-bug. The returned `createIndexInstance(type, algorithm, storage)` overload allocates no record and builds no engine (PSI — no transaction/RID parameter), and the shared `indexes` registry is not mutated (`existsIndex` returns false during the tx — test `createIndexInsideTransactionDefersToCommitAndDoesNotThrow`). The engine-less handle is a pure in-memory object discarded with the transaction; no durable or recovery-visible partial state exists.
- **`dropIndex` de-guard leaves the engine in a half-dropped state.** CONFIRMED-not-a-bug. The de-guard does not touch the engine or the registry; it records `markClassChanged` and returns. The index stays present and usable during the tx (test `dropIndexInsideTransactionDefersToCommitAndDoesNotThrow`), and a rollback keeps it. The engine drop is the commit's own atomic operation (Track 4), not this step.
- **A concurrent session observes the in-tx schema change (isolation break that recovery would then have to reconcile).** CONFIRMED-not-a-bug. The change lands only in the per-session tx-local copy and the in-memory changed set; the committed shared schema and shared index registry are untouched until commit (PSI + the `txLocal` gating). Test `concurrentSessionDoesNotSeeInTransactionCreate` pins it. No cross-session visibility means no inconsistency for recovery to reconcile.
- **Absence of WAL changes is a crash-safety gap.** CONFIRMED-not-a-bug (by design). The step changes no WAL, page-cache, or disk-cache code; schema crash-recovery and the WAL-replay fix are owned by Track 1 (lazy-consult replay, landed) and Track 4 (commit-time reconciliation in the commit's own atomic operation). The de-guard's correctness rests on free transactional rollback of in-memory tx-local state, which needs no new WAL coverage. Per review scope, the missing WAL changes are not flagged as a gap.
