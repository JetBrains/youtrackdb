## Findings

### R1 [should-fix]
**Certificate**: Exposure E2 (commit-time promotion and structure allocation are Track 4, not Track 3)
**Location**: `core/src/main/java/.../metadata/schema/SchemaEmbedded.java:108` (`createCollections` eager allocation) and `:896` `saveInternal` (`SchemaShared.java`); track-3.md `## Plan of Work` and the D1 enablement-facet caveat.
**Issue**: The track ships the *enablement* half of D1 (route schema writes to a tx-local view, de-guard the entry points) while the *structural reconciliation and promotion* half is Track 4. The intermediate branch state this produces has two concrete sharp edges the track acknowledges in prose but does not pin down operationally:

1. **Eager collection allocation still runs.** `createClass` → `createCollections` calls `session.addCollection(collectionName)` per collection at create time (verified: `createCollections(3)` body). That is the storage-leading model D1/D2 invert — and the inversion (provisional ids, deferred allocation) is Track 4 (D2). So inside a Track-3 schema tx, `createClass` still allocates a real collection eagerly before the change routes to the tx-local copy. If the tx rolls back, that eagerly-allocated collection is the exact phantom-structure leak D10/Track 4 is designed to prevent — but Track 4's revert machinery is not in place yet.
2. **Nothing promotes the tx-local copy at commit.** Promotion ("promote tx-local, publish overlay, one forceSnapshot", design.md:207) is Track 4. So a schema tx that creates a class, routes the change to the tx-local `SchemaShared`, and commits will see the change *vanish* on commit (the shared instance was never updated), unless Track 3 leaves the legacy `saveInternal`/`toStream` save path live for the commit. The track does not state which.

Likelihood: high that an end-to-end "create class in a tx, commit, reopen" test fails or that a rolled-back create leaves a stray collection. Impact: medium — it is an intermediate-state artifact, not a shipped regression, but it can break existing DDL tests that run with no explicit tx and expect the change to persist.
**Proposed fix**: In Phase A decomposition, add an explicit step note (and a Surprises/Idempotence entry) stating the Track-3 commit contract precisely: either (a) the legacy save path stays live so a committed schema tx still persists via `toStream`/`forceSnapshot` and the tx-local copy is a parallel isolation view only, or (b) the commit is a no-op promotion in Track 3 and end-to-end persistence tests are explicitly deferred to Track 4. Pick one and record it. Either way, scope the Validation tests to *isolation and rollback within the tx* (the track already says this) and add a guard step that confirms a rolled-back tx-local create does not leave a stray collection file given that eager allocation still runs — if it can, document the leak as a known intermediate-state condition closed by Track 4 (D10), not a Track-3 bug.

### R2 [should-fix]
**Certificate**: Exposure E1 (tier-3 re-routing surface, PSI-measured)
**Location**: `core/src/main/java/.../metadata/schema/SchemaClassProxy.java` (76 of 78 methods dereference `delegate.`), `SchemaPropertyProxy.java` (47 of 48). `SchemaProxy.java` read/write routing.
**Issue**: The design's tier-3 model requires that during a session's schema tx, every class/property proxy method re-resolves its target by name in the tx-local copy instead of using its captured `delegate` (the `final SchemaClassImpl` reference captured at proxy creation). PSI shows the surface is ~123 call sites: nearly every method in both proxy classes touches `delegate.` directly, including 22 mutator-named methods on `SchemaClassProxy` (`createProperty`, `addSuperClass`, `setName`, `set`, `dropProperty`, …) and 21 on `SchemaPropertyProxy`. A *single* missed method that keeps using the captured shared `delegate` during a schema tx silently breaks isolation (a mutation lands on the shared instance, or a read returns committed state mid-tx) — exactly the silent-failure class the track warns about for the self-commit guards. Per-method edits across 123 sites are error-prone and untestable line-by-line.
**Proposed fix**: Decompose the proxy routing as a *single resolution seam*, not 123 edits: introduce one `resolveDelegate()` (tier 1 snapshot / tier 2 captured / tier 3 name-rebind) helper that every method funnels through, so the routing decision is made in one place and the per-method change is mechanical (`delegate.foo()` → `resolve().foo()`). Add a boundary test (extend the existing `SchemaClassProxyBoundaryTest` / `SchemaProxyBoundaryTest`) that asserts a captured pre-tx proxy mutated inside a schema tx routes to the tx-local class and a concurrent session never sees the shared impl change — the I-A5 acceptance line. Treat this proxy-routing step as a candidate `high`-risk step in decomposition.

### R3 [should-fix]
**Certificate**: Exposure E3 (the I-C2 shared-lock-acquisition deadlock surface, with the concrete index-manager lock site)
**Location**: `core/src/main/java/.../index/IndexManagerEmbedded.java:104` and `:134` — `addCollectionToIndex` / `removeCollectionFromIndex` call `acquireSharedLock()` (the index-manager `ReentrantReadWriteLock` read lock) *before* their `executeInTxInternal`, then `acquireExclusiveLock` inside it. track-3.md `## Plan of Work` ordering constraint; design.md:711-719 (engage above shared locks).
**Issue**: The D7/I-C2 caveat is that the mutex must engage strictly above any shared metadata lock — never from inside a shared-lock acquisition — or a second tx parks on the mutex while holding a shared write lock, freezing lock-based reads and deadlocking against the commit-side schema-lock acquisition. The de-guarded membership sites are the most dangerous engage location precisely because they already take `acquireSharedLock()` (a read lock on the index-manager lock, one of the four ordered locks) before doing their work. If the engage hook is placed at or below the index-routing layer's lock acquisition — the natural place to add it when de-guarding these very methods — a second schema tx reaching the hook parks on the mutex while holding the index-manager read lock. That violates the four-lock order (mutex must precede the index-manager lock) and is the exact I-C2 deadlock. Likelihood: medium — the de-guarding work happens in the same methods that hold the lock, so the wrong placement is the path of least resistance. Impact: high if it lands (deadlock under concurrent schema txs), but the track explicitly calls out the constraint, so the residual risk is placement discipline, not unawareness.
**Proposed fix**: In decomposition, make "engage strictly above `acquireSharedLock`/`acquireSchemaWriteLock`" a checked precondition of the engage step, with an asserting test: a schema tx must already hold the mutex permit *before* either the index-manager read lock or `SchemaShared.lock` is taken on its first write. Add the lock-order assertion as a Java `assert` at the engage point (permit held ⇒ no shared metadata lock yet held by this thread) so a mis-ordered engage fails loudly in tests. The "long schema-tx does not freeze lock-based reads" acceptance line (Validation) is the integration-level guard for this; pair it with the assert-level guard.

### R5 [suggestion]
**Certificate**: Assumption A1 (the self-commit leak premise, verified against the nesting semantics)
**Location**: track-3.md:78-93 (`## Context and Orientation`), design.md:296-302; verified against `DatabaseSessionEmbedded.executeInTxInternal` / `commitImpl` / `FrontendTransactionImpl.commitInternalImpl`.
**Issue**: The track states the membership self-commit "escapes the user transaction, becomes visible to other sessions, and survives a rollback." PSI tracing shows the nuance: `executeInTxInternal` → `begin()` increments `txStartCounter` when a tx is already active, and `commitImpl` flushes only when `amountOfNestedTxs() <= 1` (`commitInternalImpl` gates real commit on `txStartCounter == 1`). So when the membership ripple runs *inside* an active user tx it nests and does NOT flush — it would ride the user tx already. The self-commit leak is real only when the ripple runs at the *top level*, with no active user tx. That condition does hold today: the throw-guards (`saveInternal`, `createIndex`, `dropIndex`) force every DDL operation to run with no active user tx, so `createClass`/`addSuperClass` (and their membership ripple) always run top-level today and the `executeInTxInternal` does self-commit. The premise is therefore correct, but the prose overstates it as unconditional.
**Proposed fix**: Add one clause to the `## Context and Orientation` self-commit paragraph: the self-commit leaks because today's throw-guards force DDL to run outside any user tx (top-level `executeInTxInternal` flushes immediately); once the throw-guards are removed and DDL rides a user tx, the *same* `executeInTxInternal` would nest harmlessly — which is why the fix is to route the membership change into the tx-local copy rather than rely on nesting. This sharpens the rollback-isolation test target (it must exercise the top-level-becomes-user-tx transition, not a path that already nests).

### R6 [suggestion]
**Certificate**: Testability T2 (the no-freeze isolation assertion's cross-track dependency)
**Location**: track-3.md Validation line 143 ("a long schema-tx body holding the mutex does not freeze concurrent lock-based schema reads"); plan Constraints lines 45-48 and D19 (lock-based read sites convert to snapshot-first in Track 4).
**Issue**: The "long schema-tx holding the mutex does not freeze concurrent lock-based schema reads" acceptance line is a real and valuable I-C2 guard, but the no-freeze property partly depends on machinery outside Track 3: the two remaining lock-based read sites convert to snapshot-first only under D19 (Track 4), and the freezer gate is Track 7. In Track 3 the guarantee that holds is narrower — "the mutex does not engage from inside a shared-lock acquisition, so a held mutex does not imply a held shared write lock" (the engage-placement property R3 covers). A test asserting reads flow unblocked while the mutex is held is feasible at Track 3 (the mutex is `Semaphore(1)`, orthogonal to `SchemaShared.lock`), but it must not over-claim the full D19 no-outage property.
**Proposed fix**: Scope the Track-3 version of this test to the mutex-orthogonality property: a thread holding the mutex permit does not hold `SchemaShared.lock` or the index-manager lock (because engage is above them and the seed/copy releases them), so a concurrent snapshot-based read and a concurrent lock-based read both proceed. Note in the test comment that the whole-commit-duration no-outage guarantee (lock-based reads converted to snapshot-first) is D19/Track 4 and the freeze-vs-read-outage guarantee is Track 7. This keeps the assertion honest about what Track 3 proves.

#### A1 Assumption: the self-commit membership site leaks across rollback
- **Track claim**: `addCollectionToIndex`/`removeCollectionFromIndex` "wrap their work in `session.executeInTxInternal(...)`, a nested transaction that commits the moment the method returns ... escapes the user transaction, becomes visible to other sessions, and survives a rollback" (track-3.md:81-86).
- **Evidence search**: PSI — `DatabaseSessionEmbedded.executeInTxInternal`/`begin`/`finishTx`/`commitImpl`, `FrontendTransactionImpl.beginInternal`/`commitInternalImpl`/`amountOfNestedTxs` bodies.
- **Code evidence**: `executeInTxInternal` → `begin()` (`DatabaseSessionEmbedded`); `begin()` with an active tx calls `currentTx.beginInternal()` which increments `txStartCounter`; `finishTx(ok)` calls `commit()` → `commitImpl`, which returns `commitInternal()` (count-down only) when `amountOfNestedTxs() > 1`, and `commitInternalImpl` flushes only when `txStartCounter == 1`. So `executeInTxInternal` self-commits (flushes) only at the top level (no active user tx). Today's throw-guards (`saveInternal:899`, `createIndex:306`, `dropIndex:458`) force DDL to run with no active user tx, so the membership ripple is always top-level today → it does self-commit and leak.
- **Verdict**: VALIDATED — the leak is real today, conditional on the top-level execution that the current throw-guards guarantee. The track prose states it unconditionally; sharpening it improves test targeting (R5).
