## Decision Log

- 2026-06-16 [ctx=safe] **D1 — Scope: build the builder machinery only.** This plan produces the generator, not the book. The first `docs/workflow-book/` content is produced by a later run of the machinery.
  - **Why:** The aim is "create workflow to generate book"; the book is the machinery's output. Separating the two keeps this plan bounded and lets the machinery be reviewed before any chapter is written.
  - **Alternatives rejected:** Build machinery + first book in one plan — too large; couples generator quality to first-draft content quality.

- 2026-06-16 [ctx=safe] **D2 — Machinery lives in a top-level `workflow-book-builder/` directory of prose prompts/briefs/agent-defs; the branch is NOT workflow-modifying.** Not a Maven module (content is markdown, not Java). Mirrors the internals-book model.
  - **Why:** Keeping the machinery as copy-paste prompts (not `.claude/` skills) avoids §1.7 staging, a drift baseline, and the heavier workflow-modifying machinery. The internals book proved the prose-prompt model works.
  - **Alternatives rejected:** Codify as `.claude/` skills+agents — more native but triggers workflow-modifying overhead for no clear payoff at this stage. Name: chose `workflow-book-builder` over `-press`/`-forge`/`-pipeline` because it reads as plain language and matches the `*-builder` convention; the user invited a better name and this one fits. The user's word "module" is taken as "directory", not a Maven module — the content is markdown, not Java.

- 2026-06-16 [ctx=safe] **D3 — Book output home is `docs/workflow-book/`.** The produced book (README + baseline, TOC, chapters, rendered diagram assets) lives here, separated from the builder per the user's exception.
  - **Why:** Reading material belongs under `docs/`; production machinery does not.
  - **Alternatives rejected:** Interleaved single dir (the internals-book layout the user explicitly wants to avoid).

- 2026-06-16 [ctx=safe] **D4 — Audience: new engineers adopting the workflow.** The teaching arc is onboarding-first: how to run the workflow and why it is shaped the way it is, ramping one concept at a time.
  - **Why:** User's answer. Sets the BOOK_BRIEF voice and the chapter ordering (concrete-before-abstract, gradual ramp).
  - **Alternatives rejected:** Extender-focused or reference-only framing — wrong altitude for onboarding.

- 2026-06-16 [ctx=safe] **D5 — Diagrams: hybrid, ASCII-default with committed rendered SVG for a short enumerated set of dense figures.** ASCII in fenced blocks is the default for all flow/box diagrams (portable, zero-tooling, diffs cleanly). Committed SVG is reserved for an explicit, bounded list of figures ASCII genuinely cannot carry — the candidates are the phase state machine, the tier-gate decision tree, and the track/step/episode hierarchy. The SVG source lives in a sidecar `.d2`; the rendered SVG is committed so readers need no tooling. The exact list is fixed at TOC time (the chapter plan is generated by the first run, D7), bounding the SVG footprint to a named set rather than an open category.
  - **Why:** Solves the viewer-portability problem the user raised. ASCII renders identically everywhere; committed SVG renders in every image-capable viewer (GitHub, IDE, PDF, static site) without a reader-side build. The all-ASCII option (already mixed into `.claude/workflow` docs such as `branch-divergence-check.md` and `conventions-execution.md`, and house-style-preferred) is the cheaper baseline; SVG earns its build cost only on the named dense figures where ASCII layout breaks down.
  - **Diverges from the base model.** The chosen base (`docs-ytdb-internals-book/`) ships inline mermaid in all 17 chapters and **zero** rendered assets. D5 deliberately departs from the model on diagrams; this is the one place the workflow-book is not a mirror of the internals book. OQ2's diffability-loss caveat for pre-rendered SVG is accepted in exchange for reader-side portability, and contained by keeping the SVG set small and the source diffable in the sidecar.
  - **Alternatives rejected:** Mermaid-only (the model's status quo — the viewer-support problem being solved); all-ASCII (too weak for the named dense figures); render-on-read (forces reader tooling).

- 2026-06-16 [ctx=safe] **D6 — Render DSL is D2 (a preference, not a verified-robustness claim).** The dense figures are authored in D2 source (sidecar `.d2`), rendered to committed SVG by `render-diagrams.sh`.
  - **Why:** Preference for a single Go binary with no headless-browser dependency over mermaid-cli's puppeteer/Chromium chain, which is the fragile part of a repeatable build. This is a design preference, not a measured robustness result.
  - **Install prerequisite (load-bearing).** Neither tool ships in the current environment: `d2` is not installed, `mmdc` is not installed, and only mermaid's `node`/`npx` prerequisite is present. So `render-diagrams.sh` must check for `d2` and print the install command on a miss, and the brief must document the one-time `d2` install as an operator step. The render path does **not** run during this plan — D1 scopes the deliverable to the builder with no chapters authored, so the renderer first runs in a later production cycle — which is why a missing-binary risk is an operator-doc requirement here, not an execution blocker for this plan.
  - **Alternatives rejected:** Mermaid via mmdc (heavy/fragile puppeteer install, though it is the only half-present toolchain today); Graphviz/DOT (lower-level syntax, weaker flowchart layout).

- 2026-06-16 [ctx=safe] **D7 — The chapter plan / TOC is generated by the first production run, not authored as a builder artifact.** `BOOK_BRIEF.md` defines the audience and arc principles; `PRODUCTION_PROMPT.md`'s first step produces the TOC + per-chapter briefs (mirroring the internals-book cycle-1 step 1). No pre-authored `CHAPTER_PLAN.md`.
  - **Why:** Keeps the machinery generic and reusable; the chapter map is an output of running the generator, consistent with how the internals book was produced.
  - **Alternatives rejected:** Ship a pre-authored chapter map — couples the generic machinery to one chapter decomposition and front-loads content design into this plan.

- 2026-06-16 [ctx=safe] **D8 — Evolution uses the full role set, not a narrow drift-sweep.** The machinery supports both initial production and ongoing evolution; evolution runs can add and restructure chapters via author → technical-reviewer → copy-editor → beta-reader waves, not just line-number sweeps.
  - **Why:** The book's subject (`.claude/workflow`) keeps changing; new phases/tiers/gates need new or rewritten chapters. The base model splits production into **two hand-driven entry points** — a lightweight `MAINTENANCE_PROMPT` drift refresh and a heavyweight author-wave cycle for new content (cycle 1 → cycle 2 in its README added new chapters this way, and it worked). That split is a working design, not a defect. We fold the two into one machinery so a single operator path covers both the refresh and the add-content cases instead of choosing a prompt up front.
  - **Alternatives rejected:** Keep the two separate hand-driven entry points — defensible, but it makes the operator pick the mode before knowing the drift size, and duplicates the wave-orchestration prose.

- 2026-06-16 [ctx=safe] **D9 — The TOC is a living artifact owned by the book (`docs/workflow-book/TOC.md`), modified during evolution.** When the source workflow gains a concept with no chapter home, the pipeline edits the TOC (add/reorder/split chapters) and updates the cross-reference matrix (chapter → source-concept map).
  - **Why:** A book that tracks an evolving subject cannot freeze its chapter structure. Consistent with D7 (chapter plan generated by the first run, never shipped as a frozen builder artifact).
  - **Alternatives rejected:** Frozen TOC with append-only chapters — cannot absorb restructuring when the subject reorganizes.

- 2026-06-16 [ctx=safe] **D10 — One unified evolution-aware pipeline; initial production = evolution from an empty baseline.** A run always: computes the drift window over `.claude/workflow/**` + skills + agents since the baseline workflow-SHA; classifies impact per area (clean / sweep / rewrite / new-or-restructure); edits the TOC for the new-or-restructure band; runs author/technical-review/copy-edit/beta waves on touched chapters; bumps the baseline + appends an evolution-history row. Initial production is the empty-baseline case (drift window = everything, TOC from scratch, every chapter through the full waves). One pipeline doc, not a separate PRODUCTION_PROMPT + MAINTENANCE_PROMPT.
  - **Why:** Avoids two copies of the wave-orchestration prose and one fewer artifact to keep in sync; the empty-baseline framing makes the from-scratch and incremental cases share one code path.
  - **Risk to carry into the pipeline design.** The from-scratch case (empty baseline, TOC from scratch, every chapter through the full waves) and the incremental case (drift window, clean/sweep/rewrite/new-or-restructure triage, a few touched chapters) have genuinely different control flow. The single pipeline must still branch on baseline-empty vs non-empty; "one code path" means one document and one role set, not identical control flow. The pipeline design must make that branch explicit so it is neither over-general for the common incremental case nor under-specified for the from-scratch case.
  - **Alternatives rejected:** Two entry points sharing the role prompts — more prose, two drift/wave descriptions to keep aligned.

- 2026-06-16 [ctx=safe] **D11 — Concrete artifact layout (settled).** The deliverable file set:
  - `workflow-book-builder/` (machinery): `BOOK_BRIEF.md` (audience, voice, conventions, diagram rules, role definitions), `PIPELINE.md` (the unified evolution-aware pipeline, D10), `DIAGRAMS.md` (the hybrid ASCII+D2 convention, the render step, "how to add a figure"), `prompts/author.md`, `prompts/technical-reviewer.md`, `prompts/copy-editor.md`, `prompts/beta-reader.md` (the four role prompts the pipeline spawns), `scripts/render-diagrams.sh` (D2 → committed SVG), and `reviews/` + `beta-feedback/` (empty run-output directories).
  - `docs/workflow-book/` (book target, layout documented now, filled by a later run): `README.md` (production record + source-tree baseline pinning a workflow-SHA + evolution history), `TOC.md` (the living chapter map + cross-reference matrix, generated and maintained by a run, D7/D9), `chapters/` (empty), `assets/diagrams/` (committed `fig-N.svg` + `.d2` sidecars, empty).
  - **Why:** Consolidates the layout scattered across D2/D3/D5/D6/D7/D9/D10 into one file list so the plan derives from the log alone. `PRODUCTION_PROMPT.md` and `MAINTENANCE_PROMPT.md` collapse into the single `PIPELINE.md` (D10).
  - **Risks/Caveats:** The book-target directories ship empty (a `.gitkeep` or a one-line README placeholder per empty dir) so the layout is visible without committing generated content (D1 — builder only).

- 2026-06-16 [ctx=safe] **D12 — The operator entry point is a copy-paste START prompt embedded as the lead section of `PIPELINE.md`.** `PIPELINE.md` opens with a self-contained "paste this into a fresh session" block, followed by operator context; the operator pastes exactly one thing to start any run, and that session orchestrates the role-prompt waves itself.
  - **Alternatives considered:** A separate `START.md` paste-target that points at `PIPELINE.md` as the detailed reference.
  - **Rationale:** Mirrors the base model exactly — the internals book's `MAINTENANCE_PROMPT.md` is a single file whose first section is the "Prompt (copy-paste this block into a new Claude Code session)" and whose remainder is operator context. The unification (D10) folded the two prompts into `PIPELINE.md` but lost the pasteable-entry-point property; D12 restores it. The four `prompts/*.md` role files are spawned by the orchestrating session, not pasted by hand, so there is exactly one human entry point. Embedding keeps the single-pipeline-file unification and the file count down.
  - **Risks/Caveats:** The start block must be self-contained — it re-reads `BOOK_BRIEF.md`, `DIAGRAMS.md`, the role prompts, the `docs/workflow-book/TOC.md`, and the source tree on its own — and it must carry the empty-vs-non-empty-baseline branch (D10) so one paste covers both initial production and evolution.
  - **Implemented in:** this track (`PIPELINE.md` authoring).

## Surprises & Discoveries

- 2026-06-16 [ctx=safe] The internals-book production pipeline is **prose-prompt-driven, not codified as `.claude/` skills or agents**. `BOOK_BRIEF.md` (voice/audience/conventions/pipeline), `MAINTENANCE_PROMPT.md` (copy-paste refresh prompt), `TOC.md` (chapter plan + cross-reference matrix), `reviews/`, `beta-feedback/`, `team/`, `maintenance/`. The operator pastes prompts and drives author/reviewer/copy-editor/beta-reader waves by hand. Implication: the workflow-book builder can stay non-workflow-modifying (no `.claude/**` edits, no §1.7 staging) if it follows the same model.
- 2026-06-16 [ctx=safe] In the internals book the production machinery and the book content live in **one interleaved directory** (`docs/ytdb-internals-book/` holds both `chapters/` and `BOOK_BRIEF.md`/`MAINTENANCE_PROMPT.md`/`reviews/`/`beta-feedback/`). The user's exception splits these: book → `docs/workflow-book/`, machinery → `workflow-book-builder/`.
- 2026-06-16 [ctx=safe] Subject corpus is large: `.claude/workflow/` 31 docs (~1.3M) + 11 prompts + 20 agents + 16 skills. The workflow book teaches phases 0–4, tiers (full/lite/minimal), tracks/steps/episodes, review agents, drift/migration, workflow-SHA stamps, §1.7 staging. Chapters map to concepts, not 1:1 to files.
- 2026-06-16 [ctx=safe] The book's drift model is the internals book's **`MAINTENANCE_PROMPT.md`**, not the repo's `workflow-drift-check` gate. The maintenance prompt runs `git log BOOK_SHA..NEW_SHA --name-only -- <source paths>` and triages chapters clean/sweep/review — the source-tree content-drift walk the book needs. The repo's `workflow-drift-check.md` gate is a different mechanism: it detects drift in a branch's `_workflow/**` plan artifacts via per-file `workflow-sha:` stamps and a `git merge-base` fold to a `BASE_SHA`, routing to `/migrate-workflow`; none of that stamp/merge-base machinery transfers to a markdown book with no per-file stamps. What **is** reused is only the **workflow-SHA value as the book's baseline SHA** — the book pins a `.claude/workflow` commit the way the internals book pins a source commit, then walks `git log <baseline>..HEAD -- .claude/workflow/** .claude/skills/** .claude/agents/**` per the maintenance-prompt pattern. Current workflow baseline = `3e9c22298d`.
- 2026-06-16 [ctx=safe] Mermaid usage: 31 blocks across 17 internals-book chapters; 22 blocks across `.claude/`. The `.claude/workflow` docs already mix mermaid with hand-drawn ASCII box diagrams (e.g. `branch-divergence-check.md`, `conventions-execution.md`), and the house style prefers ASCII diagrams in chat.

## Open Questions

- 2026-06-16 [ctx=safe] **Scope of this plan.** Does it build (a) only the builder machinery (the book is produced by a later run), or (b) the machinery AND the first production run that yields the actual `docs/workflow-book/` content? This dominates track decomposition. **Resolved → D1** (builder only).
- 2026-06-16 [ctx=safe] **Diagram approach** replacing mermaid. Candidates: ASCII/Unicode box-and-arrow (portable by construction, zero build, limited expressiveness) vs pre-rendered SVG/PNG from a DSL kept in sidecar (portable + expressive, needs a render build step + diffability loss) vs hybrid. Depends on which output targets matter (GitHub web, PDF export, IDE preview, plain terminal, static site). **Resolved → D5/D6** (hybrid: ASCII default + committed D2-rendered SVG for a named dense set; targets are all of GitHub/IDE/PDF/terminal-via-ASCII).
- 2026-06-16 [ctx=safe] **Machinery home and form.** Top-level `workflow-book-builder/` directory of prose prompts/briefs/agent-defs (keeps branch non-workflow-modifying), vs codifying as `.claude/` skills+agents (triggers §1.7 staging, drift baseline, heavier). Name: `workflow-book-builder` vs alternatives (`workflow-book-press`, `-forge`, `-pipeline`). The user said "module" — clarify whether a Maven module is intended (machinery is markdown, not Java, so a plain directory fits). **Resolved → D2** (top-level directory; name `workflow-book-builder` chosen over the alternatives; "module" = directory, not a Maven module).
- 2026-06-16 [ctx=safe] **Book audience and purpose.** Who reads it and to what end — new engineers adopting the workflow, contributors extending it, or a why-it-is-shaped-this-way narrative. Shapes the TOC and the teaching arc. **Resolved → D4** (new engineers adopting the workflow).
