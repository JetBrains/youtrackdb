## Findings

### WI1 [Recommended] Phase-A resume table bypasses the new reconciliation comparison on a post-decomposition resume

- **File:** `docs/adr/track-complexity-assessment-workflow-optimization/_workflow/staged-workflow/.claude/workflow/track-review.md` (line 1063; mechanism added at line 675-720, sub-step 6 at 586-615)
- **Axis:** gate resume path
- **Cost:** an upward-miss reconciliation that the session clears before reaching is silently dropped on resume — the missed Risk / Adversarial strategic reviewers (the entire point of D5: "run the missed reviewers before any code is written") never run, even though the reconciliation rule says they must.
- **Issue:** Step 4 added the reconciliation comparison as a new control-flow step that runs **"after decomposition (the end of sub-step 4, before the A→C commit in sub-step 6)"** (line 680-682). It compares `max(step tags)` against the prediction and, on an upward miss, runs the missed strategic reviewers and feeds their findings back into decomposition. But Step 4 did **not** thread this new step into the Phase-A `Resume actions` table — that table (line 1063) is byte-identical to the live file (confirmed by diff). Its third row matches "all planned reviews recorded `[x]` + non-empty `[ ]` steps" and routes the resume action to **"Decomposition has run; resume from sub-step 6 (commit)"**, which jumps over the reconciliation comparison entirely. Concrete failure: a predicted-`low` track runs Technical-only at sub-step 3 (recorded `[x]`), decomposes at sub-step 4/5 into steps whose `max(step tags)` is `high` (roster written with `[ ]` items), then the session clears before the reconciliation comparison runs. On resume the third row fires, the orchestrator goes straight to sub-step 6, commits, and appends `--reconciled-tag high`. Phase C is correctly floored at `high` rigor (the tag write is on the resume path), but the missed-reviewer pass — Risk + Adversarial, which an upward miss to `high` requires — is skipped, because the resume entered after the comparison's position. The reconciliation's "fires at most once" guarantee is grounded in the `high` intensity ceiling, not in a persisted "reconciliation already considered" marker, so resume cannot tell "reconciliation ran and passed" from "reconciliation never ran." This is a reachable case: D5 exists precisely because a thin `## Plan of Work` yields a weak prediction (so upward misses are expected), and the mandatory context-consumption check is placed *inside* sub-step 3/4 (line 532-546), so a pause right at the heavy decomposition sub-step — before the comparison — is exactly what the workflow anticipates.
- **Suggestion:** Add a fourth row (or extend the third row) to the Resume-actions table covering "all planned reviews `[x]` + non-empty `[ ]` steps + reconciliation not yet recorded": on re-entry, recompute `max(step tags)` over the committed roster and re-run the reconciliation comparison against the prediction *before* proceeding to sub-step 6, so an upward miss detected on resume still runs its missed reviewers. The missed-reviewer passes are already idempotent on resume (each is its own review type gated on its Outcomes & Retrospective `[x]` checkbox per line 525-531), so a re-run that re-detects the same upward miss is safe — an already-`[x]` missed reviewer is a no-op. The fix is to make the resume table *enter* the comparison, not to add new idempotence machinery.

### WI2 [Minor] Context-check handoff trigger omits the decomposition-written-but-reconciliation-pending pause state

- **File:** `docs/adr/track-complexity-assessment-workflow-optimization/_workflow/staged-workflow/.claude/workflow/track-review.md` (line 538-543)
- **Axis:** phase output → next-phase input
- **Cost:** a graceful pause taken right after the roster is written but before the reconciliation comparison runs writes a handoff file that does not record "reconciliation still owed," so the next session has no signal distinguishing this state from a fully-reconciled decomposition.
- **Issue:** The mandatory context-consumption check inside sub-step 3/4 instructs writing a `mid-phase-handoff` file when a pause "leaves Phase A mid-flight," and enumerates the qualifying states by example: "technical review PASSed but risk / adversarial reviews are still unrun, **or** all reviews PASSed but decomposition has not yet been written" (line 538-540). Both examples are *pre-decomposition* states. Step 4 introduced a new mid-flight state that sits *after* the examples — decomposition (roster + step tags) is written, but the reconciliation comparison between sub-step 4 and sub-step 6 has not yet run. A pause there is "Phase A mid-flight" in exactly the sense the trigger means, but the enumeration does not name it, so the handoff file the orchestrator writes need not carry the "reconciliation pending / upward miss detected" signal. This is the graceful-pause analogue of WI1's crash-clear gap (WI1 covers the resume table; this covers the handoff file the orchestrator writes before a context-budget pause). Lower severity than WI1 because the handoff text is by-example ("for example, …"), so a careful orchestrator may still write a useful handoff — but the enumeration does not prompt it to record reconciliation state, and the consuming resume table (WI1) would not act on it anyway.
- **Suggestion:** Extend the handoff-trigger enumeration with the new state — e.g., "…or decomposition is written but the upward-divergence reconciliation has not yet run" — and have the handoff record whether `max(step tags)` exceeds the prediction, so the next session knows the missed-reviewer pass is still owed. Pairs with the WI1 resume-table fix: the handoff supplies the signal, the resume table acts on it.

## Evidence base

#### C1 — Resume table unchanged by Step 4; reconciliation sits between sub-step 4 and sub-step 6
CONFIRMED. `diff` of the `Resume actions` third row between live `.claude/workflow/track-review.md` and the staged copy is empty (the row is byte-identical at staged line 1063); the staged reconciliation section (line 675-720) states it runs "after decomposition (the end of sub-step 4, before the A→C commit in sub-step 6)"; the staged resume table's matching row routes to "resume from sub-step 6 (commit)", which is positioned after the comparison. The missed-reviewer passes are gated on Outcomes & Retrospective `[x]` checkboxes (line 525-531), confirming they are individually idempotent — the gap is that the resume entry point skips the comparison that *decides* to run them, not that the reviewers themselves re-run unsafely.

#### C2 — Handoff-trigger enumeration is pre-decomposition only
CONFIRMED. The context-consumption check at staged line 532-546 lists the mid-flight pause examples as "technical review PASSed but risk / adversarial reviews are still unrun, or all reviews PASSed but decomposition has not yet been written" — both pre-decomposition. The new reconciliation step (line 675-720) executes after decomposition is written, a state the enumeration does not name; the trigger phrasing is by-example ("for example"), so the omission is a coverage gap rather than a contradiction.
